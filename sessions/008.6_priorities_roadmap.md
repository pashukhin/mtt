# 008.6 — Priorities + roadmap

Status: **done**   ·   Branch: `feat/s008.6-priorities-roadmap`

## Target

Give tasks a **priority** (the missing ordering axis alongside `depends_on`/hierarchy) and ship **`mtt
roadmap [--json]`** — an agent-friendly execution-order view: the non-terminal tasks in a
dependency-respecting, priority-weighted sequence, each annotated with `ready`/`blocked_by`. Bundled into one
session (batch-small-features preference); roadmap consumes priority.

## Scope

- **In:**
  - `pkg/mtt`: a closed **`Priority`** VO (`high|medium|low`, `Valid()`/`Rank()`; empty = unset, ranks medium)
    + `Task.Priority` (optional, `omitempty`, not materialized on disk).
  - `adapter/yaml`: `ymlTask.priority` round-trip (plain conversion, tolerant of unknown on-disk values).
  - `core`: `--sort priority` (`SortPriority`, Rank asc + recency tiebreak), `ListFilter.Priorities` + `Match`,
    `Editor` priority edit; **`Roadmap(tasks, cfg) []RoadmapEntry`** — priority-guided Kahn topo sort
    (dependency hard, priority soft), `Ready`/`BlockedBy` annotations, terminal excluded, cycle-safe.
  - `cli`: `--priority` on `add`/`edit`/`list`, `--sort priority`, `priority` in `show`; **`mtt roadmap
    [--json]`** (pure read).
  - Version `0.8.0-dev` → `0.8.6-dev`.
- **Out (later / YAGNI):** config-defined or numeric priority; changing the default `list`/`ready` order
  (priority is opt-in via `--sort`); roadmap dates/critical-path/waves/hierarchy-grouping/leaf-filtering; a new
  port for priority.

## Decisions (brainstormed — see the spec)

Spec: [../docs/superpowers/specs/2026-07-07-session-008.6-priorities-roadmap-design.md](../docs/superpowers/specs/2026-07-07-session-008.6-priorities-roadmap-design.md).

1. Priority = **fixed closed VO** (`StatusKind` idiom), not config-levels/numeric; empty ranks medium, not on
   disk.
2. `roadmap` = a **distinct thin command** (look-ahead sequence), not folded into `ready`/`list`.
3. Scope = **all non-terminal tasks, flat**; `parent` a field, not grouping.
4. Ordering: **dependency hard-constraint, priority soft tie-break**; satisfied (terminal) blocker doesn't
   constrain.
5. Default `list`/`ready` order **unchanged** (recency); priority via `--sort priority`; `roadmap` inherently
   priority-ordered.

## Done

**Shipped** (plan: `docs/superpowers/plans/2026-07-07-session-008.6-priorities-roadmap.md`; strict TDD, 7
green-between commits). Delivered exactly to spec:

1. **`pkg/mtt.Priority` VO** (`priority.go`) — `high|medium|low`, `Valid()` (+ empty), `Rank()` (high=0,
   medium/unset/unknown=1, low=2); the `StatusKind`/`CurrentAction` idiom. `Task.Priority` after `Status`.
2. **yaml DTO** — `ymlTask.priority` (`omitempty` after `status`), plain `mtt.Priority(...)` conversion (unknown
   tolerated → ranks medium). `task_min.yaml` byte-unchanged (back-compat); new `task_priority.yaml` golden.
3. **core** — `SortPriority` + `lessByPriority` (→ `lessByRecency`), `ListFilter.Priorities` + `Match` (stored
   label), `EditParams.Priority` (+ clear via `--priority ""`), and `terminalSatisfied` factored out of `Ready`.
4. **`core.Roadmap`/`RoadmapEntry`** — priority-guided cycle-safe Kahn: own non-terminal DAG (not `DepGraph`),
   reuses `core.Ready` (ready flag) + `terminalSatisfied` (blocked_by); tiebreak == `lessByPriority` so
   `roadmap` == a dependency-constrained `list --sort priority`. All spec cases (hard-constraint, tiebreak,
   annotations, cancelled==terminal, empty/all-terminal, chain-into-cycle, 20×-deterministic).
5. **cli** — `--priority` on `add`/`edit`/`list` (`parsePriority`/`toPriorities`), `--sort priority` (switch +
   error text), `priority:` line in `show`, `taskJSON.priority` (`omitempty`).
6. **`mtt roadmap [--json]`** — `writeRoadmap` (numbered, `[..]` omitted when unset, `↳ blocked by:`),
   `roadmapJSON` (honest `priority` `""`, non-null `blocked_by []`); e2e `roadmap.txt`.
7. **docs + version** — `model.go`, DESIGN/.ru, CLI_REFERENCE/.ru, 4× CLAUDE.md, TASKS (e5_t1a ✅),

**Deviations / notes:**
- Added a local `roadmapCfg()` test fixture (tbd/in_progress/done/**cancelled**) in `roadmap_test.go` — the
  shared `cfg()`/`flow()`/`matchCfg()` helpers lack a `cancelled` terminal, needed to pin the
  cancelled==terminal unblocking rule distinctly from `done`.
- The e2e's `--sort priority` / hard-constraint assertions are **ordering relationships** (`(?s)…`), not
  absolute positions — wall-clock ties make exact numbering non-deterministic (the s003 lesson); the exact
  order is the unit test.
- One staticcheck De Morgan nit in `priority_test.go` (T1) surfaced in the T3 `make check` and was folded into
  the T3 commit (T1 only ran `go test`, not the full gate — lesson: run `make check` per commit, not just tests).

**Rev2 — roadmap ordering reworked (same PR, post-ship user feedback):** running `mtt roadmap` showed the
greedy-by-own-priority order (`roadmap == list --sort priority`) was counter-intuitive — a high task behind a
blocker sank below independent medium work, and a parent/epic wasn't ordered vs its children. Reworked `Roadmap`
to order over **two "comes-after" axes** (`depends_on` **and `parent`** — a child precedes its parent; both
hard) with **priority propagation** (`effectivePriority` = min of own and everything a node transitively
unblocks, memoized + cycle-safe), and added `RoadmapEntry.Contains` (a parent's non-terminal children; `↳
contains:` line + non-null JSON `contains`). `Ready`/`BlockedBy` stay depends_on-only. Retracted the
`roadmap == list --sort priority` and "greedy, not a scheduler" claims (spec Revision 2). Pre-existing roadmap
tests were unchanged (no `parent` edges); added propagation / parent-axis / combined-epic / cross-axis-cycle
tests.
