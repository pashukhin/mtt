# 006.7 — Current task / working context

Status: in progress   ·   Branch: `feat/s006.7-current-task`

## Target

Give tasks a **git-`HEAD`-for-tasks**: a personal `current` pointer so an agent stops repeating the id.
`mtt use t5` selects it, `mtt done` finishes *the current task*, `mtt show` shows it. The pointer lives behind
a **capability port** (YAML → `config.local`; external → assignee), moved by a new committed-flow rule
(`Transition.Current: set|clear`); an omitted id resolves to `current` for single-task verbs only.

## Scope

- **In:**
  - `pkg/mtt`: `Transition.Current CurrentAction` (`set|clear`, value object + `NewCurrentAction`);
    `CurrentStore` capability port (`Current`/`SetCurrent`/`ClearCurrent`) + `CapCurrent`; pure primitive
    `Type.FindTransition` (`core.findTransition` delegates).
  - `adapter/yaml`: `NewCurrent(root)` impl over the `current:` key of `config.local.yaml` via `yaml.Node`
    (preserves `author`/comments); `ymlTransition.current` DTO; `default`/`coding` templates set/clear on
    take-into-work / →terminal edges (+ golden regen).
  - `cli`: `mtt use <id>` / `mtt use` (show) / `mtt use --clear`; apply `edge.Current` after a successful
    `runTransition`; `resolveTaskID` (explicit id > current) for `status`/`mtt <status>`/`show`/`edit` with an
    omitted id; stale/absent-current errors.
  - Version `0.6.5-dev` → `0.6.7-dev`.
- **Out (PARKED, do not build):** `advance`/`start`/`done`/`cancel`, modes, roles-on-edges, config
  verb→status. Omitted-id resolution for `list`/`tree`/`dep`/`ready`/bulk. See the spec's backlog for the
  captured think-items (separate `use`/`current` commands, description-on-move reminder, current for all
  single-task ops, per-agent current, multi-assignee providers, the `advance` reuse seam).

## Acceptance (must pass)

- **User scenario:** `mtt init` → gate config with `current: set` on `tbd→in_progress`, `current: clear` on
  `in_progress→done` → `mtt add A` (t1). `mtt use t1`; `mtt use` shows t1; `mtt show` (no id) shows t1;
  `mtt in_progress t1 --who a --why w`; `mtt done` (no id) finishes t1 and clears current; `mtt use` →
  `no current task`. `mtt done` with no current → non-zero.
- **e2e:** `testscript` `current_task.txt` covering the above.
- `make check` green.

**Deviations from the spec (deliberate, follow codebase idiom):**
- `CurrentAction` is a `StatusKind`-style value object (`Valid()` + `Config.Validate`), **not** a
  `NewCurrentAction` smart constructor — avoids a `toDomain` signature change; `StatusKind` works identically.
- `CapCurrent` + `Capabilities()` **deferred** to `mtt caps` (e4_t6): no `Capability` infra exists yet and
  nothing consumes discovery under CLI-applies (option ii); the `CurrentStore` port alone gives
  provider-agnosticism. A compile-time `var _ mtt.CurrentStore = (*Current)(nil)` guards the port.
- `→ cancelled` edges deliberately do **not** clear current (only `→ done` does; a deliberate release is
  `mtt use --clear`) — noted so a future flow author expects it.
- **Stale current** is proven by a unit path (`resolveTaskID` + `mtt use` show), not e2e — no `mtt rm` until
  s008.5, so the CLI can't produce a dangling pointer; documented gap mirroring s005.
