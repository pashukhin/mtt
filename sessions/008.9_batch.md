# 008.9 — Batch & pipeline

Status: **done**   ·   Branch: `feat/s008.9-batch`

## Target

Make mtt **Unix-composable**: a reusable **task-set selector** shared by every set-operating command
(explicit IDs | stdin `-` | `--filter`), an **`--ids`** output on `list`/`ready`, and **bulk** `tag add/rm` +
`rm` (the no-gate mutations) with `--dry-run` and best-effort per-item semantics. Gated bulk
(`status`/verbs/`edit`/`dep`) stays later.

## Scope

- **In:**
  - `cli/selector.go`: `selectTaskIDs(cmd, positional, allowExplicitIDs)` — 3 mutually-exclusive sources
    (explicit ids | stdin `-` | `--filter` over `core.Select`/`Ready`), dedup, never resolves `current`; the
    shared filter flags (`addSelectorFilterFlags`/`readSelectorFilter`/`filterActive`) + `stripDash`/`hasDash`/
    `readIDsFromStdin`/`idsOf`/`writeIDs`.
  - `--ids` on `list`/`ready` (one id per line, `⊕ --json`).
  - `cli/bulk.go`: `runBulk`/`reportBulk`/`previewBulk` — best-effort report (stdout summary + stderr per-item
    / `--json` per-item array), `--dry-run` preview, generic exit-1 aggregate (no `%w` of a per-item sentinel).
  - `core.Remover.RemoveMany(ids, force) []RemoveResult` — subgraph-aware bulk delete (referents inside the set
    ignored); `Remove` becomes a thin wrapper (single-id semantics + exit-4 preserved).
  - Bulk `rm` (`ArbitraryArgs`, single vs bulk) and `tag add/rm` (marker-driven `tagArgs`, single vs bulk).
  - Version `0.8.7-dev` → `0.8.9-dev`.
- **Out (later / YAGNI):** gated bulk (`status`/verbs/`edit`/`dep`); bulk over `current`; AND-mode tag filter
  (`--all-tags`); `--ids` on `tree`; a union of sources; `--sort` on the bulk commands; transactional/atomic
  bulk; a `selector` core package / a core stdin reader.

## Decisions (brainstormed — see the spec)

Spec: [../docs/superpowers/specs/2026-07-09-session-008.9-batch-design.md](../docs/superpowers/specs/2026-07-09-session-008.9-batch-design.md).
Plan: [../docs/superpowers/plans/2026-07-09-session-008.9-batch.md](../docs/superpowers/plans/2026-07-09-session-008.9-batch.md).

1. **Q1 — `tag` argument layout is context-sensitive.** A selector marker (`-` or a `--filter` flag) → bulk
   (positionals are tags, tasks from the selector); no marker → the single `tag add <id> <tag>…` (back-compat).
   Keeps the ergonomic single form, matches the roadmap examples, covers every case. `rm` has no tag conflict —
   explicit ids stay positional; for `tag`, explicit multi-id goes via stdin.
2. **Q2 — an empty selector result is a no-op (exit 0)**, distinct from *no source at all* (usage error).
   Pipeline-friendly.
3. **Q3 — any per-item failure → exit 1** (git-style, generic); best-effort report; the aggregate never wraps
   a per-item sentinel (so it can't mis-map to 3/4/6). A single explicit `rm <id>` keeps its exit 4.
4. **Q4 — bulk `rm` referenced-check ignores in-set referents** (subgraph-aware) — delete an epic + children in
   one call without `--force`; an external referent still rejects.
5. Defaults: `--ids` ⊕ `--json`; `--dry-run` previews ids on stdout + a stderr summary (or a `--json` id
   array); selector is a **CLI concern** (I/O + flags), not core.

## Acceptance (must pass)

- **User scenario:** `mtt list --tag x --ids | mtt tag rm x -` (pipeline); `mtt tag add y --status tbd`
  (filter source); `mtt rm <epic> <child>` (subgraph delete, no `--force`); `mtt rm --status … --dry-run`
  (preview); an empty selector no-ops; a partial failure exits 1 with a report; two sources = usage error.
- **e2e:** `testscript` scenario `batch.txt` covering the above. ✅
- `make check` green. ✅

**Deviations / notes:**
- The adversarial spec review caught a **MAJOR**: cobra validates `Args` **before** `RunE`, so the old
  `idAndTags`/`ExactArgs(1)` would reject the bulk forms (`tag add y --status tbd`, `rm t1 t2`). Fixed to a
  marker-driven `PositionalArgs` (`tag`) and `ArbitraryArgs` (`rm`) before coding.
- `RemoveMany` keeps a per-id `store.Get` (not just the List snapshot) so `Remove`'s `load task %q` wording
  survives; the load-error path is untested (`memStore.Get` can't return a non-`ErrNotFound` error) — same
  limitation as before.
- testscript has **no shell pipes**: the e2e models the pipe via `cp stdout ids.txt` → `stdin ids.txt`; a
  txtar `badids.txt` unpacks into `$WORK` (not `$WORK/proj`), so it is referenced as `$WORK/badids.txt`.
