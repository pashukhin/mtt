# mtt — справочник CLI

> English version (primary): [CLI_REFERENCE.md](CLI_REFERENCE.md). Источник правды — английская версия.

Полная **целевая** поверхность команд CLI `mtt`, выведенная из [DESIGN.ru.md](DESIGN.ru.md). Документ
служит двум целям: справочник для людей и агентов, и способ проверить дизайн со стороны CLI (man/usage),
а не со стороны требований.

**Статус:** это проектная поверхность. Сегодня существует только `mtt version` (фаза 0). У каждой команды
указана фаза, в которой она появляется (см. план в [DESIGN.ru.md](DESIGN.ru.md#порядок-реализации)).

**Обозначения:** `<обязательно>`, `[опционально]`, `…` повторяемо. `<id>` — идентификатор задачи вида
`t17` — плоский, per-prefix (в YAML-адаптере). `<status>` — имя статуса из flow типа (напр. `tbd`,
`in_progress`, `done`, `cancelled`).

---

## Синопсис

```
mtt [глобальные флаги] <команда> [аргументы] [флаги]
```

`mtt` — stateless CLI: читает `.mtt/` (через сконфигурированный адаптер), применяет изменение, пишет
обратно. `mtt help [команда]` или `mtt <команда> -h` — встроенная справка.

---

## Глобальные флаги (единая семантика, доступны в любой команде)

| Флаг | Env | Значение |
|---|---|---|
| `--json` | — | Машиночитаемый JSON вместо человеческого текста. На мутации печатает получившийся объект; на запросе — набор результатов. По умолчанию выключено. Для агентов. **Реализовано (сессия 003)** в `show`/`list`/`edit`. |
| `--dir <path>` | `MTT_DIR` | Корень проекта с `.mtt/`. По умолчанию — ближайший родитель текущей директории, содержащий `.mtt/`. **Реализовано (сессия 003)**: `--dir`/`MTT_DIR` — явный корень (должен сам содержать `.mtt/`, без обхода вверх); если не задан — откат к обходу предков. |
| `--role <role>` | `MTT_ROLE` | Роль вызывающего (напр. `implementer`, `reviewer`). Пишется в `history` перехода задачи. Зарезервированный шов — на роутинг пока не влияет (см. DESIGN → Роли). *(ожидает — придёт вместе с enforcement flow, фаза 3)* |
| `-q, --quiet` | — | Подавить несущественный вывод (ошибки и запрошенные данные остаются). *(ожидает)* |
| `--no-color` | `NO_COLOR` | Отключить ANSI-цвет в человеческом выводе. *(ожидает)* |
| `-h, --help` | — | Справка по команде. |
| `--version` | — | Напечатать версию и выйти (как `mtt version`). **Реализовано (сессия 003)**. В отличие от остальных флагов этой таблицы, доступен только в корне (cobra `root.Version`): `mtt --version` работает, `mtt <подкоманда> --version` — нет. |

## Флаги переходов (общие для команд смены статуса: `status`, `advance`, `start`, `done`)

| Флаг | Значение |
|---|---|
| `--no-run` | Не исполнять `commands` перехода (обойти гейты/действия). Аварийно/для отладки. |
| `--stop` | **(по умолчанию, семейство advance)** Двигать до первого упавшего гейта или неоднозначной развилки; сообщить, где и почему остановились. |
| `--atomic` | Всё-или-ничего **по статусу**: если гейт упал — не менять статус и не писать переходы. Оговорка: сайд-эффекты уже выполненных команд не откатываются (шов отката/компенсации запланирован — см. DESIGN). |
| `--force` | Двигать/переходить безусловно, игнорируя гейты (обобщает `--no-run`, а на `status` перекрывает гейт одиночного ребра). |

`--stop`, `--atomic` и `--force` взаимоисключающи.

## Конфигурация

mtt сливает слои конфига, следующий перекрывает предыдущий: встроенные дефолты → опциональный глобальный
пользовательский конфиг (`$XDG_CONFIG_HOME/mtt/config.yaml`) → коммитящийся `.mtt/config.yaml` (общие
**типы и flow**) → gitignored `.mtt/config.local.yaml` (персональные параметры подключения и локальные
настройки) → env / флаги CLI. Креды к внешним бэкендам — в локальном оверлее или env, **никогда** в
коммитящемся конфиге. См. [DESIGN.ru.md](DESIGN.ru.md) → Конфигурация.

## Проект и мета

### `mtt init` — инициализировать проект  *(фаза 1)*
Создаёт `.mtt/` с дефолтным `config.yaml` (типы `epic`/`task`/`subtask`, flow `tbd → in_progress → done`
плюс терминал `cancelled`, без команд) и каталоги `tasks/` (позже `knowledge/`). Персональный gitignored
`.mtt/config.local.yaml` может его перекрыть (параметры подключения, локальные настройки — см. Конфигурация).

- `--force` — перезаписать существующий `config.yaml`.
- `--name <name>` — имя проекта в конфиге (по умолчанию — имя каталога).
- `--template <name>` — стартовый конфиг: `default` (epic/task/subtask, без команд) или `coding`
  (feature/bugfix/refactor, у каждого свой гейтящий per-type Definition of Done). По умолчанию: `default`.

### `mtt version` — напечатать версию  *(фаза 0, реализовано)*
Печатает версию сборки. Без аргументов.

### `mtt types` — показать типы и их flow  *(фаза 3)*
Перечисляет каждый тип задачи: его `parent`, статусы (с их `kind`) и переходы (с `description` и признаком
наличия `commands`).

- `[<type>]` — показать только этот тип.

### `mtt caps` — показать возможности текущего бэкенда  *(фаза 3)*
Печатает, какие возможности поддерживает активный адаптер (история, зависимости, дерево комментариев,
поиск, база знаний). Позволяет агенту не полагаться на отсутствующую у бэкенда возможность.

### `mtt completion <shell>` — скрипт автодополнения  *(встроено в cobra)*
Генерирует скрипт автодополнения для `bash`/`zsh`/`fish`/`powershell`.

## Задачи (CRUD)

### `mtt add [title] [flags]` — создать задачу  *(фаза 1, `add`/`show` доставлены в сессии 002)*
Создаёт задачу. Задай `title` (позиционно) и/или `--description`; нужно хотя бы одно. Адаптер минтит
ID — плоский, per-prefix, напр. `e1` или `t17` — и печатает `created <id>`.

- `--type <name>` — тип задачи из конфига (по умолчанию — тип, помеченный `default`).
- `--parent <id>` — разместить задачу под существующим родителем (сессия 004). Проверяется: родитель
  существует и его **тип** разрешён полем `parents` дочернего типа. Взаимоисключающ с `--no-parent`. *(реализовано)*
- `--no-parent` — создать требующий родителя тип на верхнем уровне (осознанное исключение). *(реализовано)*
- `--description <text>` — описание задачи (stdin через `--description -` запланирован).
- `--depends-on <id>…` и `--ref <kind>:<target>…` (напр. `note:auth-design`, `task:t2`) появятся позже.

Не-корневой тип без `--parent` и без `--no-parent` даёт ошибку и подсказывает, как поступить. Отсутствующий
родитель или родитель недопустимого типа тоже дают ошибку с подсказкой.

### `mtt show <id> [flags]` — показать задачу  *(фаза 1, реализовано; родословная в сессии 004)*
Печатает задачу: id, тип, статус, заголовок, **родословную** (цепочка родителей от корня, напр.
`lineage:  e1 › t1`, опускается для корневой задачи), временные метки и описание. Зависимости, ссылки и
**backlinks**, дерево комментариев и `history` переходов (аудит-трейл) появятся, когда доедут эти
возможности в следующих фазах.

- `<id>` — задача для показа.
- `--no-history` — *(позже)* без истории/аудита.
- `--no-comments` — *(позже)* без комментариев.

### `mtt list [flags]` — список задач  *(фаза 1, `--status`/`--type`/`--sort`/`--json` доставлены в сессии 003)*
Печатает задачи в стабильном порядке. Фильтры комбинируются по AND.

- `--status <status>…` — фильтр по имени статуса. *(реализовано)*
- `--kind <initial|active|terminal>…` — фильтр по категории статуса (сессия 004). *(реализовано)*
- `--type <type>…` — фильтр по типу задачи. *(реализовано)*
- `--parent <id>` — только прямые дети этой задачи (сессия 004). *(реализовано)*
- `--ready` — только готовые (без открытых блокеров) — сокращение для `mtt ready`. *(позже)*
- `--sort <created|updated>` — ключ сортировки; по умолчанию `created`, обе по убыванию, тай-брейк по ID.
  *(реализовано)*

### `mtt edit <id> [flags]` — правка не-flow полей  *(фаза 1, реализовано в сессии 003)*
Меняет заголовок и/или описание. **Статус здесь не редактируется** — смена статуса идёт через `status` /
`advance`, чтобы соблюдался flow. Смена родителя/типа — не простая правка (в YAML-адаптере это переминтило
бы ID — см. Заметки).

- `--title <text>` — новый заголовок.
- `--description <text>` — новое описание (`-` для stdin, тоже позже).

### `mtt tree [<id>] [flags]` — показать иерархию  *(сессия 004, реализовано)*
Печатает дерево эпик → задача → подзадача как ASCII-дерево (коннекторы `├─`/`└─`/`│`; узел —
`<id>  <type>  [<status>]  <title>`). Без `<id>` — лес от всех корней; с `<id>` — корень на этой задаче.
Дети **вычисляются** (инверсный индекс в `core`, не хранятся); порядок сиблингов детерминирован (`Created`
по убыванию, тай-брейк по ID). Сирота (задача, чей родитель отсутствует) показывается как корень, не теряется.

- `--status <status>…` / `--kind <initial|active|terminal>…` — фильтр узлов. Семантика **keep-ancestors**:
  узел виден, если совпал сам или любой потомок; несовпавшие предки остаются как путь к совпадению (так
  совпавший лист не теряется под несовпавшим родителем).
- `--depth <n>` — ограничить число видимых уровней, как `tree -L n` (`--depth 1` — только корни;
  `0`/без флага — без лимита).
- `--json` — **вложенное** дерево (`{…поля задачи…, "children": [ … ]}`); верхний уровень всегда JSON-массив
  (`[]` при пустоте, не `null`); пустые `children` у листьев опускаются.

## Flow (смена статуса)

### `mtt status <id> <status> [flags]` — одиночный переход  *(фаза 3)*
Двигает задачу по **одному** ребру в `<status>`, валидируя по `transitions` типа и исполняя `commands`
этого ребра (гейт). Падает, если переход не разрешён или гейт вернул не 0. Принимает флаги переходов
(`--no-run`, `--force`).

### `mtt advance <id> --to <status> [flags]` — обход до целевого статуса  *(фаза 3)*
Мета-команда: проводит задачу по цепочке переходов до `--to <status>`, исполняя гейты на рёбрах. Идёт
только по прогрессирующим рёбрам, не заходит в чужой терминал, стоп на настоящей развилке, защита от
циклов, ошибка если цель недостижима. Принимает все флаги переходов (по умолчанию `--stop`).

- `--to <status>` — целевой статус (обязателен).

### `mtt start <id> [flags]` — алиас: advance до `in_progress`  *(фаза 3)*
Эквивалент `mtt advance <id> --to in_progress`. Принимает флаги переходов.

### `mtt done <id> [flags]` — алиас: advance до `done`  *(фаза 3)*
Эквивалент `mtt advance <id> --to done`. Исполняет гейт `→ done` (напр. lint/test). По умолчанию
предупреждает, если задача не `ready` (открытые зависимости).

### `mtt cancel <id> [reason] [flags]` — перевести в терминал `cancelled`  *(фаза 3)*
Переводит задачу в `cancelled` (терминал, разблокирующий её зависимые). `[reason]` пишется в историю. Гейт
`done` не исполняется.

### `mtt ready [flags]` — список задач в работу  *(фаза 2)*
Перечисляет не-терминальные задачи, все блокеры которых в терминальном статусе (`done`/`cancelled`) — «что
можно брать дальше». Принимает фильтры `list`.

## Ссылки  *(поле: фаза 1; команды: фаза 2; цели `note` требуют KB, фаза 5)*

Ссылки — информационные проверяемые связи (`kind` ∈ `note`/`task`/`comment`/`url`), не блокирующие
зависимости. Ссылка идентифицируется естественным ключом — **парой `(kind, target)`** (отдельного id у
ссылки нет). Target — часть ключа, поэтому у сущности может быть много ссылок одного `kind` на разные цели
(`note:auth-design` + `note:login-spec` — две разные ссылки); схлопывается только точный дубль
`kind`+`target` (обновляется его `--label`). `--label` — аннотация, не часть идентичности.

### `mtt ref add <id> <kind>:<target> [--label <text>]` — добавить ссылку
Добавляет ссылку из задачи `<id>` на `<kind>:<target>` (напр. `note:auth-design`, `task:t2`).
Идемпотентно: повторное добавление того же ключа обновляет `--label`. При успехе печатает сохранённую
ссылку; если цель не резолвится (`note` без KB, отсутствующая задача) — ссылка всё равно сохраняется, но
помечается **unverified/dangling** с предупреждением (не жёсткая ошибка). С `--json` — печатает объект
ссылки `{kind, id, label, status}`.

### `mtt ref rm <id> <kind>:<target>` — удалить ссылку
Удаляет ссылку с этим ключом из задачи `<id>`. Выход `4`, если такой ссылки нет.

### `mtt ref list <id>` — ссылки и backlinks
Печатает исходящие ссылки задачи (каждая: `kind:target`, label, статус резолва
`ok`/`unverified`/`dangling`) и входящие **backlinks** — задачи/комментарии, ссылающиеся на эту.

### `mtt check [flags]` — верификация ссылок  *(фаза 5)*
Обходит репозиторий на висячие ссылки (цели, которых нет / нельзя резолвить). Capability-aware: `note`-
ссылки проверяемы только при наличии базы знаний.

- `--fix` — интерактивно удалить висячие ссылки (опционально).

## База знаний  *(фаза 5; capability `KnowledgeStore`)*

Без KB-бэкенда эти команды возвращают `ErrUnsupported`, а знания живут в задачах/комментариях.
**Заметки версионируются** — записи не разрушают старое содержимое; `edit` сохраняет новую версию со
ссылкой на предыдущую (см. DESIGN → База знаний). Внешние бэкенды используют своё нативное версионирование.

### `mtt note add <slug> [flags]` — создать заметку БЗ
Создаёт заметку в `<slug>` (первую версию). Отклоняет существующий slug — для новой версии используй `edit`.
- `<slug>` — стабильный идентификатор / имя файла.
- `--title <text>` — человеческий заголовок.
- `--body <text>` — содержимое (`-` для stdin).

### `mtt note edit <slug> [flags]` — сохранить новую версию
Сохраняет новую версию заголовка/тела заметки, **со ссылкой на предыдущую версию**; старые версии остаются.

### `mtt note show <slug> [--version <n>]` — напечатать заметку (с backlinks)
Показывает текущую версию, либо версию `<n>` с `--version`.

### `mtt note history <slug>` — список версий заметки
Перечисляет версии (сначала новые) с автором/временем; каждая ссылается на предшественника.

### `mtt note list` — список заметок

### `mtt search <query> [flags]` — текстовый поиск  *(фаза 5)*
Простой поиск по подстроке/токенам по задачам и заметкам (без RAG).

- `--tasks` / `--notes` — ограничить область.

## Отдельный бинарь: `mtt-ui`  *(фаза 7)*

**Опциональный** driving-адаптер (маленький локальный веб-сервер) поверх того же core — не часть
агентского бинаря. Не нужен при внешнем бэкенде с собственным UI.

```
mtt-ui [--addr <host:port>] [--dir <path>]
```
- `--addr <host:port>` — адрес прослушивания (по умолчанию `127.0.0.1:8080`).
- `--dir <path>` — корень проекта (как `--dir`/`MTT_DIR` выше).

## Коды возврата (предлагаемые)

Разные коды позволяют агентам ветвиться по результату без парсинга текста.

| Код | Значение |
|---|---|
| `0` | Успех |
| `1` | Общая ошибка |
| `2` | Ошибка использования (неверные флаги/аргументы) |
| `3` | Переход заблокирован — команда-гейт вернула не 0 |
| `4` | Не найдено (задача/заметка/цель не существует) |
| `5` | Не поддерживается — у активного адаптера нет нужной возможности (`ErrUnsupported`) |
| `6` | Недопустимый переход — не разрешён flow типа |

Эта расширенная таксономия пока **предлагаемая**: сессия 003 держит один общий код ошибки (`1`) для любого
пути отказа; коды `2`–`6` появятся вместе с поведением, которое они различают (валидация использования,
flow, capability-гейты, …).

## Заметки / наблюдения (из обзора со стороны CLI)

То, что вскрывает этот справочник и что стоит держать согласованным с дизайном:

- **Чистое разделение: `edit` vs flow-команды.** `edit` трогает только не-flow поля
  (заголовок/описание); всё движение статуса — через `status`/`advance`/`start`/`done`/`cancel`, чтобы
  flow всегда соблюдался.
- **`done` и `cancel` заменяют общий `close`.** Закрытие задачи = достижение терминала: `done` (со своим
  гейтом) или `cancel`. Отдельной команды `close` нет. *(В TASKS.md `close` ещё упомянут в фазе 1 —
  свести к `done`/`cancel`.)*
- **Смена родителя меняет только `parent`; смена типа — по-прежнему не `edit`.** ID плоские и per-prefix
  (не кодируют цепочку родителей), поэтому **смена родителя** (запланированная `mtt reparent`/`move`)
  меняет только поле `parent` — ID остаётся стабильным, без переминта и без поломки входящих ссылок.
  **Смена типа** — операция крупнее (префикс завязан на тип): она остаётся вне scope для `edit` — см.
  рекатегоризацию в DESIGN.ru.md.
- **Capability-gated команды.** `dep*`, `comment*`, `note*`, `search` и история опираются на опциональные
  возможности адаптера; против бэкенда без них — выход `5` (`ErrUnsupported`), а не молчаливый сбой.
- **`--json` везде.** Все команды поддерживают JSON-вывод, чтобы агенты рулили mtt без парсинга текста;
  мутации возвращают получившийся объект.
- **`--role` пишется, но не энфорсится.** Сейчас пишется в историю (не откладываемый шов); роут `start`/
  `done` по роли отложен.
