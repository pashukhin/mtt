# mtt — справочник CLI

> English version (primary): [CLI_REFERENCE.md](CLI_REFERENCE.md). Источник правды — английская версия.

Полная **целевая** поверхность команд CLI `mtt`, выведенная из [DESIGN.ru.md](DESIGN.ru.md). Документ
служит двум целям: справочник для людей и агентов, и способ проверить дизайн со стороны CLI (man/usage),
а не со стороны требований. Чтобы **собрать флоу** (типы/статусы/переходы + гейты/`post:`),
см. [FLOW_GUIDE.ru.md](FLOW_GUIDE.ru.md).

**Статус:** это целевая поверхность команд. **Реализовано сегодня (по сессию 008.98, `0.8.98-dev`):**
`version`, `init`, `types`, `add`, `show`, `list`, `edit`, `tree`, `dep add/rm/list`, `ready`, `roadmap`,
`status` (плюс verb-сахар `mtt <status> <id>`), `use`, `rm`, `tag add/rm` — с **селектором множества задач**
(явные id | stdin `-` | `--filter`) + выводом **`--ids`** у `list`/`ready`, питающими **массовые** `tag add/rm`
и `rm` (s008.9) — и встроенные cobra `completion`/`help`. Всё остальное
ниже — проектная поверхность, у каждой команды указана фаза/сессия появления (см. план в
[DESIGN.ru.md](DESIGN.ru.md#порядок-реализации)). Мета-обход `advance`/`start`/`done`/`cancel` — **PARKED**
(одно-рёберный `status` — норма; см. заметку в разделе «Flow» ниже).

**Обозначения:** `<обязательно>`, `[опционально]`, `…` повторяемо. `<id>` — идентификатор задачи вида
`t17` — плоский, per-prefix (в YAML-адаптере). `<status>` — имя статуса из flow типа (напр. `tbd`,
`in_progress`, `done`, `cancelled`).

---

## Синопсис

```
mtt [глобальные флаги] <команда> [аргументы] [флаги]
```

`mtt` — stateless CLI: читает `.mtt/` (через сконфигурированный адаптер), применяет изменение, пишет
обратно. `mtt help [команда]` или `mtt <команда> -h` — встроенная справка.

## Глобальные флаги (единая семантика, доступны в любой команде)

| Флаг | Env | Значение |
|---|---|---|
| `--json` | — | Машиночитаемый JSON вместо человеческого текста. На мутации печатает получившийся объект; на запросе — набор результатов. По умолчанию выключено. Для агентов. **Реализовано (сессия 003)** в `show`/`list`/`edit`. |
| `--dir <path>` | `MTT_DIR` | Корень проекта с `.mtt/`. По умолчанию — ближайший родитель текущей директории, содержащий `.mtt/`. **Реализовано (сессия 003)**: `--dir`/`MTT_DIR` — явный корень (должен сам содержать `.mtt/`, без обхода вверх); если не задан — откат к обходу предков. |
| `--role <role>` | `MTT_ROLE` | Роль вызывающего (напр. `implementer`, `reviewer`). Пишется в `history` перехода задачи. Зарезервированный шов — на роутинг пока не влияет (см. DESIGN → Роли). **Реализовано (сессия 006)** — пишется, не энфорсится. |
| `--by <subject>` | `MTT_BY` | Субъект действия («кто»), пишется в `history` перехода. Отдельно от `--role` («в какой роли»). Фолбэк: `MTT_BY`, затем `author` из `config.local.yaml` (долговременный персональный дефолт). **Реализовано (сессия 006)**. |
| `--who <subject>` | `MTT_BY` | Симметричный алиас `--by` (читается в паре с `--why`). **Взаимоисключающий** с `--by` (задайте только один). **Реализовано (сессия 006.5)**. |
| `--why <text>` | — | Долговременная свободная причина перехода, пишется в `history` и рендерится в `mtt show`. **Реализовано (сессия 006.5)**. |
| `-v, --verbose` | — | Стримить собственный вывод команды гейта в stderr (имеет смысл только на гейтящемся переходе). **Реализовано (сессия 006; root-persistent с 006.5)**. |
| `--log-file <path>` | — | Писать собственный вывод команды гейта в файл. **Реализовано (сессия 006; root-persistent с 006.5)**. |
| `-q, --quiet` | — | Подавить несущественный вывод (ошибки и запрошенные данные остаются). *(отложено — не реализовано; backlog t55. Для чистого машинного вывода — `--json`.)* |
| `--no-color` | `NO_COLOR` | Отключить ANSI-цвет в человеческом выводе. *(отложено — вывод сейчас монохромный; backlog t55.)* |
| `-h, --help` | — | Справка по команде. |
| `--version` | — | Напечатать версию и выйти (как `mtt version`). **Реализовано (сессия 003)**. В отличие от остальных флагов этой таблицы, доступен только в корне (cobra `root.Version`): `mtt --version` работает, `mtt <подкоманда> --version` — нет. |

## Флаги переходов (общие для команд смены статуса: `status`, `advance`, `start`, `done`)

| Флаг | Значение |
|---|---|
| `--no-run` | Не исполнять `commands` перехода (обойти гейты/действия — включая перемещающие состояние команды; см. «Семантика flow»). Аварийно/для отладки. **Форсит `--who`+`--why` (t5)** — гейт пропустить можно, но нужно подписаться (при отсутствии → exit 2). |
| `--stop` | **(по умолчанию, семейство advance)** Двигать до первого упавшего гейта или неоднозначной развилки; сообщить, где и почему остановились. |
| `--atomic` | Всё-или-ничего **по статусу**: если гейт упал — не менять статус и не писать переходы. Оговорка: сайд-эффекты уже выполненных команд не откатываются (шов отката/компенсации запланирован — см. DESIGN). |
| `--force` | Двигать/переходить безусловно, игнорируя гейты (обобщает `--no-run`, а на `status` перекрывает гейт одиночного ребра). |

`--stop`, `--atomic` и `--force` взаимоисключающи.

## Конфигурация

mtt сливает слои конфига, следующий перекрывает предыдущий: встроенные дефолты → опциональный глобальный
пользовательский конфиг (`$XDG_CONFIG_HOME/mtt/config.yaml`) → коммитящийся `.mtt/config.yaml` (общие
**типы и flow**) → gitignored `.mtt/config.local.yaml` (персональные параметры подключения и локальные
настройки) → env / флаги CLI. Креды к внешним бэкендам — в локальном оверлее или env, **никогда** в
коммитящемся конфиге. См. [DESIGN.ru.md](DESIGN.ru.md) → Конфигурация.

**`command_timeout`** (верхний уровень, напр. `command_timeout: 5m`) ограничивает каждую команду-гейт
перехода (на команду). Это исполнительная/адаптерная настройка (вне чистого домена), по умолчанию `5m` при
отсутствии, переопределяема через `config.local.yaml`. Команда может переопределить его своим **таймаутом на
команду** (см. «Команды перехода» ниже); глобальный — это fallback.

**Команды перехода (структурные — сессия 007; `rollback` — сессия 008).** `commands` перехода — это список,
где каждый элемент это либо **голая строка** (команда), либо **map** `{run, timeout, rollback}`:

```yaml
transitions:
  - from: tbd
    to: in_progress
    commands:
      - run: git checkout -b task/{{.ID}}
        rollback: git branch -D task/{{.ID}}  # отменить ЭТУ команду, если следующая упадёт
      - make test                             # поздний гейт; если упадёт — ветка удаляется
  - from: in_progress
    to: done
    commands:
      - make lint                      # голая строка — использует глобальный command_timeout
      - {run: make test, timeout: 30s} # таймаут на команду переопределяет глобальный
```

- **Плейсхолдеры** в `run` раскрываются до запуска гейта: `{{.ID}}`, `{{.Type}}`, `{{.From}}` (покидаемый
  статус), `{{.To}}` (целевой). Доступны только эти безопасные по форме поля — свободный текст
  (title/description) никогда не интерполируется; случайный `{{.Title}}` — ошибка. Запускается и пишется в
  `history` именно развёрнутая команда.
- **Показываемые описания/guidance раскрывают те же плейсхолдеры (t16).** `description` перехода или статуса,
  печатаемое как inline-guidance (при ходе и в `mtt show`, human **и** `--json`), тоже раскрывает
  `{{.ID}}`/`{{.Type}}`/`{{.From}}`/`{{.To}}` — но **best-effort**: некорректный шаблон или неизвестное поле
  показываются **raw** (вся строка целиком), так что guidance никогда не ломает команду. **Правило узла:** в
  описании **статуса** (узла) `{{.From}}`/`{{.To}}` оба раскрываются в имя самого статуса (статус — не переход;
  не ждите там семантики ребра/направления).
- **`timeout`** (Go-длительность вроде `30s`, `2m`) ограничивает эту команду, переопределяя `command_timeout`.
- **`rollback`** (сессия 008) — **компенсатор** этой команды, сам скаляр или `{run, timeout}` (те же
  плейсхолдеры). **Внутри-пайплайновая компенсация:** когда **следующая** команда того же пайплайна падает,
  компенсаторы уже успевших команд исполняются в **обратном порядке** (удалить ветку, созданную первой
  командой, …). Она **best-effort** (все компенсаторы прогоняются, продолжая сквозь упавший) и **не меняет
  исход** — переход всё равно **заблокирован** (exit `3`), задача остаётся на месте, `history` **не пишется**
  (файл задачи не тронут). Свой rollback упавшей команды **не** запускается. Гейт печатает живую фазу
  `↩ compensating (N)`, а сообщение блока добавляет `compensated N commands`. (Компенсация через несколько
  рёбер / `--atomic` пока не сделана.)
- `mtt types` печатает команду как `$ <run>` (`(timeout <d>)`, если есть) и, следующей строкой, `↩ <rollback>`,
  когда у команды объявлен компенсатор.

**`author`** (верхний уровень, обычно в gitignored `config.local.yaml`) — долговременный дефолт для поля
`by` в истории («кто действует»), применяется, когда не заданы ни `--by`, ни `MTT_BY` (прецедентность
`--by` > `MTT_BY` > `author`). Персональное — место в локальном оверлее, не в коммитящемся конфиге.

**`require`** (верхний уровень, в **коммитящемся** конфиге, напр. `require: {who: true, why: true}`) делает
`--who`/`--why` обязательными при смене статуса — проверяется **до** гейта, не байпасится `--no-run`;
`config.local` может только **ужесточать** (коммитнутое требование нельзя ослабить локально). Нарушение
агрегирует все недостающие поля в одну usage-ошибку (exit `2`). **Реализовано (сессия 006.5)**.

**`require` на переходе** (внутри записи `transitions[]`, напр. `require: {who: true, why: true}`) помечает
ребро **критическим**: форсит `--who`/`--why` только для этого перехода, объединяясь с глобальной политикой
(только ужесточение). **Реализовано (t5).**

**Опасные операции форсят `--who`/`--why` (t5).** Независимо от `require`, **обход гейта** (`--no-run`) форсит
**и who, и why** на любом ребре, а разрушительный **`mtt rm --force`** форсит who+why как pre-flight-проверку
(exit `2` при отсутствии, и для одиночного, и для bulk). `rm --force` также дописывает запись в
`.mtt/audit.log` (JSONL `{at, who, why, action, id}`, коммитится с `.gitattributes` `merge=union`) **до**
удаления.

**`post:` на переходе (t21).** У перехода может быть список `post:` (та же форма, что `commands:`),
выполняемый **после** записи статуса — фаза финализации. Это репо использует его для авто-коммита `.mtt` на
каждом move (`git add .mtt && git commit -m "{{.ID}}: {{.From}} → {{.To}}" -- .mtt`); рёбра `deliver`/`cancel`
коммитят на `main`, поэтому **сужают** этот add-pathspec до файла задачи (+`.mtt/audit.log`, если он есть) —
грязный `.mtt/config.yaml` не должен уехать в коммит на main мимо ревью (c3). И (c1) для **авто-пуша**:
`approve` также запускает `git push -u origin task/{{.ID}}` (ветка задачи), затем открывает/обновляет PR (`gh
pr create`, идемпотентно — нужны `gh`+`jq`; тело из `docs/superpowers/pr/{{.ID}}.md` если есть, иначе
сгенерированное; t27); `deliver`/`cancel` — `git push origin main` (доставленное или отменённое терминальное состояние не должно жить только локально — c5). Семантика провала иная,
чем у гейта: провал `post:` **сохраняет** move (статус уже записан) и даёт exit **5** (провал `commands:`-гейта
→ exit 3, статус не изменён). `--no-run` пропускает **обе** фазы (`commands:` и `post:`) — включая любые
**перемещающие состояние** команды, положенные автором флоу на ребро (`git switch`, шаг деплоя): обойдённое
контекст-переключающее ребро запишет состояние там, где дерево находится сейчас. Опасен никогда не флаг —
опасны команды на ребре, и их автор отвечает за них одинаково при исполнении и при обходе. Показывается в
`mtt types` строкой `⇢` под ребром.

## Проект и мета

### `mtt init` — инициализировать проект  *(фаза 1, реализовано в сессии 001)*
Создаёт `.mtt/` с дефолтным `config.yaml` (типы `task`/`chore`, flow `tbd → in_progress → done`
плюс терминал `cancelled`, без команд) и каталоги `tasks/` (позже `knowledge/`). Персональный gitignored
`.mtt/config.local.yaml` может его перекрыть (параметры подключения, локальные настройки — см. Конфигурация).
Также кладёт `.mtt/.gitignore`, игнорирующий этот оверлей `config.local.yaml`, чтобы он никогда не стал
коммитабельным (create-if-absent: существующий `.gitignore` не перезаписывается, даже с `--force`).
Проектам, инициализированным раньше, нужно добавить его вручную: `echo config.local.yaml > .mtt/.gitignore`.

- `--force` — перезаписать существующий `config.yaml`.
- `--name <name>` — имя проекта в конфиге (по умолчанию — имя каталога).
- `--template <name>` — стартовый конфиг: `default` (плоские `task`+`chore`, без команд), `coding`
  (feature/bugfix/refactor, у каждого свой гейтящий per-type Definition of Done) или `hierarchy`
  (epic/task/subtask, без команд). По умолчанию: `default`.
- `--json` — эмитит сводку созданного конфига `{path, template, name, created}` (абсолютный `path`) вместо
  человекочитаемой строки. *(t45)*

Любая другая команда вне проекта (`.mtt/` не найден при discovery или `--dir` без него) падает с подсказкой
`run 'mtt init' to create one` (сессия 008.97/U4).

### `mtt version` — напечатать версию  *(фаза 0, реализовано)*
Печатает версию сборки. Без аргументов. `--json` эмитит `{"version": …}` — как и корневой флаг `--version`
(`mtt --version --json`), так что флаг и команда согласованы. *(t45)*

### `mtt types` — показать типы и их flow  *(реализовано в сессии 001; детали flow/команд росли до s008)*
Перечисляет каждый тип задачи: его `parent`, статусы (с их `kind`) и переходы (с `description` и признаком
наличия `commands`).

- `[<type>]` — показать только этот тип.
- `--json` — эмитит граф флоу: массив `{name, prefix, parents, default?, description?, statuses:[{name, kind,
  default?, description?}], transitions:[{name?, from, to, description?, current?, require?, commands:[{run,
  timeout?, rollback?}], post:[…]}]}` (структурные массивы non-null; `mtt types <type> --json` → массив из
  одного элемента; неизвестный тип → код 1). *(t45)*

### `mtt caps` — показать возможности текущего бэкенда  *(фаза 3 — ещё не реализовано; проектная поверхность)*
Печатает, какие возможности поддерживает активный адаптер (история, зависимости, дерево комментариев,
поиск, база знаний). Позволяет агенту не полагаться на отсутствующую у бэкенда возможность.

### `mtt completion <shell>` — скрипт автодополнения  *(встроено в cobra)*
Генерирует скрипт автодополнения для `bash`/`zsh`/`fish`/`powershell`.

## Задачи (CRUD)

### `mtt add [title] [flags]` — создать задачу  *(фаза 1, `add`/`show` доставлены в сессии 002)*
Создаёт задачу. Задай `title` (позиционно) и/или `--description`; нужно хотя бы одно. Адаптер минтит
ID — плоский, per-prefix, напр. `e1` или `t17` — и печатает `created <id>` (или, с `--json`, объект
созданной задачи — сессия 008.97). **Заголовок должен быть однострочным** — перевод строки даёт
usage-ошибку (c13); многострочный текст — в `--description`.

- `--type <name>` — тип задачи из конфига (по умолчанию — тип, помеченный `default`).
- `--parent <id>` — разместить задачу под существующим родителем (сессия 004). Проверяется: родитель
  существует и его **тип** разрешён полем `parents` дочернего типа. Взаимоисключающ с `--no-parent`. *(реализовано)*
- `--no-parent` — создать требующий родителя тип на верхнем уровне (осознанное исключение). *(реализовано)*
- `--description <text>` — описание задачи (stdin через `--description -` запланирован).
- `--priority <high|medium|low>` — приоритет планирования задачи (сессия 008.6). По умолчанию **не задан**
  (сортируется как `medium`, на диск ничего не пишется). Неизвестное значение — ошибка использования.
  **Реализовано (сессия 008.6)**.
- `--depends-on <id>…` — задать блокирующие зависимости при создании (повторяемый, через запятую). Каждая
  цель должна существовать (иначе `add` завершается ошибкой и задача не создаётся); валидация в `core.Adder`.
  **Реализовано (сессия 008.5)**.
- `--ref <kind>:<target>` — прикрепить информационную ссылку при создании (повторяемый; `note:auth-design`/
  `task:t2`/`url:https://…`). warn-not-block: висячая цель сохраняется с предупреждением (выход 0). **t1.**
- `--tag <tag>…` — добавить тег (повторяемый или через запятую `--tag a,b`, сессия 008.7). `#хэштеги` в title/description тоже извлекаются
  и мёржатся в тот же набор (`#fragment` внутри `scheme://`-URL пропускается — вставленная ссылка не минтит
  теги, c13). Значения нормализуются (Unicode-нижний регистр над буквами/цифрами плюс `. _ -`,
  любой скрипт; допускается ведущий `#`); символ вне charset — ошибка использования. **Реализовано (сессия
  008.7)**.

Не-корневой тип без `--parent` и без `--no-parent` даёт ошибку и подсказывает, как поступить. Отсутствующий
родитель или родитель недопустимого типа тоже дают ошибку с подсказкой.

### `mtt show [<id>] [flags]` — показать задачу  *(фаза 1, реализовано; родословная в сессии 004; опущенный id → current в 006.7; приоритет в 008.6)*
Печатает задачу: id, тип, статус, заголовок, **приоритет** (строка `priority:`, показывается только когда
задан — сессия 008.6), **теги** (строка `tags:` — отсортированный набор, показывается только когда непусто —
сессия 008.7), **подсказку по флоу** для текущего статуса (сессия 008.95 — `description` статуса строкой `▸ …`,
когда задано, и **следующие ходы** `next: <статус> (<описание перехода>) · …`, когда статус не терминальный),
**зависимости** (строка `depends:`, перечисляющая каждый блокер из `depends_on` с его статусом —
`depends:  t2 [tbd], t3 [done ✓], t9 (missing)` — терминальный (удовлетворённый) блокер помечен `✓`,
висячий — `(missing)`; показывается только когда блокеры есть — c12),
**родословную**, сводку **потомков**, временные метки и
описание. Родословная — путь «ты здесь» от корня **до самой задачи включительно** (`lineage:  e1 › t1 › s1`),
показывается только когда у задачи есть родитель; у корня её нет. Строка потомков перечисляет прямых детей
(`children: 2 (t1, t2)`), показывается только когда они есть. Отдельной строки `parent:` нет — родитель это
предпоследний элемент родословной. Ссылки и **backlinks** и `history` переходов (аудит-трейл) тоже
рендерятся (см. ниже); дерево комментариев появится вместе с комментариями (t2).

- `<id>` — задача для показа.
- `--json` — объект задачи плюс **`status_description`** (omitempty), **`next`** (массив `{name?, to, description?}`
  следующих ходов, omitempty — `name` это глагол ребра для `mtt do`, сессия 008.98) и **`history`** (массив записей перехода
  `{at, by?, role?, why?, from, to, checks?:[{cmd, exit}]}`, omitempty — сессия 008.97/U3, чтобы JSON-потребитель
  видел checks + атрибуцию, которые рендерит человекочитаемый вид) — структурированная, запрашиваемая форма
  подсказки + аудита (сессия 008.95). Общий `--json` для `list`/`edit` не затронут (эти поля только у `show`).
- `--no-history` — *(позже)* без истории/аудита.
- `--no-comments` — *(позже)* без комментариев.

### `mtt list [flags]` — список задач  *(фаза 1, `--status`/`--type`/`--sort`/`--json` доставлены в сессии 003)*
Печатает задачи в стабильном порядке. Фильтры комбинируются по AND.

- `--status <status>…` — фильтр по имени статуса. *(реализовано)*
- `--kind <initial|active|terminal>…` — фильтр по категории статуса (сессия 004). *(реализовано)*
- `--type <type>…` — фильтр по типу задачи. *(реализовано)*
- `--parent <id>` — только прямые дети этой задачи (сессия 004). *(реализовано)*
- `--priority <high|medium|low>…` — фильтр по приоритету (сессия 008.6, повторяемый). Совпадает по
  **сохранённой** метке: задача без приоритета попадает только когда `--priority` не задан. *(реализовано)*
- `--tag <tag>…` — фильтр по тегу (сессия 008.7, повторяемый или через запятую; на `list`, `ready` и `tree`). **OR внутри**
  измерения (задача совпадает, если несёт **любой** из указанных тегов), AND с остальными фильтрами. Значения
  нормализуются как `--tag` у `add`. *(реализовано)*
- `--exclude-tag <tag>…` — **негативный** фильтр (c8, повторяемый или через запятую, на `list`, `ready` **и** `tree` — `tree`
  добавлен в c10; с c16 также на селекторном `--filter` у `rm`/`tag`): отбросить задачу, несущую **любой** из
  этих тегов. Композиция с `--tag` — AND, при
  пересечении exclude побеждает. Напр. `mtt ready --exclude-tag backlog` де-нойзит очередь. *(реализовано)*
- `--ready` — только готовые (без открытых блокеров) — сокращение для `mtt ready`. *(реализовано, сессия 005)*
- `--sort <created|updated|priority>` — ключ сортировки; по умолчанию `created`. `created`/`updated` — по
  убыванию, тай-брейк по ID; `priority` (сессия 008.6) сортирует high→low (без приоритета — в полосе medium),
  тай-брейк по свежести. *(реализовано)*
- `--ids` — печатать только id задач, по одному на строку, для конвейеров (сессия 008.9). Учитывает фильтры и
  `--sort`; **взаимоисключающий с `--json`**. Напр. `mtt list --tag x --ids | mtt tag rm x -`. *(реализовано)*

### `mtt edit [<id>] [flags]` — правка не-flow полей  *(фаза 1, реализовано в сессии 003; опущенный id → current в 006.7; приоритет в 008.6)*
Меняет заголовок, описание и/или приоритет. **Статус здесь не редактируется** — смена статуса идёт через `status` /
`advance`, чтобы соблюдался flow. Смена родителя/типа — не простая правка (в YAML-адаптере это переминтило
бы ID — см. Заметки).

- `--title <text>` — новый заголовок (однострочный; перевод строки — usage-ошибка, многострочный текст — в `--description`, c13).
- `--description <text>` — новое описание (`-` для stdin, тоже позже).
- `--priority <high|medium|low>` — новый приоритет (сессия 008.6). `--priority ""` **сбрасывает** его в
  «не задан». Неизвестное значение — ошибка использования. *(реализовано)*
- **Теги согласуются при правке текста** (сессия 008.7): правка `--title`/`--description` пере-извлекает
  `#хэштеги` — тег, чей `#хэштег` ушёл из текста, снимается, новый добавляется, а ручные теги (из `mtt tag
  add`) сохраняются. Флага `--tag` у `edit` нет; точечные изменения тегов — через `mtt tag add/rm`.

### `mtt rm [<id>...] [-] [--force]` — удалить задачи (hard delete)  *(сессия 008.5; массовый + селектор в 008.9)*
Безвозвратно удаляет задачи (в отличие от `cancel` — терминального *статуса*, а не удаления). `rm` — для
гигиены бэклога (снести ошибочную/устаревшую задачу). Для удаления **нет history** (файл удалён); git-коммит,
убирающий `.mtt/tasks/<id>.yaml`, плюс запись в `.mtt/audit.log` от `--force` (см. ниже), — аудит-след.

**`--force` форсит атрибуцию (t5).** Поскольку удаляет задачу вопреки входящим ссылкам, `mtt rm --force`
требует **и** `--who`, **и** `--why` (при отсутствии → **exit 2**, ничего не удалено — pre-flight-проверка и
для одиночного, и для bulk) и дописывает `{at, who, why, action: "rm --force", id}` в `.mtt/audit.log` **до**
каждого удаления. Обычный `rm` (без `--force`) не изменился (reject-if-referenced; без атрибуции и аудита).

**Селектор множества задач (сессия 008.9).** `rm` берёт набор из **одного** из трёх взаимоисключающих
источников: явные id (`mtt rm t1 t2`), stdin `-` (id построчно — `mtt list … --ids | mtt rm -`), или
`--filter` (`--status/--type/--kind/--parent/--priority/--tag/--exclude-tag/--ready` — `mtt rm --status
cancelled`; `--exclude-tag` с c16, так что `mtt rm --type task --exclude-tag keep` де-нойзит, щадя
помеченные задачи). Больше
одного источника или ни одного — usage error; источник, который ничего не выбрал, — no-op (exit 0). **Один
явный id** сохраняет прежнее одиночное поведение (ниже); удаление multi-id / `-` / `--filter` — **массовое**:
best-effort по задачам, сводка `removed N task(s): …` на stdout, отказы по задачам на stderr, **exit 1** при
любом отказе. `--dry-run` показывает затронутые id (по одному на строку) без удаления. Массовый `rm`
**учитывает подграф** — проверка reject-if-referenced игнорирует рефереров, которые **сами в наборе
удаления**, поэтому `mtt rm <epic> <child>…` сносит всё поддерево одной командой **без** `--force`.

- Требует **явный `<id>`** для одиночной формы — `rm` **не** резолвит указатель текущей задачи (деструктивная операция берёт
  явную цель). Если удалённая задача была текущей — указатель очищается.
- По умолчанию `rm` **отклоняется**, если задача **referenced** — другая задача `depends_on` её, или у неё
  есть дети (`parent` указывает на неё) — с перечислением ссылающихся id. Так удаление не оставляет молча
  висячие ссылки (код `1`).
- `--force` — удалить всё равно, оставив ссылки **висячими** (система их терпит: `ready` консервативен —
  висячий блокер оставляет зависимую не-ready — а `tree` поднимает сирот в корни). **Оговорка (переиспользование
  id):** YAML-адаптер минтит id как `max+1` на префикс, поэтому удаление задачи с **наибольшим** номером
  освобождает её id — последующий `add` может его **переиспользовать**, молча перенаправив висячую ссылку на
  новую (неродственную) задачу. Для referenced-задачи предпочитайте `cancel` вместо `rm --force` или сначала
  снимите висячие рёбра. Фикс монотонного/never-reuse минтинга — в t10 (мультиагентный кластер).
- Отсутствующий `<id>` — код `4` (not found). При успехе печатает `removed <id>`; с `--json` одиночная форма
  эмитит **объект удалённой задачи** (снятый до удаления, как `add --json`), а массовый `--json` остаётся
  массивом исходов по элементам. *(t45)*

### `mtt tree [<id>] [flags]` — показать иерархию  *(сессия 004, реализовано)*
Печатает дерево задач (родитель → ребёнок) как ASCII-дерево (коннекторы `├─`/`└─`/`│`; узел —
`<id>  <type>  [<status>]  <title>`). Без `<id>` — лес от всех корней; с `<id>` — корень на этой задаче.
Дети **вычисляются** (инверсный индекс в `core`, не хранятся); порядок сиблингов детерминирован (`Created`
по убыванию, тай-брейк по ID). Сирота (задача, чей родитель отсутствует) показывается как корень, не теряется.

- `--status <status>…` / `--type <type>…` / `--kind <initial|active|terminal>…` / `--priority <high|medium|low>…`
  / `--tag <tag>…` / `--exclude-tag <tag>…` / `--parent <id>` — фильтр
  узлов (`--tag` — сессия 008.7, OR-внутри; `--exclude-tag` — c10, негативный; `--type`/`--priority`/`--parent`
  закрывают паритет с набором фильтров `list` — c16). Семантика **keep-ancestors**:
  узел виден, если совпал сам или любой потомок;
  несовпавшие предки остаются как путь к совпадению (так совпавший лист не теряется под несовпавшим родителем).
- `--depth <n>` — ограничить число видимых уровней, как `tree -L n` (`--depth 1` — только корни;
  `0`/без флага — без лимита).
- `--json` — **вложенное** дерево (`{…поля задачи…, "children": [ … ]}`); верхний уровень всегда JSON-массив
  (`[]` при пустоте, не `null`); пустые `children` у листьев опускаются.

### `mtt tag add|rm <id> <tag>... | <tag>... (- | --filter)` — управление тегами  *(сессия 008.7; массовый в 008.9)*
Теги — сквозные лейблы. **Главный** способ пометить — `#хэштег` в title/description (извлекается при
`add`/`edit`; `#fragment` внутри `scheme://`-URL пропускается — c13); `mtt tag add/rm` — вторичный, точечный путь. Оба берут **один или несколько** тегов (вариадик),
так что весь набор меняется за одну запись. Теги хранятся как нормализованный, дедуплицированный,
**отсортированный** набор и едут на `Task.Tags` (без нового порта — как `depends_on`).

**Массово над набором задач (сессия 008.9).** Раскладка аргументов **контекстно-зависима**: при маркере
селектора — `-` (id из stdin) или `--filter`-флаг (`--status/--type/--kind/--parent/--priority/--tag/--exclude-tag/--ready`)
— **позиционные становятся тегами** и применяются к каждой выбранной задаче (`mtt tag add urgent --status
tbd`, `mtt list --tag x --ids | mtt tag rm x -`). Без маркера — **одиночная** форма `mtt tag add <id> <tag>…`
(без изменений). Массовый режим — best-effort по задачам (сводка `tagged/untagged N task(s): …` на stdout,
отказы по задачам — напр. гуард `rm` — на stderr, exit 1 при любом отказе); `--dry-run` показывает затронутые
id. У `tag` флаг `--tag` — это **фильтр** (выбрать задачи с тегом), в отличие от позиционных тегов, которые
добавляются/снимаются.

- `mtt tag add <id> <tag>...` — добавить теги (идемпотентно: повторное добавление существующего тега не пишет
  ничего). Печатает `tagged <id>: <added>` (только реально добавленные теги; при no-op —
  `<id>: already tagged <tags> (no change)`) или объект задачи с `--json`.
- `mtt tag rm <id> <tag>...` — снять теги. **Защита:** тег, чей `#хэштег` ещё присутствует в title или
  description, **отклоняется** (`cannot remove tag "x": #x is present in the title …`) — правьте текст, чтобы
  его убрать (защита верна принципу «текст авторитетен», обхода нет). Защита проверяется для **всех** целей до
  любого изменения, так что мультитеговый вызов атомарен; снятие отсутствующего тега — no-op. Печатает
  `untagged <id>: <removed>` (только реально снятые теги; при no-op — `<id>: no such tag <tags> (no change)`)
  или объект задачи с `--json`.
- Отсутствующий `<id>` — код `4` (not found); защита `rm` — обычная ошибка (код `1`).
- Значения тегов нормализуются: Unicode-нижний регистр над буквами/цифрами плюс `. _ -` (любой скрипт —
  `#Бэкенд` → `бэкенд`), с необязательным ведущим `#`. Пробелы или иные символы — ошибка использования.
  (Сравнение по code points после нижнего регистра; NFC-фолдинга **нет**.)

### `mtt tags [flags]` — словарь тегов со счётчиками  *(c9, реализовано)*
Только чтение: считает теги по учитываемым задачам и печатает строки `count  tag`, самые частые сверху (count
по убыванию, затем tag по возрастанию). Область по умолчанию — **открытые** задачи (виды `initial`+`active`) —
голая команда показывает живой словарь. **Статус-скоуп**-флаг задаёт область явно и снимает open-дефолт:
`--all` считает все задачи; явный `--status`/`--kind` скоупит на них (так `mtt tags --status done` доходит до
терминальных). Остальные фильтры `list` (`--type`/`--priority`/`--tag`/`--exclude-tag`/`--parent`) сужают
**внутри** области. `--json` выдаёт массив `[{tag, count}]`. Заменяет обходной `mtt list … | jq … | uniq -c`.

## Flow (смена статуса)

### `mtt status [<id>] <status> [flags]` — одиночный переход  *(сессия 006, реализовано; опущенный id в 006.7)*
Двигает задачу по **одному** ребру в `<status>`, валидируя по `transitions` типа и исполняя `commands`
этого ребра (гейт: все `0`, иначе переход **заблокирован** — код `3` — и задача не меняется, история не
пишется). На успехе добавляет запись в `history` (`from→to`, `at`, `by`/`role`/`why` из
`--who`/`--by`/`--role`/`--why`, `checks`) и печатает `t1: tbd → in_progress` (плюс строку на каждый check),
либо объект задачи с `--json`. Переход вне flow даёт код `6`. Если политика `require` не удовлетворена —
код `2` **до** запуска гейта (см. Конфигурация → `require`).

Гейт печатает **live-прогресс пайплайна** в stderr (`▶ <cmd>` / `✓|✗ <cmd> (exit N, <elapsed>)`) по мере
выполнения; вывод самих команд по умолчанию скрыт. При **блоке** последние ~10 строк вывода упавшей команды
печатаются под её строкой `✗`, а ошибка несёт `hint: re-run with -v or --log-file …` (сессия 008.97) — так
агент видит, *почему* гейт упал, без повторного прогона; для полного вывода — `-v`/`--log-file`.

После успешного перехода mtt печатает **подсказку по флоу** в stdout под строкой `from → to` (сессия 008.95):
`description` пройденного ребра и `description` целевого статуса (каждый строкой `▸ …`, только когда задано) и
**следующие ходы** (`next: <статус> (<описание перехода>) · …`; для терминала опускается). Так авторский текст
флоу становится **инлайновой инструкцией агенту** — зачем переход, что делать сейчас и куда можно дальше.
**Заблокированный** переход подсказку не печатает (задача не вошла в статус). С `--json` вывод перехода
остаётся объектом задачи; структурированную подсказку запрашивай через `mtt show --json` (выше).

- `--no-run` — пропустить `commands` ребра (байпас гейта). Локальный у `mtt status` (сахар не может обойти
  гейт); required-атрибуцию **не** байпасит. *(реализовано)*
- `-v`, `--verbose` / `--log-file <path>` — управление выводом гейта (root-persistent глобальные флаги). *(реализовано)*
- `--force` — *(пока нет — придёт с семейством advance, s007)*

#### Verb-сахар: `mtt <status> <id>`  *(сессия 006.5, реализовано)*
Сокращение для одноребёрного перехода: `mtt in_progress t1` ≡ `mtt status t1 in_progress` (обратите внимание
на **обратный** порядок аргументов — `<status> <id>`). Разрешается через **fallback-роутинг**, а не
регистрацию команды: при ровно двух аргументах, где первый — не реальная подкоманда, `<id>` — существующая
задача, а `<status>` — статус во flow её типа, mtt маршрутизирует в путь `status` (переиспользуя всю
валидацию, гейты, коды выхода и `--who`/`--why`). Реальная команда всегда выигрывает клеш имён. Если
`<status>` — правдоподобный статус-верб, а `<id>` не существует, это **not-found** (код `4`, как у явной формы
`mtt status`); первый аргумент — известное **имя ребра** в типе задачи, но недопустимое из её текущего
статуса — маршрутизируется в actionable-ошибку `mtt do` (код `6`, перечисляет доступные действия — c14); всё
прочее, что не классифицируется как переход статуса, — `unknown command` (код `1`). Сахар не принимает гейт-флаги
(`--no-run`/`-v`/`--log-file` остаются у `mtt status`); он forward-compatible — семантику можно нарастить
single-edge → `advance` позже без изменения поверхности.

### `mtt do [<id>] <edge>` — переход по именованному ребру  *(сессия 008.98, реализовано)*
Двигает задачу по **именованному ребру**, выходящему из её текущего статуса — явная форма работы *в терминах
текущего статуса* (семантический глагол вроде `decline`/`approve`), симметрично `mtt status` для перехода по
имени статуса-цели. У перехода может быть опциональное **`name:`** в конфиге; `mtt do <edge>` находит ребро с
этим именем из текущего статуса задачи и двигает в его цель, с тем же гейтом/атрибуцией/`--json`, что и
`mtt status` (общий путь). id опционален (резолвится в текущую задачу). **Только** по имени ребра — без отката
на статус; неизвестное ребро из текущего статуса даёт код `6` (invalid transition) и **перечисляет доступные
именованные действия**. `--no-run` обходит гейт (как у `status`). `mtt types` показывает имя каждого ребра.

- **Verb-сахар `mtt <edge> [<id>]`** (сессия 008.98) — короткая форма `mtt do`, зеркалит `mtt <status>`:
  `mtt decline t1` ≡ `mtt do t1 decline`. Тем же fallback-роутингом; имя ребра из текущего статуса пробуется
  **раньше** имени статуса-цели (безопасно — см. правила пространств имён ниже).
- **Правила пространств имён (энфорсит `Config.Validate`, т.е. на `mtt add`/`mtt types`):** `name` ребра
  уникален в пределах исходного статуса, **дизъюнктен с именами статусов** типа (сахар не затеняет статус), и
  каждая пара `(from,to)` уникальна в типе.

### `mtt use [<id>] [--clear]` — текущая задача (рабочий контекст)  *(сессия 006.7, реализовано)*
git-`HEAD`-для-задач: персональный указатель **текущей задачи** (в `config.local.yaml`, gitignored), чтобы не
повторять id.
- `mtt use <id>` — задать текущую задачу (id должен существовать). Печатает `current: <id>` (или задачу с `--json`).
- `mtt use` — показать текущую задачу одной строкой (или `no current task`).
- `mtt use --clear` — сбросить указатель (печатает `current cleared`).

С `--json` любая форма эмитит объект текущей задачи или `null`, если её нет — так что `mtt use --clear --json`
и `mtt use --json` без текущей печатают `null`. *(t45)*

**Резолвинг опущенного id:** если опустить id у **одиночного прямого глагола** — `mtt status <status>`, сахар
`mtt <status>` (напр. `mtt done`), `mtt show`, `mtt edit` — mtt берёт текущую задачу. Порядок: **явный id >
current**; протухший или незаданный current даёт внятную ошибку. **Никогда** не применяется к
`list`/`tree`/`dep`/`ready` (операции над множеством). Полный цикл читается так: `mtt use t1` →
`mtt in_progress` → … → `mtt done` (id не повторяется).

**Движение указателя через flow:** коммитящийся переход может нести `current: set` (взять в работу) или
`current: clear` (отпустить); mtt применяет это после перехода. Шаблоны `default`/`coding` ставят `set` на
`→ in_progress` и `clear` на `→ done` (не трогая `→ cancelled`), так что `mtt in_progress t1` делает `t1`
текущей, а `mtt done` — сбрасывает. (Хранение указателя — capability: YAML-адаптер пишет `config.local`;
внешний адаптер может замаппить на нативного assignee.)

> **PARKED (по требованию) — семейство `advance` НЕ реализовано.** Большинство переходов — одно ребро,
> поэтому `mtt status` и сахар `mtt <status> <id>` — норма; много-рёберный обход (и режимы `--stop`/
> `--atomic`/`--force`) всплывёт, только когда flow реально ветвится. Четыре секции ниже — **целевая
> семантика** на тот момент. Учтите: `mtt done t1` работает *сегодня* как одно-рёберный verb-сахар (переход
> в статус `done` за одно ребро), а не как этот мета-обход `advance`. См. DESIGN.ru.md → «Продвижение по flow».

### `mtt advance <id> --to <status> [flags]` — обход до целевого статуса  *(фаза 3, PARKED)*
Мета-команда: проводит задачу по цепочке переходов до `--to <status>`, исполняя гейты на рёбрах. Идёт
только по прогрессирующим рёбрам, не заходит в чужой терминал, стоп на настоящей развилке, защита от
циклов, ошибка если цель недостижима. Принимает все флаги переходов (по умолчанию `--stop`).

- `--to <status>` — целевой статус (обязателен).

### `mtt start <id> [flags]` — алиас: advance до `in_progress`  *(фаза 3, PARKED)*
Эквивалент `mtt advance <id> --to in_progress`. Принимает флаги переходов.

### `mtt done <id> [flags]` — алиас: advance до `done`  *(фаза 3, PARKED — но `mtt done <id>` работает сегодня как verb-сахар)*
Эквивалент `mtt advance <id> --to done`. Исполняет гейт `→ done` (напр. lint/test). По умолчанию
предупреждает, если задача не `ready` (открытые зависимости).

### `mtt cancel <id> [reason] [flags]` — перевести в терминал `cancelled`  *(фаза 3, PARKED — `mtt cancelled <id>` работает сегодня как verb-сахар)*
Переводит задачу в `cancelled` (терминал, разблокирующий её зависимые). `[reason]` пишется в историю. Гейт
`done` не исполняется.

### `mtt ready [flags]` — список задач в работу  *(сессия 005, реализовано)*
Перечисляет не-терминальные задачи, все блокеры которых в терминальном статусе (`done`/`cancelled`) — «что
можно брать дальше». Принимает фильтры `list` (`--status`/`--type`/`--kind`/`--priority`/`--parent`/`--tag`/
`--exclude-tag` — `--priority` закрыл паритет в c16),
`--json` и `--ids` (сессия 008.9; id построчно, взаимоисключающий с `--json`). `--tag`/`--exclude-tag <tag>…`
(повторяемые; `--exclude-tag` из c8, `--tag` на `ready` добавлен в c10) включают/де-нойзят очередь, напр.
`mtt ready --tag urgent`, `mtt ready --exclude-tag backlog`.
Готовность **консервативна**: dangling-блокер или статус не из текущего flow оставляют задачу не готовой
(`mtt list --ready` — то же подмножество через `list`).

### `mtt roadmap [--json]` — вид «порядок исполнения»  *(сессия 008.6, реализовано)*
Печатает **не-терминальные** задачи в порядке исполнения, каждая с аннотацией: готова ли она сейчас (`ready`),
что её ещё блокирует (`blocked_by`) и — для предка — что она содержит (`contains`). Это вид «что делать дальше
и чего оно ждёт» — то, что порознь упускают `ready` (разблокировано *сейчас*, плоско) и `list --sort priority`
(без порядка по зависимостям, только свой приоритет). **Не** временнóй планировщик: без дат и критического пути.

Порядок строится по **двум осям «идти после»**, обе — **жёсткие**: `depends_on` (явное блокирующее ребро —
не-терминальный блокер идёт перед задачей, которую блокирует) **и `parent`** (предок закрывается только когда
закрыты потомки, поэтому не-терминальный потомок идёт перед предком). Приоритет — **мягкий** тай-брейк, и он
**распространяется**: блокер наследует наивысший приоритет всего, что он (транзитивно) разблокирует, — так
высокоприоритетная задача тянет свои предпосылки вперёд, впереди независимой низкоприоритетной работы (*low*
-задача, блокирующая *high*, может обойти независимую *medium*). Готовность остаётся только про `depends_on` —
ось `parent` влияет на порядок и аннотацию `contains`, но не на `ready`/`blocked_by` (предок с открытыми
потомками может быть `ready`, но стоять последним). Подтверждённо-терминальные (`done`/`cancelled`) исключаются.
Порядок детерминирован и cycle-safe по обеим осям (руками наведённый цикл не повесит вид — все узлы всё равно
возвращаются, best-effort).

- Человеку: нумерованный список — `1. t3  [high]  (tbd)  schema design`, метка приоритета `[..]` **опускается,
  когда не задан** (как в `show`); строка `  ↳ blocked by: t1, t2` под задачей с `depends_on`-блокером и строка
  `  ↳ contains: c1, c2` под предком (его не-терминальные потомки).
- `--json`: `[{"id","title","status","priority","ready","blocked_by":[…],"contains":[…]}]`. `priority` —
  **сохранённое** значение (`""` когда не задан — честно; потребитель применяет свой дефолт); `blocked_by` и
  `contains` всегда массивы (`[]` когда пусто, никогда не `null`); пустой roadmap — `[]`.

## Зависимости  *(сессия 005, реализовано; capability `DependencyStore` — только для внешних адаптеров)*

`depends_on` — **блокирующее** ребро (отличное от иерархии `parent` и информационных `refs`). Едет на поле
`Task` и round-trip'ится через `TaskStore.Update` — YAML-референсу **выделенный порт не нужен**.
С c12 хранимое ребро — часть **агентского контракта**: объект задачи несёт `depends_on` (omitempty, как
`priority`) везде, где он эмитится — на каждой задачной `--json` поверхности (`show`/`list`/`ready`/
`add`/`edit`/`rm`/`status`/`tag`/`use`/`dep add`/`dep rm`/`dep list`), с единственным исключением: у узлов
`dep list --tree --json` `depends_on` — это *вложенный массив дочерних узлов* (форма дерева затеняет плоский
список id), — а `mtt show` печатает строку `depends:` со статусом каждого блокера (`✓` =
терминальный/удовлетворённый, `(missing)` = висячий) — без обхода `dep list` по задачам.

### `mtt dep add <id> <depends-on-id>` — добавить блокирующую зависимость
Делает `<id>` зависящей от `<depends-on-id>`. Обе задачи должны существовать. Отклоняется, если создаст
**цикл** или это **self-ребро**; повторное добавление существующего ребра — идемпотентный no-op. С `--json`
печатает обновлённую задачу.

### `mtt dep rm <id> <depends-on-id>` — снять зависимость
Снимает ребро. Идемпотентно: снятие уже отсутствующего ребра — no-op (задача при этом должна существовать).

### `mtt dep list <id>` — зависимости и зависимые задачи
Печатает прямые блокеры задачи (`depends on:`, dangling-цели помечены `(missing)`) и её **вычисляемых**
зависимых (`required by:`). С `--json` — `{id, depends_on, required_by}` (не-null массивы).
- `--tree` — транзитивное дерево зависимостей (cycle-safe; вложенный `--json`). **Повторно посещённый** узел —
  общий блокер «ромба» или обратное ребро рукотворного цикла — рендерится **без детей** и в текстовом дереве,
  и в JSON (до c16 JSON выбрасывал его целиком, и вторая ветвь ромба выглядела свободной от зависимостей).
- `--cycles` — сообщить о циклах зависимостей в проекте (defensive — `dep add` отклоняет циклы, так что
  срабатывает только на рукотворных данных).

## Ссылки  *(выпущено в t1)*

Ссылки — информационные проверяемые связи (`kind` ∈ `note`/`task`/`url`; **`comment` появляется в t2**), не
блокирующие зависимости. Живут на **задачах и заметках**. Ссылка идентифицируется естественным ключом —
**парой `(kind, target)`** (отдельного id у ссылки нет). Target — часть ключа, поэтому у сущности может быть
много ссылок одного `kind` на разные цели (`note:auth-design` + `note:login-spec` — две разные ссылки);
схлопывается только точный дубль `kind`+`target`. `--label` — аннотация, не часть идентичности. Синтаксис
цели режется по **первому** `:` (URL сохраняет свои). Ссылки хранятся **отсортированными по `(kind, target)`**
(чистые диффы). `comment:`-цель, неизвестный kind, битый URL (нужны scheme + host), не-kebab slug заметки или
пустой id — usage-ошибка (**выход 1**).

### `mtt ref add <id> <kind>:<target> [--label <text>]` — добавить ссылку задаче
Добавляет ссылку из задачи `<id>` на `<kind>:<target>` (напр. `note:auth-design`, `task:t2`,
`url:https://…`). **Идемпотентный upsert**: повторное добавление того же ключа перезаписывает label только с
`--label` (без — не меняет). **warn-not-block:** висячая цель (отсутствующая `task`/`note`) всё равно
**сохраняется**, с предупреждением в stderr и **выходом 0**; задача-носитель обязана существовать (иначе
**выход 4**). Корректный `url` сохраняется как `unverified` (внешний, не резолвится) и не предупреждает.
`--json` — объект ссылки `{kind, id, label, status}` (`status` ∈ `ok`/`dangling`/`unverified`).

### `mtt ref rm <id> <kind>:<target>` — удалить ссылку у задачи
Удаляет ссылку с этим ключом. **Идемпотентный no-op** — удаление отсутствующего ключа тоже успешно
(**выход 0**), как `dep rm`/`tag rm`. Отсутствующая задача-носитель — выход `4`.

### `mtt ref list <id>` — ссылки и backlinks задачи
Печатает исходящие ссылки задачи (каждая: `kind:target`, статус `ok`/`dangling`/`unverified`, label) и
входящие **backlinks** — задачи **и заметки**, ссылающиеся на эту задачу (вычисляемые, не хранятся). Пустая
секция `refs:`/`backlinks:` печатает `(none)` (как `dep list`), а не голый заголовок. `--json`
— `{refs, backlinks}` (оба — non-null массивы). Ссылки/backlinks также видны в `mtt show`.

### `mtt note ref add/rm/list <slug> …` — ссылки на заметке
Те же три операции для носителя-**заметки** (`mtt note ref add auth-design task:t2`, …).

### `mtt check [--json]` — верификация ссылок  *(выпущено в t1)*
Read-only обход репозитория на **висячие** ссылки (цель `task`/`note`, которой нет). Печатает каждую находку
(`carrier → kind:target [status]`) и сводку, и **выходит `7`** при наличии висячих — годится как гейт CI/flow.
Capability-aware: `note`-ссылки проверяемы только при наличии базы знаний (в YAML она всегда есть); `url` —
внешний, репортится `unverified`. **`unverified` — не провал** (выход 0). `--json` — non-null массив
`{carrier, ref, status}`. (Автоматический `--fix` отложен.)

## База знаний  *(capability `KnowledgeStore`; базовый store — t47)*

Markdown-заметки, один файл на заметку в `.mtt/knowledge/<slug>.md` (YAML-frontmatter + markdown-тело).
`<slug>` — идентичность и имя файла: явная kebab-ASCII строка (`^[a-z0-9]+(-[a-z0-9]+)*$`); невалидный slug
отклоняется. Отсутствующая заметка — exit `4`. Базовый store **одноверсионный** (история — git);
**версионирование + полнотекстовый поиск отложены** (t6), а **refs на заметках — это t1**.

### `mtt note add <slug> [flags]` — создать заметку БЗ
Создаёт заметку в `<slug>`. Отклоняет существующий slug (без перезаписи). Тело по умолчанию пустое.
- `<slug>` — стабильный идентификатор / имя файла (kebab-ASCII).
- `--title <text>` — человеческий заголовок (опционально).
- `--tag <t>` — добавить тег (повторяемый; только явно — хэштеги из тела НЕ извлекаются).
- `--priority high|medium|low` — важность (t51; переиспользует task-`Priority` VO). Управляет `mtt prime`.
- `--body <text>` — содержимое; **или** `--file <path>` (`--file -` читает stdin). `--body`/`--file` взаимоисключающие.
- `--ref <kind>:<target>` — прикрепить ссылку при создании (повторяемый; warn-not-block, t1). У `mtt add` тоже есть.
- `--json` — вывести объект созданной заметки.

Ссылки/backlinks показывают `mtt note show` и `mtt note ref list`.

### `mtt note list [--tag <t>]… [--priority <p>]… [--sort <k>] [--json]` — список заметок
Перечисляет заметки (сначала новые), опц. фильтр по тегам (OR внутри) и/или `--priority` (повторяемый; по
**хранимому** ярлыку — заметка без priority матчится только без фильтра). `--sort created|updated|priority`
(`priority` = high→low, тайбрейк по свежести). `--json` — (ненулевой) массив.

### `mtt note show <slug> [--json]` — напечатать заметку
Печатает заголовок (title/tags/**priority**/created/updated) и тело; `--json` — объект заметки (`slug` всегда
присутствует, `tags` — ненулевой массив, `priority` — omitempty).

### `mtt note edit <slug> [flags]` — редактировать заметку
Обновляет только переданные поля; `--tag` **заменяет** весь набор тегов. Бампит `updated`, сохраняет `created`.
- `--title <text>`, `--tag <t>` (повторяемый), `--priority <p>` (`--priority ""` очищает), `--body <text>` /
  `--file <path>` (`-` = stdin), `--json`.

### `mtt note rm <slug> [--force --who <w> --why <r>]` — удалить заметку
Удаляет файл заметки. **По умолчанию отказывает**, если на неё ссылается задача или заметка (t1); `--force`
перекрывает — оставляя ссылки висячими — и как разрушающий override **требует `--who`/`--why`** (иначе выход
2) и пишет audit-запись. Собственная self-ссылка заметки не блокирует. Отсутствующий slug — выход 4. `--json`
выводит удалённую заметку.

### `mtt search <query> [flags]` — текстовый поиск  *(отложено, t6)*
Простой поиск по подстроке/токенам по задачам и заметкам (без RAG).

- `--tasks` / `--notes` — ограничить область.

### `mtt prime [--min-priority <p>] [--limit N] [--json]` — курированный дайджест БЗ  *(выпущено в t51)*
Печатает компактный, **pointer-only** дайджест **важных** заметок БЗ для инжекта в начале сессии (аналог
beads-`prime`). Курация **opt-in**: попадают только заметки с **явным** priority на уровне ≥ `--min-priority`
(дефолт **`high`**) — заметка **без priority никогда** не в дайджесте; **тела не выводятся** (только slug /
priority / tags / title; детали — `mtt note show <slug>`). Порядок: по бэнду priority, затем **backlink-count**
(чаще упоминаемые выше), затем свежесть; cap — `--limit` (дефолт 20). Футер `(<N> of <M> …)`, где `M` —
eligible до cap. Пусто ⇒ одна actionable-строка (выход 0). `--json` — ненулевой массив
`{slug, title, tags, priority, backlinks}`. `mtt prime` **не печатает задачи** — при желании компонуй с
`mtt roadmap` на уровне хука.

**Проводка в старт сессии** (Claude Code `settings.json` — mtt даёт только команду, хук — конфиг):

```json
{ "hooks": { "SessionStart": [ { "hooks": [ { "type": "command", "command": "mtt prime" } ] } ] } }
```

## Самообновление  *(выпущено в t44)*

### `mtt self-update [--check-only] [--force] [--json]` — обновить установленный бинарь

Заменяет запущенный `mtt` на последний опубликованный GitHub-релиз. **Первичный путь:** скачать ассет
платформы `mtt_<tag>_<os>_<arch>[.exe]` **и** `SHA256SUMS`, проверить SHA-256 ассета **до** атомарной замены
на месте (Unix: temp в той же папке + `rename`; Windows: rename-then-swap). **Fallback:** когда для платформы
нет проверяемого ассета (нет ассета **или** нет `SHA256SUMS`) и есть Go-тулчейн — `go install
github.com/pashukhin/mtt/cmd/mtt@<tag>` (ставит в `GOBIN`/`$GOPATH/bin`; команда сообщает этот путь, если он
отличается от запущенного бинаря).

- `--check-only` — сообщить `current` / `latest` / доступность; ничего не писать. **Exit 0** для любого
  разрешимого состояния (ненулевой код — только при сбое сети/API): доступность в выводе, не в коде выхода.
- `--force` — обновиться с **dev-сборки** (не-SemVer версия, напр. `dev` или голый commit-SHA, иначе
  отклоняется — её нельзя упорядочить) или переустановить, когда уже на последней.
- `--json` — `{current, latest, update_available, updated, via, asset, path, reason, error}`; `via ∈
  {"", "asset", "go-install", "none"}` (`none` = новее релиз есть, но способа установки на этой платформе нет).

Уже последняя ⇒ no-op (`already up to date`, exit 0). Сбой или отказ ⇒ `1` с actionable-сообщением (без
выделенного кода таксономии — в отличие от `mtt check`). Потребляет ровно то, что публикует `make release`
(см. [RELEASING.md](RELEASING.md)).

## Коды возврата (предлагаемые)

Разные коды позволяют агентам ветвиться по результату без парсинга текста.

| Код | Значение |
|---|---|
| `0` | Успех |
| `1` | Общая ошибка |
| `2` | Ошибка использования — здесь: отсутствует обязательная атрибуция (`ErrMissingAttribution`) |
| `3` | Переход заблокирован — команда-гейт вернула не 0 |
| `4` | Не найдено (задача/заметка/цель не существует) |
| `5` | Post-action упал после закоммиченного перехода (`ErrPostAction`, t21) — переход сохранён |
| `6` | Недопустимый переход — не разрешён flow типа |
| `7` | Целостность: `mtt check` нашёл висячие ссылки (`ErrDanglingRefs`, t1) |

Коды `3` (гейт заблокирован) и `6` (недопустимый переход) **реализованы (сессия 006)**, `2` (отсутствует
обязательная атрибуция) — **реализован (сессия 006.5)**, `4` (not found) — **реализован (сессия 008.5)**,
применён **единообразно** ко всем путям с одиночной задачей по id (`rm`/`show`/`edit`/`tree`/`use`/`status`/
`dep`), которые оборачивают `mtt.ErrNotFound` — `5` (post-action, t21) и `7` (висячие ссылки, t1). `Execute()`
маппит `core.ErrBlocked`→3, `core.ErrInvalidTransition`→6, `core.ErrMissingAttribution`→2, `mtt.ErrNotFound`→4,
`core.ErrPostAction`→5 и `core.ErrDanglingRefs`→7. У неподдерживаемой capability (`ErrUnsupported`) пока нет
отдельного кода (появится с capability-гейтами); прочие пути отказа держат общий `1`.

**Действенные сообщения (t28).** Помимо кода, сообщение на stderr подсказывает, как исправиться: **exit 2**
печатает, как задать who/why (`author:` в `.mtt/config.local.yaml`, `MTT_BY` или `--who`/`--why`); **exit 5**
говорит, что move **уже сохранён** (не повторяй его), и печатает точные оставшиеся `post:`-команды для ручного
завершения (в текстовом и в `--json`-режиме, на stderr); **exit 4** указывает на `mtt roadmap`/`mtt list`;
**exit 6** перечисляет ходы, допустимые из текущего статуса (или отмечает, что статус терминальный).

**Массовые мутации (сессия 008.9)** — best-effort: частичный или полный отказ даёт exit `1` (общий,
git-style) с отчётом по элементам — агрегат намеренно **не** мапится на `3`/`4`/`6` (у разнородного набора нет
единого кода). **Одиночный** `rm <id>` / `tag add/rm <id>` по-прежнему даёт `4` при not-found. Пустой селектор
(источник ничего не выбрал) — успешный no-op (exit `0`); два источника или ни одного — usage error (exit `1`).

## Заметки / наблюдения (из обзора со стороны CLI)

То, что вскрывает этот справочник и что стоит держать согласованным с дизайном:

- **Чистое разделение: `edit` vs flow-команды.** `edit` трогает только не-flow поля
  (заголовок/описание); всё движение статуса — через `status`/`advance`/`start`/`done`/`cancel`, чтобы
  flow всегда соблюдался.
- **`done` и `cancel` заменяют общий `close`.** Закрытие задачи = достижение терминала: `done` (со своим
  гейтом) или `cancel`. Отдельной команды `close` нет.
- **Смена родителя меняет только `parent`; смена типа — по-прежнему не `edit`.** ID плоские и per-prefix
  (не кодируют цепочку родителей), поэтому **смена родителя** (запланированная `mtt reparent`/`move`)
  меняет только поле `parent` — ID остаётся стабильным, без переминта и без поломки входящих ссылок.
  **Смена типа** — операция крупнее (префикс завязан на тип): она остаётся вне scope для `edit` — см.
  рекатегоризацию в DESIGN.ru.md.
- **Capability-gated команды.** `dep*`, `comment*`, `note*`, `search` и история опираются на опциональные
  возможности адаптера; против бэкенда без них — `ErrUnsupported` (отдельный код появится с capability-гейтами),
  а не молчаливый сбой.
- **`--json` везде.** Все команды поддерживают JSON-вывод, чтобы агенты рулили mtt без парсинга текста;
  мутации возвращают получившийся объект — включая `add --json`, который эмитит созданную задачу
  (сессия 008.97/U3), так что свежий id берётся как `mtt add … --json | jq -r .id`, а не парсится из текста.
  С **t45** это верно для каждой команды mtt — `types`/`version`/`init` эмитят JSON, `rm <id> --json` эмитит
  объект удалённой задачи (снятый до удаления; массовый `--json` — массив исходов по элементам), а `use --json`
  — текущую задачу или `null`. Исключены только сгенерированные cobra `completion`/`help` (framework built-ins).
  (Без `--json` по-прежнему печатаются строки `created <id>`/`removed <id>`.)
- **`--role` пишется, но не энфорсится.** Сейчас пишется в историю (не откладываемый шов); роут `start`/
  `done` по роли отложен.
