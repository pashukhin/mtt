# MY_TT — Design (по-русски)

> English version (primary): [DESIGN.md](DESIGN.md). Версии держим в синхроне.

Источник правды по архитектуре. Меняется только вместе с изменением поведения.
Верхнеуровневое «зачем» — в [README.ru.md](README.ru.md).

**Видение:** mtt — агент-дружелюбная, лёгкая связка «задачи + знания» (как Jira+Confluence,
но без монструозности). Конкретное хранилище **абстрагировано за портами**: дефолт — YAML-файлы,
но через адаптер подключается пользовательская связка «таск-трекер + база знаний»
(Jira+Confluence, GitHub Issues+Wiki, beads, …). Домен о хранилище не знает.

**Human UI — тоже опция.** Агентам он не нужен: их интерфейс — CLI. Для людей (view-диаграммы,
CRUD/CQRS) есть дефолтная утилита `mtt-ui` (маленький локальный веб-сервер); а если бэкенд —
готовая связка (Jira+Confluence), человек пользуется её родным UI, и `mtt-ui` не нужен. Итог:
проект удобен и одиночкам (zero-footprint дефолт), и корпоративным монстрам (тонкий агентский
слой поверх существующего стека).

## Принятые решения

| Тема | Решение |
|---|---|
| Язык / форма | Go CLI `mtt` (`mtt add`, `mtt list`, …) |
| Архитектура | Hexagonal: `домен+usecase` ← порты; хранилище — сменный адаптер |
| Порты | `TaskStore` + `KnowledgeStore` (2 независимых); «связка» = пара адаптеров |
| Контракт | Публичный `pkg/mtt` (домен-типы + порты) — для внешних Go-адаптеров |
| Хранение (дефолт) | YAML-адаптер: **один файл на задачу**, каталог `.mtt/` |
| Источник правды | Файлы (в YAML-адаптере); БД/индекс — производный, gitignored |
| Human UI | Опционален: дефолт `mtt-ui` (локальный веб); при внешнем бэкенде — его родной UI |
| ID/slug | Минтит **адаптер** (YAML: стабильный плоский per-prefix `e1`/`t17`/`s3`); домен знает лишь логическую задачу |
| Flow | Исполняемые переходы: `description` + `commands` (все → 0, иначе переход блокируется) |
| Продвижение | `advance --to` (мета: обход до цели); режимы `--stop`(деф)/`--atomic`/`--force`; без config-DSL |
| Роли | Семантика `start`/`done` зависит от роли — шов заложен (`role` в history, `--role`, config `roles`); реализация отложена |
| Статусы | Категория `kind` (initial/active/terminal) по **топологии** flow; ≥1 каждой на flow; несколько initial допустимо; идентичность — per-flow `(type,name)`; имена — из конфига, не литералы в коде |
| Доменная модель | Чистый `pkg/mtt` (без тегов сериализации, без `prefix`); адаптеры маппят через DTO; DDD value-объекты (`StatusKind`); ссылки по идентичности, back-refs вычисляются |
| Провайдеры | Типы/flow могут приходить от внешнего провайдера; домен требует **обязательный минимум**, остальное опционально; подключается позже через `mtt connect` |
| Рекатегоризация | **Тип задачи неизменяем**; рекатегоризация = закрыть старую + создать новую + связать через `refs` |
| История | append-only `history` переходов в задаче (аудит + реконструкция); flow — через git |
| Capabilities | Возможности опциональны per-адаптер (`Capabilities()` / `ErrUnsupported`); YAML — референс |
| КБ и ссылки | KB — опц. capability; `refs` (note/task/comment/url) — проверяемые ссылки, ≠ `depends_on` |
| Хостинг | GitHub `github.com/pashukhin/mtt`, GitHub Actions |
| Ветвление | Ветка+PR на задачу → CI зелёный → squash в `main` |
| Гейт | `make check`: gofmt + vet + golangci-lint + `go test -race` |

## Позиционирование (честно, vs beads)

**Рамка.** mtt — не «ещё один таск-трекер для агентов» (переполненный слот), а **слой детерминированного
enforcement workflow**: *исполняемая state machine задачи*, переходы которой гейтятся твоими проверками.
Враг — не Jira, а `TODO.md`, строчка в промпте «запусти тесты перед done» и агент, написавший «готово», не
запустив ничего. Питчем ведём это; хранилище/бэкенды и КБ — лестница adoption, не заголовок.

[beads](https://github.com/gastownhall/beads) (Steve Yegge, ~25k★) — мощный, но тяжёлый:
хранилище **Dolt** (бинарь 44–48 МБ, cgo), опц. server-процессы с портами, git-хуки, двойная
история git+Dolt, огромная экосистема. Его сильные стороны — **зависимости** (типизированные +
gates + ready-кэш), **collision-free хэш-ID** и **cell-level merge** одной задачи — мы туда
**не лезем**.

Наш клин — там, где beads слаб или тяжёл:

- **Реальный flow по типу с исполняемыми переходами** — на переход вешаются `commands`
  (гейты/действия, все обязаны вернуть 0). У beads только плоский enum. **Наш killer-фича.**
- **Zero-footprint** — только файлы, без демона/портов/хуков/cgo; крошечный бинарь.
- **Адаптивность** — тонкий агентский слой поверх *твоего* бэкенда (YAML или Jira+Confluence):
  агентам чистый CLI, людям — их родные UI.
- **Опциональный человеческий UI** (`mtt-ui`) — маленький локальный веб; в core beads его нет.
- Читаемые sequential-ID и явная config-иерархия.

Явно **не конкурируем**: распределённость/federation, богатство зависимостей, экосистема.
Зависимости у нас — **простые и «достаточные»**. База знаний — **низкий приоритет** (у beads
уже есть `remember`/`prime`), делаем только если дёшево.

> **Честная оговорка:** две претензии из исходной мотивации устарели — у текущего beads **есть**
> иерархия epic/task/subtask и **опциональный** sequential-режим ID. Живые основания для
> отдельного проекта — именно **лёгкость** и **per-type flow**, а не «у beads этого нет».

### Конкурентный ландшафт (скан 2026)

Скан ~20 инструментов (beads, Task Master, Backlog.md, git-bug, dstask, Taskwarrior, Fossil, MCP-серверы
трекеров, unified-ticketing API, osmove/backlog, AgentWrapper AO) подтверждает и уточняет сказанное выше.
Другие **оркеструют** агентов (GitHub Agentic Workflows, Operator, AgentWrapper AO), **интегрируют** их с
трекером (Jira+Copilot, Linear automations, MCP-серверы трекеров) или **автоматизируют** PR/issue-flow —
mtt делает у́же: **гейтит переход статуса задачи** локальными per-type проверками, так что агент не объявит
`done`, не пройдя их.

- **Нескопированное ядро (две наши ставки).** (1) Config-driven **per-type flow с исполняемыми гейтами** —
  не делает никто: у beads/Task Master плоские enum-статусы; `onStatusChange` у Backlog.md — неблокирующий
  callback; у Taskwarrior hook'и ветируют, но глобально, не per-type/per-transition; TH1 у Fossil не умеет
  shell-out. (2) Тонкий агентский CLI над **сменным существующим трекером+КБ как источником правды** —
  свободно: beads/git-bug *синхронизируются* с Jira/GitHub, храня свой стор (инверсия); MCP-серверы трекеров
  одно-бэкендные; unified-API (Merge.dev, unified.to) — облачный B2B без CLI/агентов/локального бэкенда/гейтов.
- **Не отличие (честно).** *Бандл* «задачи+знания» стар (Fossil несёт tickets+wiki+forum десяток лет;
  Backlog.md бандлит tasks+docs). Типизированные зависимости, ready/blocked, иерархические ID, история и
  агент-память есть у beads. Просто «файловый таск-CLI для агентов» — переполненный, консолидирующийся слот
  (git-bug, Backlog.md, Task Master, beads), сам по себе не защита.
- **Соседи-угрозы.** `osmove/backlog` и AgentWrapper «AO» совмещают сменный трекер с lifecycle/гейт-слоем
  для агентов, но это тяжёлые оркестраторы с односторонним write-back, фиксированным kanban (не
  config-driven per-type гейты) и без КБ-пары. `Backlog.md` — быстрее всех растущий идейный сосед. `beads`
  (~25k★) — богатейший sync трекеров; развернёт адаптеры в primary storage — оспорит ставку 1.
- **Вывод.** Ниша реальная, но узкая и закрывается. Позиционируемся как *единый гейтящий control-plane над
  любым трекером*, а не ещё один формат хранения; вкладываемся в две ставки и zero-footprint; КБ держим
  поддерживающей фичей, не заголовком.

### Почему не просто pre-commit хуки / CI?

Ближайшее возражение — mtt сидит слоем **выше**:

- **pre-commit** — на *коммит*, не на жизненный цикл задачи; не per-type; агент обходит `--no-verify`;
  понятия «эта задача не может стать `done`» нет.
- **CI required checks** — постфактум, на *PR*, удалённо; не гейтят *локальное* заявление агента о
  готовности, требуют PR/хост, не знают про per-type Definition of Done.
- **mtt** гейтит **переход в жизненном цикле задачи** с **per-type DoD**, в терминах агента (`mtt done`),
  локально, над любым бэкендом. Он не заменяет CI — он *вызывает* те же проверки как гейты и добавляет
  задачные (выполнены acceptance criteria, обновлены docs, создана ветка, PR связан с issue).

## Архитектура: домен, порты, адаптеры

Гексагональная (ports & adapters). Внутри — домен и usecase; снаружи — адаптеры двух родов:
**driving** (входные: CLI, опц. `mtt-ui`) вызывают `core`; **driven** (выходные: хранилище)
вызываются из `core` через порты.

- **`pkg/mtt`** — публичный контракт: домен-типы (`Task`, `Type`, `Flow`, `Status`, `Comment`,
  `Note`, `Config`), базовый порт **`TaskStore`** и **`KnowledgeStore`**, плюс опциональные
  capability-интерфейсы (см. ниже). Публичный — чтобы внешние Go-адаптеры его реализовали.
- **`internal/core`** — usecase-логика (add/list/ready/переход-flow/поиск): работает **только
  через порты**, о конкретном хранилище не знает.
- **`internal/adapter/yaml`** — дефолтный *driven*-адаптер: реализует оба порта поверх `.mtt/`;
  **минтит ID/slug** (плоский, per-prefix — напр. `e1`/`t17`).
- **`internal/adapter/exec`** — реализация порта **`Runner`** (запуск команд перехода); в тестах
  подменяется фейком. `Runner` определён в `core` (нужен только ему, не третьим сторонам).
- **`cmd/mtt` + `internal/cli`** — тонкий CLI (*driving*): разбор → usecase → форматирование;
  на старте собирает адаптеры по конфигу и внедряет в core.
- **`cmd/mtt-ui`** — *опциональный driving-адаптер*: маленький локальный веб-сервер (view/CRUD,
  Гант) поверх того же `core`. Не нужен агентам и не нужен при внешнем бэкенде с собственным UI.

Два порта независимы: **«связка» = сконфигурированная пара адаптеров**, при этом можно
миксовать (напр. задачи из Jira, а КБ — локальный YAML). Дефолт — YAML для обоих.

**Внешние бэкенды** подключаются двумя путями: (1) in-process Go-адаптер, реализующий
`pkg/mtt`-порты (для этого контракт и публичный); (2) — позже — субпроцесс-адаптер по
документированному wire-протоколу (JSON stdin/stdout), не требующий импорта нашего Go.
Сейчас проектируем **шов** и отдаём YAML-адаптер; рантайм внешних адаптеров — когда понадобится.

> **Открытый вопрос (внешние адаптеры):** чей flow главный — наш конфиг или родной workflow
> бэкенда (напр. Jira), и как наши `commands` соотносятся с его переходами. Для YAML-дефолта
> вопрос не стоит; решаем при проектировании конкретного внешнего адаптера.

### Возможности адаптера (capabilities)

Не каждый бэкенд умеет всё: внешний трекер может не давать историю переходов, дерево
комментариев, зависимости или поиск. Поэтому возможности **опциональны на уровне адаптера**:

- **Обязательный минимум** — базовый `TaskStore` (CRUD + list/get + минтинг ID): реализует любой адаптер.
- **Опциональные возможности** — отдельные интерфейсы поверх базового: `HistoryStore`,
  `DependencyStore`, `CommentStore` (дерево), `SearchStore` (и сам `KnowledgeStore`). Адаптер
  реализует что умеет; `core` проверяет наличие через type-assertion (идиоматичный Go).
- **Дискаверабилити** — метод `Capabilities()` у бэкенда: CLI/агент знает, что доступно
  (`mtt caps`). Отсутствующая возможность даёт типизированный `ErrUnsupported` с внятным
  сообщением, а не молчаливый сбой.
- **YAML-адаптер — референс**: реализует **все** возможности (пишет историю и т.д.). Внешние
  адаптеры частичные; `core` написан на минимум и «зажигает» доступное поверх.

> **Инвариант слоёв:** `core` не импортирует `adapter/*`; адаптеры не содержат бизнес-правил
> (только CRUD/запросы за портом). Правила (flow, ready, циклы) — в `core`; **минтинг ID/slug —
> в адаптере** (кодировка ID backend-специфична).

## Почему дефолт — файлы (YAML-адаптер)

Главная боль beads — демон/локи/бинарная БД: конфликты с плагинами и болезненные
merge-конфликты в ветках агентов. Файл-на-задачу это чинит:

- нет демона и фонового состояния — `mtt` это stateless CLI (прочитал → изменил → записал);
- чистые git-мержи: два агента в разных ветках добавляют разные задачи без конфликта;
- задачи ревьюабельны в PR как обычный diff.

Масштаб трекера (сотни–тысячи задач) позволяет грузить всё в память на каждый вызов;
поиск, диаграмма Ганта и зависимости считаются in-memory. SQLite для этого не нужен.

## Раскладка данных (YAML-адаптер)

```
.mtt/
  config.yaml            # проект, типы задач и flow (общий, коммитится)
  config.local.yaml      # персональный оверлей: параметры подключения, локальные настройки (gitignored)
  tasks/
    e1.yaml              # эпик 1
    t17.yaml             # задача 17 (parent: e1)
    s3.yaml              # подзадача 3 (parent: t17)
  knowledge/
    <slug>.md            # заметки БЗ (markdown + YAML frontmatter)   [фаза 5]
```

`.mtt/` **коммитится** (это данные проекта). Это раскладка **YAML-адаптера**; правки — только
через порт (`TaskStore`/`KnowledgeStore`), руками файлы не редактируем, иначе теряется
детерминизм и валидация.

## Конфигурация (слои и локальные оверрайды)

Конфиг собирается слиянием слоёв (следующий перекрывает предыдущий) — так общие проектные и персональные
настройки держатся раздельно:

1. встроенные дефолты (шаблон `mtt init`);
2. опциональный глобальный пользовательский конфиг (`$XDG_CONFIG_HOME/mtt/config.yaml`) — персональные
   дефолты между проектами;
3. **`.mtt/config.yaml`** — общий, коммитящийся конфиг проекта (**типы и flow — здесь**);
4. **`.mtt/config.local.yaml`** — **gitignored** персональный оверлей для этого проекта;
5. переменные окружения / флаги CLI — высший приоритет.

Локальный оверлей — для **персональных параметров подключения к внешним бэкендам** (напр. разные
пользователи/креды Jira в одном проекте) и локальных настроек. **Креды никогда не кладём в коммитящийся
`config.yaml`** — только в gitignored-оверлей или env. Перекрывать общие **типы/flow** в локальном оверлее
не рекомендуется (рассинхронит команду). Deep-merge по ключам/секциям. Шов заложен сейчас (loader +
gitignore); богатые схемы подключения per-адаптер — вместе с внешними адаптерами.

## Типы и иерархия (домен) vs ID/slug (адаптер)

**Домен** знает про *логическую* задачу: её **тип**, **родителей** и **flow**. Тип задаёт:

- `name` — имя типа (напр. `epic`, `task`, `subtask`);
- `description` — опциональная ориентация для человека/агента;
- `parents` — имена допустимых родительских типов (пусто = корневой уровень) — **этим задаётся иерархия**
  (тип может стоять под несколькими родительскими типами); обратное (дети) — **вычисляется**, не хранится;
- `default` — помечает тип, используемый `add` без `--type` (не более одного; полный YAML-провайдер
  помечает ровно один); литерала `task` в коде нет;
- `statuses` (каждый — **value-объект** с категорией `kind`: initial/active/terminal) / `transitions` —
  flow (ниже).

Иерархия эпик → задача → подзадача — **не хардкод**, а следствие дефолтного конфига:
`epic` (root) ← `task` (`parents: [epic]`) ← `subtask` (`parents: [task]`).

**Именование (ID/slug) — дело адаптера, не домена.** `core` создаёт логическую задачу
(«task типа X под родителем Y»), а конкретный ID минтит `TaskStore`: у YAML это плоский `t17`,
у Jira — `PROJ-123`, у GitHub — `#42`. Поэтому `prefix` — поле **YAML-адаптера** (в его
`config.yaml`), и ID-генерация живёт **в адаптере**, за портом.

В YAML-адаптере ID **плоский и per-prefix**: `<prefix><N>`, где `N` — последовательный по префиксу
(`max+1`, `O_EXCL`). ID **не** кодирует цепочку родителей (`epic` #1 → `e1`; `task` #17 → `t17`;
`subtask` #3 → `s3`), так что идентичность отвязана от позиции: **смена родителя** меняет только
поле `parent` — ID остаётся стабильным, файл не переименовывается. ID **стабилен** и не зависит от
текста; имя — в поле `title`; иерархия — в `parent` и **вычисляется** для отображения (напр. `mtt show`
рисует линию наследования). Имя файла = `<id>.yaml`.

### Инварианты модели (проверяются загрузкой конфига)

Чисто **структурные и не завязанные на имена** — в коде **нет** литералов имён типов/статусов или
структуры ID. Типы/иерархия/категории — из конфига (домен), кодировка ID — из адаптера; дефолты — в
шаблоне `mtt init`, не в логике.

- **`kind` определяется топологией flow** (и валидируется против объявленного значения): `initial` = нет
  входящих (≥1 исходящее); `active` = ≥1 входящее и ≥1 исходящее; `terminal` = нет исходящих (≥1 входящее).
  `kind` — это **value-объект** (`StatusKind`), а не имя — та абстракция, что держит имена вне кода.
- **≥1 каждой категории на flow** (`initial`/`active`/`terminal`). Минимальный flow: `initial → active →
  terminal` (flow из 2 статусов невалиден). **Несколько `initial`-статусов допустимо** (пользователь сам
  решает, сколько точек входа хочет). Логика `ready`/завершённости/`list` работает **по категории**,
  никогда по литеральному имени.
- **Flow — замкнутый граф на тип.** Идентичность статуса — `(type, name)`: одноимённые статусы в разных
  flow — **разные**; **межфлоу-переходов нет**. Всё пространство статусов — **лес непересекающихся
  per-type flow**.
- **Reopen** — переход в *отдельный `active`-статус*, никогда обратно в `initial` и никогда из `terminal`
  (терминалы финальны) — так история задачи остаётся линейной и честной.
- **Дефолтный тип** помечается `default: true` — не более одного на уровне домена (`DefaultType`
  откатывается к первому типу); полный YAML-провайдер обязан пометить **ровно один**. Литерала `task` нет.
- **Входной статус.** Задача стартует в **initial**-статусе своего типа. Когда во flow больше одного
  `initial`, точка входа — тот, что помечен `default: true` на статусе (зеркалит дефолтный маркер **типа**),
  иначе — первый `initial` в порядке конфига (`Type.InitialStatus`). Валидация: не более одного `default`-
  статуса на flow, и `default`-статус обязан быть `initial`.
- **Размещение `add` / `--no-parent`.** Тип, у которого `parents` не пуст, требует родителя (`--parent`);
  как **осознанное исключение**, `--no-parent` создаёт его на верхнем уровне (плоский корневой ID). Корневым
  типам не нужно ни то, ни другое. (Так пользователя никогда не блокирует создание задачи верхнего уровня.)
- **Санитарность иерархии:** каждый элемент в `parents` типа именует существующий тип; тип не является
  собственным родителем.
- **Тип задачи неизменяем.** Смена типа = **рекатегоризация**: закрыть старую задачу (→ терминал),
  создать новую целевого типа и связать их через `refs` (kind `task`, backlinks в обе стороны).

> **Известное ограничение YAML-адаптера (осознанный размен):** sequential-ID коллизируют при
> параллельном создании в разных ветках — `e2` в двух ветках даёт видимый git add/add конфликт.
> Приемлемо для low-concurrency; при росте параллелизма — namespace-префикс по branch/agent.
> У других адаптеров (напр. Jira) своя схема без этой проблемы.

### Доменная модель vs сериализация (DDD)

- **Чистый домен.** Типы `pkg/mtt` не несут **никаких** yaml/json-тегов и **никаких** полей адаптера
  (`prefix` — только у адаптера). У каждого адаптера свои DTO, которые он **маппит** в домен и обратно;
  контракт никогда не зависит от формата хранения.
- **Ссылки по идентичности.** Внутри flow переходы ссылаются на статусы по **имени**; между агрегатами
  (типы, а позже задачи) ссылки — **имена/ID**, никогда не указатели. **Back-references вычисляются**
  (обратный индекс: дети, backlinks), не хранятся — forward-ссылки единственный источник правды.
- **Разрешённый граф — производный.** Связанный неизменяемый in-memory граф объектов (для обхода: `ready`,
  `advance`, детект циклов) строит `internal/core` по мере надобности — он **не** часть контракта.
- **Провайдер-агностичность.** Домен требует **обязательный минимум** (у `Type` есть имя + flow, чьи
  статусы имеют name+kind, а переходы — from/to) и считает остальное опциональным (`description`, `parents`,
  `default` и `commands` — последнее *наше* локальное усиление-гейт, отсутствующее во внешних трекерах).
  Поэтому типы/flow могут приходить от **внешнего провайдера** (подключается позже через `mtt connect`);
  YAML-адаптер — **полный провайдер** (даёт всё).

## Модель задачи

Поля сериализуются в фиксированном порядке (порядок полей структуры) → детерминированный diff.

```yaml
id: s3
type: subtask
title: fix login redirect loop
status: in_progress
parent: t17
depends_on:
  - t2
refs:
  - {kind: note, id: auth-design, label: spec}
  - {kind: task, id: t2}
created: 2026-07-03T09:20:00Z
updated: 2026-07-03T10:00:00Z
description: |
  Многострочное описание.
comments:
  - id: 1
    author: agent
    created: 2026-07-03T09:25:00Z
    body: первый коммент
    replies:
      - id: 2
        author: human
        created: 2026-07-03T09:40:00Z
        body: ответ (дерево через вложенные replies)
history:
  - {at: 2026-07-03T09:25:00Z, by: agent, role: implementer, from: tbd, to: in_progress}
  - {at: 2026-07-03T10:00:00Z, by: agent, role: implementer, from: in_progress, to: done,
     checks: [{cmd: "make lint", exit: 0}, {cmd: "make test", exit: 0}]}
```

- `parent` — пусто для эпика.
- `depends_on` — список ID (в т.ч. межэпиковых); **блокирующее** ребро (влияет на `ready`).
- `refs` — проверяемые ссылки на `note`/`task`/`comment`/`url` (см. «База знаний и ссылки»);
  **информационные**, не блокирующие. Комментарии тоже могут нести `refs`.
- `comments` — дерево через вложенные `replies`; `id` комментария последователен в пределах задачи.
- `history` — **append-only** аудит переходов (`from→to`, `at`, `by`, результаты `checks`);
  задним числом не восстановить, поэтому пишем сразу — основа аудита и реконструкции графа.

## Список (`list`) и правка (`edit`) — сессия 003

- **Порядок по умолчанию в `list` provider-agnostic.** Первичный ключ — **доменная временная метка**:
  `Created` по убыванию (сначала свежие), либо `Updated` по убыванию с `--sort updated` — никогда не
  структура ID (у внешнего адаптера ID может не сортироваться осмысленно). Тай-брейк — сравнение ID как
  **непрозрачной строки**, так что при равных метках порядок остаётся стабильным и детерминированным между
  запусками.
- **`edit` трогает только заголовок/описание.** Смена статуса идёт через flow (`status`/`advance`/…), чтобы
  гейты соблюдались; смена родителя (`parent`) и смена типа — **отдельные операции**, не `edit` — уже
  отмечено выше и в бэклоге.

## Иерархия: размещение и рендер — сессия 004

- **`add --parent <id>` — нормальный путь размещения.** Ребёнок хранит только `parent` (forward-ссылка по
  ID); адаптер по-прежнему минтит **плоский per-prefix** ID, так что идентичность независима от позиции.
  Размещение валидируется как мутация в `core.Adder`: родитель должен существовать, а его **тип** — быть
  разрешён полем `parents` дочернего типа (чистый предикат `Type.AcceptsParent`). `--no-parent` остаётся
  escape-hatch.
- **Дети/предки вычисляются, не хранятся.** `core.Index` — производное in-memory представление из
  `TaskStore.List` (чистое значение — без стора и часов; «резолвнутый граф производный», не часть контракта
  `pkg/mtt`): parent→children, цепочка предков, корни; cycle-safe; сироты (висячий parent) всплывают как
  корни. Порядок сиблингов переиспользует provider-agnostic сортировку `Select` (`Created` desc, тай-брейк по ID).
- **`tree` и родословная в `show` — чистые чтения** (без usecase): `mtt tree` рисует лес (фильтр
  keep-ancestors, `--depth`, вложенный `--json`); `mtt show` печатает breadcrumb-родословную. Единый предикат
  `core.Match` (status/type/kind/parent) переиспользуется `list` и обходом `tree` (DRY).

## Flow: исполняемые переходы (killer-фича) и `mtt init`

Тип задаёт **flow** — граф статусов с переходами. На каждый переход можно повесить:

- `description` — текст «что именно делаем» (понимание для агента/человека);
- `commands` — последовательность консольных команд; **все обязаны вернуть 0**, иначе переход
  **блокируется** (задача остаётся в исходном статусе).

Это превращает flow из совета в **исполняемый гейт + действие**. Примеры:

- `in_progress → done`: `["make lint", "make test"]` — не пускаем в `done`, пока не зелено.
- `tbd → in_progress`: ревью спеки + создание ветки под задачу.

Смысл: **агент работает в терминах задач** (`mtt start t17`, `mtt done t17`), а механику
статус-flow (проверки, ветка, …) прячет сам переход — меньше отвлечения на детали.

Исполнение — за портом **`Runner`**: `core` оркеструет переход, вызывает `Runner`, гейтит по
exit-кодам. `Runner` определён в `core`, реализован в `internal/adapter/exec`, в тестах — фейк.
Команды идут по порядку, прерываемся на первом ненулевом; рабочая директория — корень проекта;
таймаут на команду; escape-hatch `--no-run` (форсировать переход без команд, для аварий).
Команды берутся из конфига (доверенный, как Makefile/git-хуки), не из сети.

> **Отгружено (s006, `mtt status <id> <new>`):** **одно** гейтящееся ребро. `Runner.Run(commands)` без
> `dir` — `exec`-адаптер конструируется с `cwd = корень проекта` (core без файловых путей). Ненулевой exit —
> **данные** (`Check`), не Go-ошибка; заблокированный гейт оставляет задачу неизменной и не пишет `history`.
> Таймаут на команду задаётся конфигом (`command_timeout`, дефолт `5m`, адаптерная настройка). Core-sentinels
> `ErrBlocked`/`ErrInvalidTransition` маппятся в exit-коды `3`/`6`. Достаточно линейного поиска ребра в
> `Type.Transitions` (пока без `ResolvedFlow` — он окупится в мульти-обходе `advance` в s007). Мета-обход
> `advance`/`start`/`done` и режимы — s007. Гейт печатает **live-прогресс пайплайна** в stderr (`▶`/`✓`/`✗` +
> тайминг на команду); вывод самих команд по умолчанию скрыт, стримится с `-v`, и/или пишется с `--log-file`.
> `by` резолвится `--by` > `MTT_BY` > `author` из `config.local.yaml` (долговременный персональный дефолт;
> `role` — только флаг/env).

> Оговорка (для планирования): команды с побочными эффектами (создание ветки) ставим **после**
> проверок; при падении после сайд-эффекта переход не коммитим, но сайд-эффект уже случился —
> это на совести порядка команд. Двухфазную модель (checks → actions) вводим лишь при нужде.

> **Шов (отложено): откат / компенсация.** У перехода можно опционально объявить компенсирующие команды
> (напр. `rollback:` / `on_failure:`), исполняемые в **обратном порядке** по уже успешным командам, когда
> следующая команда того же пайплайна падает (внутри-пайплайновая компенсация), а также при обрыве
> `--atomic`/мульти-шаг `advance` после сайд-эффектов — чтобы отменить сделанное частично применённым
> переходом (удалить созданную ветку, …). Сейчас не делаем; точка расширения — путь обрыва в исполнителе.

> **Шов (отложено): структурные команды.** Три желаемые фичи — **таймаут на команду** поверх глобального
> `command_timeout` (fail-fast, если быстрая команда затянулась), `rollback` выше и **плейсхолдеры** в
> команде (`git checkout -b task/{{.ID}}` на `tbd → in_progress`) — сходятся к эволюции `Transition.Commands`
> из `[]string` в value-object `Command` (`{run, timeout?, rollback?}`) с раскрытием плейсхолдеров в `run`.
> Аддитивно/обратносовместимо (голая строка ⇒ `{run: …}`), но это **смена доменной формы** — одним
> осознанным слайсом. **Кавеат плейсхолдеров:** подставляемые значения попадают в `sh -c`, поэтому
> ограничиваемся безопасными по форме полями (`id`/`type`/`status`) либо shell-квотируем произвольные
> (`title`) — никогда не подставляем сырой текст без квотирования. См. TASKS.md → Later.

### Продвижение по flow: `advance` / `start` / `done`

`start`/`done` — не одиночный переход, а **мета-команда**: примитив `advance <task> --to <status>`
проводит задачу по цепочке переходов до целевого статуса, исполняя гейты на рёбрах. `start` =
`--to in_progress`, `done` = `--to done` (встроенные алиасы). **Config-DSL команд не делаем** —
режимы это флаги.

Семантика обхода (предсказуемость важнее «умности»):

- идём только по рёбрам, **ведущим к цели**; **никогда не заходим в чужой терминал** (не
  авто-`cancel`); защита от циклов; цель недостижима → ошибка;
- на настоящей развилке (≥2 прогрессирующих ребра) — **стоп, а не догадка**;
- каждое пройденное ребро исполняет свои `commands` и пишет запись в `history`;
- не-`ready` задачу (незакрытые `depends_on`) в терминал по умолчанию не гоним — предупреждаем.

Благодаря линейному дефолтному flow (`initial → active → terminal`) дефолтный случай однозначен; развилки —
только если пользователь сам добавил ветвление. (Имена статусов — из конфига, никогда из кода.)

Режимы (флаги):

- `--stop` (**дефолт**) — двигать до первого упавшего гейта или развилки, сообщить где/почему;
- `--atomic` — всё-или-ничего **по статусу** (упал гейт → не двигаем и не пишем переходы);
  **оговорка:** сайд-эффекты уже выполненных команд не откатываются;
- `--force` — двигать безусловно, игнорируя гейты (аварийный; обобщает `--no-run`).

### Роли — шов (реализацию откладываем)

Семантика `start`/`done` (и переходов вообще) зависит от **роли** вызывающего: у агента-ревьюера
`done` значит иное, чем у имплементера (напр. имплементер `done` уводит в `review`, ревьюер
`done` — из `review` в терминал). Полноценно это — роли в конфиге + `--role`/`MTT_ROLE` +
опциональный тег роли на переходах; резолвер `advance` тогда параметризуется ролью.

**Сейчас не реализуем**, но закладываем швы:

- **Единственный не-откладываемый шов — писать `role` в `history`** (рядом с `by`): задним
  числом не восстановить (как и саму историю). Поле резервируем в модели сразу.
- `advance` и verb→target — резолвер **параметризуем ролью** (сигнатура принимает role); сейчас
  role-agnostic (одна неявная роль).
- CLI резервирует `--role=` / `MTT_ROLE`; конфиг может дорасти до секции `roles` и role-тега на
  переходах — аддитивно, loader forward-compatible.

Ограничения (чтобы не разрослось): роли — **семантический роутинг** (что значит глагол для роли),
**не** RBAC/энфорсмент (агенты кооперативны — роутим, не полицейские). Имена ролей — из конфига,
в коде не хардкодятся (как типы/статусы).

> **Направление (отложено): профили актора.** Секцию `roles` ожидаемо разворачиваем в именованные
> **профили**, связывающие идентичность с ролью — `(by, role)`, напр. `(coding-agent, implementer)` /
> `(Alice, reviewer)` — ровно один помечен `default: true` и применяется, когда не заданы ни `--profile`, ни
> `--by`/`--role`. Так как mtt используют в основном **кодовые агенты**, делящие конфиг репозитория с
> человеком, профиль агента — эргономичный дефолт, а человек переопределяет. `mtt profile add/list/rm`
> управляет **только** персональными профилями в `config.local.yaml` (общие проектные профили, если есть,
> для команды read-only). Это **поглощает шов `author` из s006** (`author` = `by` дефолтного профиля) и
> forward-compatible. См. TASKS.md → Later.

`mtt init` пишет `.mtt/config.yaml` с примерными типами `epic`/`task`/`subtask` и линейным flow
(`initial → active → terminal`, плюс второй терминал для отмены). Эти имена — из **шаблона**, не из кода.
**Команд по умолчанию нет** — их навешивает пользователь под свой проект.

```yaml
version: 1
project:
  name: my_tt
types:
  - name: epic
    description: Большой объём работы, охватывающий несколько задач.
    prefix: e                    # prefix — поле YAML-адаптера (кодировка ID), не домена
    parents: []                  # корневой уровень
    statuses:
      - {name: tbd,         kind: initial}
      - {name: in_progress, kind: active}
      - {name: done,        kind: terminal}
      - {name: cancelled,   kind: terminal}
    transitions:
      - {from: tbd,         to: in_progress}
      - {from: tbd,         to: cancelled}
      - {from: in_progress, to: done, description: "все задачи эпика закрыты"}
      - {from: in_progress, to: cancelled}
  - name: task                   # ДЕФОЛТНЫЙ тип (add без --type)
    description: Единица работы под эпиком.
    prefix: t
    parents: [epic]
    default: true
    statuses:
      - {name: tbd,         kind: initial}
      - {name: in_progress, kind: active}
      - {name: done,        kind: terminal}
      - {name: cancelled,   kind: terminal}
    transitions:
      - {from: tbd,         to: in_progress, description: "ревью спеки, создать ветку"}
      - {from: tbd,         to: cancelled}
      - {from: in_progress, to: done, description: "гейт качества"}
      - {from: in_progress, to: cancelled}
  - name: subtask
    description: Небольшой шаг внутри задачи.
    prefix: s
    parents: [task]
    statuses:
      - {name: tbd,         kind: initial}
      - {name: in_progress, kind: active}
      - {name: done,        kind: terminal}
      - {name: cancelled,   kind: terminal}
    transitions:
      - {from: tbd,         to: in_progress}
      - {from: tbd,         to: cancelled}
      - {from: in_progress, to: done}
      - {from: in_progress, to: cancelled}
```

Навесить команды — правкой перехода (без изменений в коде):

```yaml
      - from: in_progress
        to: done
        description: "гейт качества"
        commands: ["make lint", "make test"]   # все → 0, иначе done заблокирован
```

Так же добавляется тип `bug` (`prefix: b`, `parents: [epic]`, flow с active-статусом `review`) — только конфигом.

`mtt init --template coding` кладёт примерные coding-типы — `feature`/`bugfix`/`refactor` — каждый со своим
гейтящим Definition of Done (ветка + lint/test; у `bugfix` ещё сначала падающий тест; у `refactor` — без
diff публичного API), как готовое демо ценности enforcement.

## Зависимости

- `depends_on: [id, …]` — **блокирующее** ребро (отличное от иерархического `parent` и информационных
  `refs`). Едет на поле `Task` и round-trip'ится через `TaskStore.Update` — **без выделенного порта** для
  YAML-референса (capability `DependencyStore` нужен только внешним адаптерам, которые не могут встроить
  поле). Добавление ребра отклоняется, если оно создаёт **цикл** (правило `core`), и self-ребро отклоняется;
  проверка цикла строит производный `core.DepGraph` над `depends_on` (параллельно `Index` над `parent` из
  s004, держится **отдельно** — у графов разная форма).
- Задача **ready** ⇔ её статус не `terminal` И все `depends_on` в `terminal`-статусе (по категории `kind`:
  `done` или `cancelled`, а не по литералу). Готовность **консервативна**: неразрешимый блокер (dangling
  `depends_on` или статус не из текущего flow) оставляет задачу **не** ready — `ready` требует позитивного
  подтверждения.
- `mtt ready` — «что можно брать в работу»; `mtt list --ready` — шорткат-компаньон (один общий примитив
  `core.Ready` за обоими). `mtt dep list <id>` показывает прямые блокеры задачи и её вычисляемых зависимых,
  с `--tree` (транзитивно) и `--cycles` (по проекту, defensive).

> **Отложенный дизайн-вопрос — семантика `cancelled`-блокера.** `terminal`-блокер разблокирует зависимую
> задачу, а `cancelled` — это `terminal`-`kind`, поэтому задача с блокерами `done` **и** `cancelled`
> формально ready — хотя *cancelled* (брошенный, а не выполненный) блокер, возможно, означает, что зависимую
> надо пересмотреть, а не молча разблокировать. Текущее поведение сохраняет terminal-по-`kind` (cancelled
> разблокирует). **Пересмотрено в s006** (терминалы теперь достижимы через `mtt status`): решение —
> **сохранить** terminal-по-`kind` (cancelled разблокирует); корректный фикс (различие succeeded-vs-abandoned,
> hard/soft-ребро) требует нового доменного моделирования, которое flow-gate-сессия не должна протаскивать, и
> рисковал бы name-agnostic принципом. s006 добавляет e2e (`cancel_unblock`), демонстрирующий текущую
> семантику на достижимом состоянии; более глубокий фикс остаётся отложенным. См. TASKS.md → Later.

## Версионирование flow и история задач

- **Определение flow версионируется git'ом** (в YAML-адаптере): `git log .mtt/config.yaml` — вся
  эволюция графа. Явные `flow-version`/миграции существующих задач при переименовании статуса —
  откладываемо. Домен трактует `status` как **данные** и валидирует по текущему flow **лениво**
  (не падает, а флагует расхождение), чтобы дрейф конфига не ломал старые задачи.
- **История переходов** пишется в саму задачу (`history`, append-only) — это даёт аудит и, позже,
  **реконструкцию наблюдаемого графа** (read-only агрегация историй всех задач). Пишем в задаче,
  не в центральный лог, чтобы сохранить чистые пофайловые мержи. История — **опциональная
  возможность** (`HistoryStore`): YAML-адаптер её пишет, внешний бэкенд может не поддерживать
  (тогда `ErrUnsupported`), и `core` деградирует мягко.

## База знаний и ссылки (`refs`)

**KB — опциональная capability** (`KnowledgeStore`, как Confluence поверх Jira). Если её нет,
знания живут прямо в задачах и комментариях; «база знаний» — это они и ссылки между ними.

**Заметки версионируются.** `KnowledgeStore` не разрушает содержимое: сохранение заметки (`note edit`)
создаёт **новую версию со ссылкой на предыдущую**, история сохраняется — тот же принцип, что и `history`
переходов задачи. Это capability KB: YAML-референс реализует сам (файлы также трекает git); внешние
бэкенды (Confluence, …) полагаются на своё **нативное** версионирование — это их дело. Доменный шов —
у `Note` есть идентичность версии + ссылка на предшественника. Отложено на фазу 5 вместе с KB.

Задачи и комментарии несут **`refs`** — структурный список проверяемых ссылок:

```yaml
refs:
  - {kind: note, id: auth-design, label: "spec"}
  - {kind: task, id: t2}
  - {kind: url,  id: "https://…"}
```

- `kind` ∈ `note` | `task` | `comment` | `url`. Это **не** `depends_on` (то — блокирующее ребро);
  `refs` — информационная, проверяемая на целостность связь.
- **Верификация capability-aware:** `task`/`comment` резолвятся через `TaskStore` (всегда);
  `note` — только при `KnowledgeStore` (иначе «не могу проверить: нет KB»); `url` — внешняя,
  не резолвим (опц. HEAD-проверка позже).
- **Семантика:** на запись — предупреждаем о битой ссылке (не хард-блок); `mtt check` — обход на
  висячие ссылки; `mtt show` — ссылки и **backlinks** («что ссылается сюда»); при удалении цели —
  предупреждаем о входящих ссылках.
- **Фазы:** поле `refs` — в модель уже в Фазе 1; резолв `task`/`comment` — Фаза 2; `note` +
  `mtt check`/backlinks — Фаза 5 (с KB).

## Поиск (фаза 5)

Встроенный текстовый поиск (подстрока/токены) по задачам и БЗ. Без RAG.
«Внешний индексатор» — опциональный hook (внешняя команда) через конфиг.

## Human UI — опционально (`mtt-ui`, фаза 7)

`mtt-ui` — **отдельная опциональная утилита** (driving-адаптер), а не часть агентского бинаря:
`net/http` + `embed.FS`, минимальный web UI, Гант через SVG, поверх того же `core`/портов.
Нужен на YAML-дефолте; при внешнем бэкенде (Jira+Confluence) человек использует его родной UI.
Агентам UI не нужен. В CLI при этом есть текстовый/ASCII Гант. Самая поздняя фаза.

**Позже (бэклог):**

- позже — **смена родителя** (`mtt reparent`/`move`): изменить `parent` задачи; возможно благодаря плоским,
  не завязанным на позицию ID.
- позже — **теги**: сквозной `[]string`-лейбл на задачах (зарезервирован в модели уже сейчас); фильтрация —
  вместе с `list`.
- позже — **доски / views**: запрос/представление над тегами/статусом/типом (связано с `list` и `mtt-ui`);
  бэклог — такое представление.
- позже — **устойчивый, не зависящий от git аудит правок**: `edit` сегодня только двигает `updated`, а
  фактической историей служит git; журнал изменений или версионирование полей (аддитивно, без слома
  обратной совместимости) сделали бы историю правок доступной для запросов без git. В паре с
  **источником субъектной идентичности (`By`)** — кто «действует» для атрибуции, вероятно поле
  `.mtt/config.local.yaml`, отдельное от `--role` (`--role` — это *какую шляпу* носят, `By` — это *кто*).
  Оба отложены; поле `history` остаётся **только про переходы** (фаза 3) — оно не расширяется на обычные
  `edit`.
- позже — **валидация значений фильтра `list`**: сейчас `list --status X`/`--type Y` фильтрует только по
  задачам, поэтому неизвестное значение даёт пустой результат — неотличимо от «валидный статус, задач пока
  нет». Валидировать значения фильтра по конфигу (статус должен существовать в *каком-то* flow, тип — быть
  сконфигурированным типом) и падать с ошибкой на неизвестном значении, чтобы «нет задач в этом статусе» и
  «такого статуса не может быть во flow» различались. Требует, чтобы `list` загружал конфиг (сейчас он читает
  только хранилище).
- позже — **`mtt edit <id> --editor`**: открыть задачу в `$EDITOR` (править поля интерактивно / через
  отрисованный файл) вместо передачи `--title`/`--description` в командной строке — человеко-удобная
  альтернатива правке через флаги.
- позже — **отступ многострочного описания в `show`**: `mtt show` печатает описание через `"\n  %s\n"`,
  поэтому у многострочного описания отступ `  ` получает **только первая** строка, а продолжения идут вровень
  слева. Данные корректны (round-trip как YAML block scalar) — это косметический нит в `formatTask` (из
  сессии 002). Фикс: отступать каждую строку описания. Дёшево, косметика, не блокирует.

> **Снимок доменной модели.** [docs/architecture/model.go](docs/architecture/model.go) — code-form,
> tiered (T1/T2/T3) индекс целевой поверхности контракта: домен-типы + порты + опциональные capability,
> core-usecase с их зависимостями, производный резолвнутый граф и открытые пробелы. Это дизайн-референс
> (компилируется, lint-clean, browsable через `go doc`), в бинарь **не** входит; авторитетна эта проза, а
> снимок — её структурная карта. Два слоя: **A** (контракт/персистентные агрегаты) — идентичность
> типизированным значением (`TaskID`/`TypeName`/`StatusName`), граница сериализации/провайдера; **B**
> (резолвнутый граф в core) — указательные связи для обхода, не сериализуется.
