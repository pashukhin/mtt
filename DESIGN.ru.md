# MY_TT — Design (по-русски)

> English version (primary): [DESIGN.md](DESIGN.md). Версии держим в синхроне.

Источник правды по архитектуре. Меняется только вместе с изменением поведения.
Верхнеуровневое «зачем» — в [README.ru.md](README.ru.md).

**Видение:** mtt — агент-дружелюбная, лёгкая связка «задачи + знания» (как Jira+Confluence,
но без монструозности). Конкретное хранилище **абстрагировано за портами**: дефолт — YAML-файлы,
но через адаптер подключается пользовательская связка «таск-трекер + база знаний»
(Jira+Confluence, GitHub Issues+Wiki, beads, …). Домен о хранилище не знает.

**Human UI — тоже опция.** Агентам он не нужен: их интерфейс — CLI. Для людей (view-диаграммы,
CRUD/CQRS) есть дефолтная утилита `mtt-ui` (маленький локальный веб-сервер); а если бэкенд —
готовая связка (Jira+Confluence), человек пользуется её родным UI, и `mtt-ui` не нужен. Итог:
проект удобен и одиночкам (zero-footprint дефолт), и корпоративным монстрам (тонкий агентский
слой поверх существующего стека).

## Принятые решения

| Тема | Решение |
|---|---|
| Язык / форма | Go CLI `mtt` (`mtt add`, `mtt list`, …) |
| Архитектура | Hexagonal: `домен+usecase` ← порты; хранилище — сменный адаптер |
| Порты | `TaskStore` + `KnowledgeStore` (2 независимых); «связка» = пара адаптеров |
| Контракт | Публичный `pkg/mtt` (домен-типы + порты) — для внешних Go-адаптеров |
| Хранение (дефолт) | YAML-адаптер: **один файл на задачу**, каталог `.mtt/` |
| Источник правды | Файлы (в YAML-адаптере); БД/индекс — производный, gitignored |
| Human UI | Опционален: дефолт `mtt-ui` (локальный веб); при внешнем бэкенде — его родной UI |
| ID/slug | Минтит **адаптер** (YAML: стабильный плоский per-prefix `e1`/`t17`/`s3`); домен знает лишь логическую задачу |
| Flow | Исполняемые переходы: `description` + `commands` (все → 0, иначе переход блокируется) |
| Продвижение | `advance --to` (мета: обход до цели); режимы `--stop`(деф)/`--atomic`/`--force`; без config-DSL |
| Роли | Семантика `start`/`done` зависит от роли — шов заложен (`role` в history, `--role`, config `roles`); реализация отложена |
| Статусы | Категория `kind` (initial/active/terminal) по **топологии** flow; ≥1 каждой на flow; несколько initial допустимо; идентичность — per-flow `(type,name)`; имена — из конфига, не литералы в коде |
| Доменная модель | Чистый `pkg/mtt` (без тегов сериализации, без `prefix`); адаптеры маппят через DTO; DDD value-объекты (`StatusKind`); ссылки по идентичности, back-refs вычисляются |
| Провайдеры | Типы/flow могут приходить от внешнего провайдера; домен требует **обязательный минимум**, остальное опционально; подключается позже через `mtt connect` |
| Рекатегоризация | **Тип задачи неизменяем**; рекатегоризация = закрыть старую + создать новую + связать через `refs` |
| История | append-only `history` переходов в задаче (аудит + реконструкция); flow — через git |
| Capabilities | Возможности опциональны per-адаптер (`Capabilities()` / `ErrUnsupported`); YAML — референс |
| КБ и ссылки | KB — опц. capability (базовый `KnowledgeStore` + `mtt note` — t47; версионирование/поиск → t6); `refs` на задачах+заметках — проверяемые ссылки, ≠ `depends_on` (выпущено в t1: `note`/`task`/`url` + `check`/backlinks/refuse-guard; `comment` → t2) |
| Хостинг | GitHub `github.com/pashukhin/mtt`, GitHub Actions |
| Ветвление | Ветка+PR на задачу → CI зелёный → squash в `main` |
| Гейт | `make check`: gofmt + vet + golangci-lint + `go test -race` |

## Позиционирование (честно, vs beads)

**Рамка.** mtt — не «ещё один таск-трекер для агентов» (переполненный слот), а **слой детерминированного
enforcement workflow**: *исполняемая state machine задачи*, переходы которой гейтятся твоими проверками.
Враг — не Jira, а `TODO.md`, строчка в промпте «запусти тесты перед done» и агент, написавший «готово», не
запустив ничего. Питчем ведём это; хранилище/бэкенды и КБ — лестница adoption, не заголовок.

[beads](https://github.com/gastownhall/beads) (Steve Yegge, ~25k★) — мощный, но тяжёлый:
хранилище **Dolt** (бинарь 44–48 МБ, cgo), опц. server-процессы с портами, git-хуки, двойная
история git+Dolt, огромная экосистема. Его сильные стороны — **зависимости** (типизированные +
gates + ready-кэш), **collision-free хэш-ID** и **cell-level merge** одной задачи — мы туда
**не лезем**.

Наш клин — там, где beads слаб или тяжёл:

- **Реальный flow по типу с исполняемыми переходами** — на переход вешаются `commands`
  (гейты/действия, все обязаны вернуть 0). У beads только плоский enum. **Наш killer-фича.**
- **Zero-footprint** — только файлы, без демона/портов/хуков/cgo; крошечный бинарь.
- **Адаптивность** — тонкий агентский слой поверх *твоего* бэкенда (YAML или Jira+Confluence):
  агентам чистый CLI, людям — их родные UI.
- **Опциональный человеческий UI** (`mtt-ui`) — маленький локальный веб; в core beads его нет.
- Читаемые sequential-ID и явная config-иерархия.

Явно **не конкурируем**: распределённость/federation, богатство зависимостей, экосистема.
Зависимости у нас — **простые и «достаточные»**. База знаний — **низкий приоритет** (у beads
уже есть `remember`/`prime`), делаем только если дёшево.

> **Честная оговорка:** две претензии из исходной мотивации устарели — у текущего beads **есть**
> иерархия epic/task/subtask и **опциональный** sequential-режим ID. Живые основания для
> отдельного проекта — именно **лёгкость** и **per-type flow**, а не «у beads этого нет».

### Конкурентный ландшафт (скан 2026)

Скан ~20 инструментов (beads, Task Master, Backlog.md, git-bug, dstask, Taskwarrior, Fossil, MCP-серверы
трекеров, unified-ticketing API, osmove/backlog, AgentWrapper AO) подтверждает и уточняет сказанное выше.
Другие **оркеструют** агентов (GitHub Agentic Workflows, Operator, AgentWrapper AO), **интегрируют** их с
трекером (Jira+Copilot, Linear automations, MCP-серверы трекеров) или **автоматизируют** PR/issue-flow —
mtt делает у́же: **гейтит переход статуса задачи** локальными per-type проверками, так что агент не объявит
`done`, не пройдя их.

- **Нескопированное ядро (две наши ставки).** (1) Config-driven **per-type flow с исполняемыми гейтами** —
  не делает никто: у beads/Task Master плоские enum-статусы; `onStatusChange` у Backlog.md — неблокирующий
  callback; у Taskwarrior hook'и ветируют, но глобально, не per-type/per-transition; TH1 у Fossil не умеет
  shell-out. (2) Тонкий агентский CLI над **сменным существующим трекером+КБ как источником правды** —
  свободно: beads/git-bug *синхронизируются* с Jira/GitHub, храня свой стор (инверсия); MCP-серверы трекеров
  одно-бэкендные; unified-API (Merge.dev, unified.to) — облачный B2B без CLI/агентов/локального бэкенда/гейтов.
- **Не отличие (честно).** *Бандл* «задачи+знания» стар (Fossil несёт tickets+wiki+forum десяток лет;
  Backlog.md бандлит tasks+docs). Типизированные зависимости, ready/blocked, иерархические ID, история и
  агент-память есть у beads. Просто «файловый таск-CLI для агентов» — переполненный, консолидирующийся слот
  (git-bug, Backlog.md, Task Master, beads), сам по себе не защита.
- **Соседи-угрозы.** `osmove/backlog` и AgentWrapper «AO» совмещают сменный трекер с lifecycle/гейт-слоем
  для агентов, но это тяжёлые оркестраторы с односторонним write-back, фиксированным kanban (не
  config-driven per-type гейты) и без КБ-пары. `Backlog.md` — быстрее всех растущий идейный сосед. `beads`
  (~25k★) — богатейший sync трекеров; развернёт адаптеры в primary storage — оспорит ставку 1.
- **Вывод.** Ниша реальная, но узкая и закрывается. Позиционируемся как *единый гейтящий control-plane над
  любым трекером*, а не ещё один формат хранения; вкладываемся в две ставки и zero-footprint; КБ держим
  поддерживающей фичей, не заголовком.

### Почему не просто pre-commit хуки / CI?

Ближайшее возражение — mtt сидит слоем **выше**:

- **pre-commit** — на *коммит*, не на жизненный цикл задачи; не per-type; агент обходит `--no-verify`;
  понятия «эта задача не может стать `done`» нет.
- **CI required checks** — постфактум, на *PR*, удалённо; не гейтят *локальное* заявление агента о
  готовности, требуют PR/хост, не знают про per-type Definition of Done.
- **mtt** гейтит **переход в жизненном цикле задачи** с **per-type DoD**, в терминах агента (`mtt done`),
  локально, над любым бэкендом. Он не заменяет CI — он *вызывает* те же проверки как гейты и добавляет
  задачные (выполнены acceptance criteria, обновлены docs, создана ветка, PR связан с issue).

## Архитектура: домен, порты, адаптеры

Гексагональная (ports & adapters). Внутри — домен и usecase; снаружи — адаптеры двух родов:
**driving** (входные: CLI, опц. `mtt-ui`) вызывают `core`; **driven** (выходные: хранилище)
вызываются из `core` через порты.

- **`pkg/mtt`** — публичный контракт: домен-типы (`Task`, `Type`, `Flow`, `Status`, `Comment`,
  `Note`, `Config`), базовый порт **`TaskStore`** и **`KnowledgeStore`**, плюс опциональные
  capability-интерфейсы (см. ниже). Публичный — чтобы внешние Go-адаптеры его реализовали.
- **`internal/core`** — usecase-логика (add/list/ready/переход-flow/поиск): работает **только
  через порты**, о конкретном хранилище не знает.
- **`internal/adapter/yaml`** — дефолтный *driven*-адаптер: реализует оба порта поверх `.mtt/`;
  **минтит ID/slug** (плоский, per-prefix — напр. `e1`/`t17`).
- **`internal/adapter/exec`** — реализация порта **`Runner`** (запуск команд перехода); в тестах
  подменяется фейком. `Runner` определён в `core` (нужен только ему, не третьим сторонам).
- **`cmd/mtt` + `internal/cli`** — тонкий CLI (*driving*): разбор → usecase → форматирование;
  на старте собирает адаптеры по конфигу и внедряет в core.
- **`cmd/mtt-ui`** — *опциональный driving-адаптер*: маленький локальный веб-сервер (view/CRUD,
  Гант) поверх того же `core`. Не нужен агентам и не нужен при внешнем бэкенде с собственным UI.

Два порта независимы: **«связка» = сконфигурированная пара адаптеров**, при этом можно
миксовать (напр. задачи из Jira, а КБ — локальный YAML). Дефолт — YAML для обоих.

**Внешние бэкенды** подключаются двумя путями: (1) in-process Go-адаптер, реализующий
`pkg/mtt`-порты (для этого контракт и публичный); (2) — позже — субпроцесс-адаптер по
документированному wire-протоколу (JSON stdin/stdout), не требующий импорта нашего Go.
Сейчас проектируем **шов** и отдаём YAML-адаптер; рантайм внешних адаптеров — когда понадобится.

> **Открытый вопрос (внешние адаптеры):** чей flow главный — наш конфиг или родной workflow
> бэкенда (напр. Jira), и как наши `commands` соотносятся с его переходами. Для YAML-дефолта
> вопрос не стоит; решаем при проектировании конкретного внешнего адаптера.

### Возможности адаптера (capabilities)

Не каждый бэкенд умеет всё: внешний трекер может не давать историю переходов, дерево
комментариев, зависимости или поиск. Поэтому возможности **опциональны на уровне адаптера**:

- **Обязательный минимум** — базовый `TaskStore` (CRUD + list/get + минтинг ID): реализует любой адаптер.
- **Опциональные возможности** — отдельные интерфейсы поверх базового: `HistoryStore`,
  `DependencyStore`, `CommentStore` (дерево), `SearchStore` (и сам `KnowledgeStore`). Адаптер
  реализует что умеет; `core` проверяет наличие через type-assertion (идиоматичный Go).
- **Дискаверабилити** — метод `Capabilities()` у бэкенда: CLI/агент знает, что доступно
  (`mtt caps`). Отсутствующая возможность даёт типизированный `ErrUnsupported` с внятным
  сообщением, а не молчаливый сбой.
- **YAML-адаптер — референс**: реализует **все** возможности (пишет историю и т.д.). Внешние
  адаптеры частичные; `core` написан на минимум и «зажигает» доступное поверх.

> **Инвариант слоёв:** `core` не импортирует `adapter/*`; адаптеры не содержат бизнес-правил
> (только CRUD/запросы за портом). Правила (flow, ready, циклы) — в `core`; **минтинг ID/slug —
> в адаптере** (кодировка ID backend-специфична).

## Почему дефолт — файлы (YAML-адаптер)

Главная боль beads — демон/локи/бинарная БД: конфликты с плагинами и болезненные
merge-конфликты в ветках агентов. Файл-на-задачу это чинит:

- нет демона и фонового состояния — `mtt` это stateless CLI (прочитал → изменил → записал);
- чистые git-мержи: два агента в разных ветках добавляют разные задачи без конфликта;
- задачи ревьюабельны в PR как обычный diff.

Масштаб трекера (сотни–тысячи задач) позволяет грузить всё в память на каждый вызов;
поиск, диаграмма Ганта и зависимости считаются in-memory. SQLite для этого не нужен.

## Раскладка данных (YAML-адаптер)

```
.mtt/
  config.yaml            # проект, типы задач и flow (общий, коммитится)
  config.local.yaml      # персональный оверлей: параметры подключения, локальные настройки (gitignored)
  tasks/
    e1.yaml              # эпик 1
    t17.yaml             # задача 17 (parent: e1)
    s3.yaml              # подзадача 3 (parent: t17)
  knowledge/
    <slug>.md            # заметки БЗ (markdown + YAML frontmatter); базовый store — t47 (поиск + версионирование [t6])
```

`.mtt/` **коммитится** (это данные проекта). Это раскладка **YAML-адаптера**; правки — только
через порт (`TaskStore`/`KnowledgeStore`), руками файлы не редактируем, иначе теряется
детерминизм и валидация.

## Конфигурация (слои и локальные оверрайды)

Конфиг собирается слиянием слоёв (следующий перекрывает предыдущий) — так общие проектные и персональные
настройки держатся раздельно:

1. встроенные дефолты (шаблон `mtt init`);
2. опциональный глобальный пользовательский конфиг (`$XDG_CONFIG_HOME/mtt/config.yaml`) — персональные
   дефолты между проектами;
3. **`.mtt/config.yaml`** — общий, коммитящийся конфиг проекта (**типы и flow — здесь**);
4. **`.mtt/config.local.yaml`** — **gitignored** персональный оверлей для этого проекта;
5. переменные окружения / флаги CLI — высший приоритет.

Локальный оверлей — для **персональных параметров подключения к внешним бэкендам** (напр. разные
пользователи/креды Jira в одном проекте) и локальных настроек. **Креды никогда не кладём в коммитящийся
`config.yaml`** — только в gitignored-оверлей или env. Перекрывать общие **типы/flow** в локальном оверлее
не рекомендуется (рассинхронит команду). Deep-merge по ключам/секциям. Шов заложен сейчас (loader +
gitignore); богатые схемы подключения per-адаптер — вместе с внешними адаптерами.

## Типы и иерархия (домен) vs ID/slug (адаптер)

**Домен** знает про *логическую* задачу: её **тип**, **родителей** и **flow**. Тип задаёт:

- `name` — имя типа (напр. `epic`, `task`, `subtask`);
- `description` — опциональная ориентация для человека/агента;
- `parents` — имена допустимых родительских типов (пусто = корневой уровень) — **этим задаётся иерархия**
  (тип может стоять под несколькими родительскими типами); обратное (дети) — **вычисляется**, не хранится;
- `default` — помечает тип, используемый `add` без `--type` (не более одного; полный YAML-провайдер
  помечает ровно один); литерала `task` в коде нет;
- `statuses` (каждый — **value-объект** с категорией `kind`: initial/active/terminal) / `transitions` —
  flow (ниже).

Иерархия эпик → задача → подзадача — **не хардкод**, а следствие шаблона `hierarchy`
(плоский шаблон `default` даёт `task`+`chore`):
`epic` (root) ← `task` (`parents: [epic]`) ← `subtask` (`parents: [task]`).

**Именование (ID/slug) — дело адаптера, не домена.** `core` создаёт логическую задачу
(«task типа X под родителем Y»), а конкретный ID минтит `TaskStore`: у YAML это плоский `t17`,
у Jira — `PROJ-123`, у GitHub — `#42`. Поэтому `prefix` — поле **YAML-адаптера** (в его
`config.yaml`), и ID-генерация живёт **в адаптере**, за портом.

В YAML-адаптере ID **плоский и per-prefix**: `<prefix><N>`, где `N` — последовательный по префиксу
(`max+1`, `O_EXCL`). ID **не** кодирует цепочку родителей (`epic` #1 → `e1`; `task` #17 → `t17`;
`subtask` #3 → `s3`), так что идентичность отвязана от позиции: **смена родителя** меняет только
поле `parent` — ID остаётся стабильным, файл не переименовывается. ID **стабилен** и не зависит от
текста; имя — в поле `title`; иерархия — в `parent` и **вычисляется** для отображения (напр. `mtt show`
рисует линию наследования). Имя файла = `<id>.yaml`.

### Инварианты модели (проверяются загрузкой конфига)

Чисто **структурные и не завязанные на имена** — в коде **нет** литералов имён типов/статусов или
структуры ID. Типы/иерархия/категории — из конфига (домен), кодировка ID — из адаптера; дефолты — в
шаблоне `mtt init`, не в логике.

- **`kind` определяется топологией flow** (и валидируется против объявленного значения): `initial` = нет
  входящих (≥1 исходящее); `active` = ≥1 входящее и ≥1 исходящее; `terminal` = нет исходящих (≥1 входящее).
  `kind` — это **value-объект** (`StatusKind`), а не имя — та абстракция, что держит имена вне кода.
- **≥1 каждой категории на flow** (`initial`/`active`/`terminal`). Минимальный flow: `initial → active →
  terminal` (flow из 2 статусов невалиден). **Несколько `initial`-статусов допустимо** (пользователь сам
  решает, сколько точек входа хочет). Логика `ready`/завершённости/`list` работает **по категории**,
  никогда по литеральному имени.
- **Flow — замкнутый граф на тип.** Идентичность статуса — `(type, name)`: одноимённые статусы в разных
  flow — **разные**; **межфлоу-переходов нет**. Всё пространство статусов — **лес непересекающихся
  per-type flow**.
- **Reopen** — переход в *отдельный `active`-статус*, никогда обратно в `initial` и никогда из `terminal`
  (терминалы финальны) — так история задачи остаётся линейной и честной.
- **Дефолтный тип** помечается `default: true` — не более одного на уровне домена (`DefaultType`
  откатывается к первому типу); полный YAML-провайдер обязан пометить **ровно один**. Литерала `task` нет.
- **Входной статус.** Задача стартует в **initial**-статусе своего типа. Когда во flow больше одного
  `initial`, точка входа — тот, что помечен `default: true` на статусе (зеркалит дефолтный маркер **типа**),
  иначе — первый `initial` в порядке конфига (`Type.InitialStatus`). Валидация: не более одного `default`-
  статуса на flow, и `default`-статус обязан быть `initial`.
- **Размещение `add` / `--no-parent`.** Тип, у которого `parents` не пуст, требует родителя (`--parent`);
  как **осознанное исключение**, `--no-parent` создаёт его на верхнем уровне (плоский корневой ID). Корневым
  типам не нужно ни то, ни другое. (Так пользователя никогда не блокирует создание задачи верхнего уровня.)
- **Санитарность иерархии:** каждый элемент в `parents` типа именует существующий тип; тип не является
  собственным родителем.
- **Тип задачи неизменяем.** Смена типа = **рекатегоризация**: закрыть старую задачу (→ терминал),
  создать новую целевого типа и связать их через `refs` (kind `task`, backlinks в обе стороны).

> **Известное ограничение YAML-адаптера (осознанный размен):** sequential-ID коллизируют при
> параллельном создании в разных ветках — `e2` в двух ветках даёт видимый git add/add конфликт.
> Приемлемо для low-concurrency; при росте параллелизма — namespace-префикс по branch/agent.
> У других адаптеров (напр. Jira) своя схема без этой проблемы.

### Доменная модель vs сериализация (DDD)

- **Чистый домен.** Типы `pkg/mtt` не несут **никаких** yaml/json-тегов и **никаких** полей адаптера
  (`prefix` — только у адаптера). У каждого адаптера свои DTO, которые он **маппит** в домен и обратно;
  контракт никогда не зависит от формата хранения.
- **Ссылки по идентичности.** Внутри flow переходы ссылаются на статусы по **имени**; между агрегатами
  (типы, а позже задачи) ссылки — **имена/ID**, никогда не указатели. **Back-references вычисляются**
  (обратный индекс: дети, backlinks), не хранятся — forward-ссылки единственный источник правды.
- **Разрешённый граф — производный.** Связанный неизменяемый in-memory граф объектов (для обхода: `ready`,
  `advance`, детект циклов) строит `internal/core` по мере надобности — он **не** часть контракта.
- **Провайдер-агностичность.** Домен требует **обязательный минимум** (у `Type` есть имя + flow, чьи
  статусы имеют name+kind, а переходы — from/to) и считает остальное опциональным (`description`, `parents`,
  `default` и `commands` — последнее *наше* локальное усиление-гейт, отсутствующее во внешних трекерах).
  Поэтому типы/flow могут приходить от **внешнего провайдера** (подключается позже через `mtt connect`);
  YAML-адаптер — **полный провайдер** (даёт всё).

## Модель задачи

Поля сериализуются в фиксированном порядке (порядок полей структуры) → детерминированный diff.

```yaml
id: s3
type: subtask
title: fix login redirect loop
status: in_progress
parent: t17
depends_on:
  - t2
refs:
  - {kind: note, id: auth-design, label: spec}
  - {kind: task, id: t2}
created: 2026-07-03T09:20:00Z
updated: 2026-07-03T10:00:00Z
description: |
  Многострочное описание.
comments:
  - id: 1
    author: agent
    created: 2026-07-03T09:25:00Z
    body: первый коммент
    replies:
      - id: 2
        author: human
        created: 2026-07-03T09:40:00Z
        body: ответ (дерево через вложенные replies)
history:
  - {at: 2026-07-03T09:25:00Z, by: agent, role: implementer, from: tbd, to: in_progress}
  - {at: 2026-07-03T10:00:00Z, by: agent, role: implementer, from: in_progress, to: done,
     checks: [{cmd: "make lint", exit: 0}, {cmd: "make test", exit: 0}]}
```

- `parent` — пусто для эпика.
- `depends_on` — список ID (в т.ч. межэпиковых); **блокирующее** ребро (влияет на `ready`).
- `refs` — проверяемые ссылки на `note`/`task`/`comment`/`url` (см. «База знаний и ссылки»);
  **информационные**, не блокирующие. Комментарии тоже могут нести `refs`.
- `comments` — дерево через вложенные `replies`; `id` комментария последователен в пределах задачи.
- `history` — **append-only** аудит переходов (`from→to`, `at`, `by`, результаты `checks`);
  задним числом не восстановить, поэтому пишем сразу — основа аудита и реконструкции графа.

## Список (`list`) и правка (`edit`) — сессия 003

- **Порядок по умолчанию в `list` provider-agnostic.** Первичный ключ — **доменная временная метка**:
  `Created` по убыванию (сначала свежие), либо `Updated` по убыванию с `--sort updated` — никогда не
  структура ID (у внешнего адаптера ID может не сортироваться осмысленно). Тай-брейк — сравнение ID как
  **непрозрачной строки**, так что при равных метках порядок остаётся стабильным и детерминированным между
  запусками.
- **`edit` трогает только заголовок/описание.** Смена статуса идёт через flow (`status`/`advance`/…), чтобы
  гейты соблюдались; смена родителя (`parent`) и смена типа — **отдельные операции**, не `edit` — уже
  отмечено выше и в бэклоге.

## Иерархия: размещение и рендер — сессия 004

- **`add --parent <id>` — нормальный путь размещения.** Ребёнок хранит только `parent` (forward-ссылка по
  ID); адаптер по-прежнему минтит **плоский per-prefix** ID, так что идентичность независима от позиции.
  Размещение валидируется как мутация в `core.Adder`: родитель должен существовать, а его **тип** — быть
  разрешён полем `parents` дочернего типа (чистый предикат `Type.AcceptsParent`). `--no-parent` остаётся
  escape-hatch.
- **Дети/предки вычисляются, не хранятся.** `core.Index` — производное in-memory представление из
  `TaskStore.List` (чистое значение — без стора и часов; «резолвнутый граф производный», не часть контракта
  `pkg/mtt`): parent→children, цепочка предков, корни; cycle-safe; сироты (висячий parent) всплывают как
  корни. Порядок сиблингов переиспользует provider-agnostic сортировку `Select` (`Created` desc, тай-брейк по ID).
- **`tree` и родословная в `show` — чистые чтения** (без usecase): `mtt tree` рисует лес (фильтр
  keep-ancestors, `--depth`, вложенный `--json`); `mtt show` печатает breadcrumb-родословную. Единый предикат
  `core.Match` (status/type/kind/parent) переиспользуется `list` и обходом `tree` (DRY).

## Flow: исполняемые переходы (killer-фича) и `mtt init`

Тип задаёт **flow** — граф статусов с переходами. На каждый переход можно повесить:

- `description` — текст «что именно делаем» (понимание для агента/человека);
- `commands` — последовательность консольных команд; **все обязаны вернуть 0**, иначе переход
  **блокируется** (задача остаётся в исходном статусе).

Это превращает flow из совета в **исполняемый гейт + действие**. Примеры:

- `in_progress → done`: `["make lint", "make test"]` — не пускаем в `done`, пока не зелено.
- `tbd → in_progress`: ревью спеки + создание ветки под задачу.

Смысл: **агент работает в терминах задач** (`mtt start t17`, `mtt done t17`), а механику
статус-flow (проверки, ветка, …) прячет сам переход — меньше отвлечения на детали.

Исполнение — за портом **`Runner`**: `core` оркеструет переход, вызывает `Runner`, гейтит по
exit-кодам. `Runner` определён в `core`, реализован в `internal/adapter/exec`, в тестах — фейк.
Команды идут по порядку, прерываемся на первом ненулевом; рабочая директория — корень проекта;
таймаут на команду (на Unix по таймауту SIGKILL'ится **вся группа процессов** команды — фоновый демон,
порождённый гейтом, не переживёт дедлайн; на Windows — best-effort); escape-hatch `--no-run`
(форсировать переход без команд, для аварий).
Команды берутся из конфига (доверенный, как Makefile/git-хуки), не из сети.

> **Отгружено (s006, `mtt status <id> <new>`):** **одно** гейтящееся ребро. `Runner.Run(commands)` без
> `dir` — `exec`-адаптер конструируется с `cwd = корень проекта` (core без файловых путей). Ненулевой exit —
> **данные** (`Check`), не Go-ошибка; заблокированный гейт оставляет задачу неизменной и не пишет `history`.
> Таймаут на команду задаётся конфигом (`command_timeout`, дефолт `5m`, адаптерная настройка). Core-sentinels
> `ErrBlocked`/`ErrInvalidTransition` маппятся в exit-коды `3`/`6`. Достаточно линейного поиска ребра в
> `Type.Transitions` (пока без `ResolvedFlow` — он окупится в мульти-обходе `advance` в s007). Мета-обход
> `advance`/`start`/`done` и режимы — s007. Гейт печатает **live-прогресс пайплайна** в stderr (`▶`/`✓`/`✗` +
> тайминг на команду); вывод самих команд по умолчанию скрыт, стримится с `-v`, и/или пишется с `--log-file` —
> **но заблокированный гейт эхом печатает последние ~10 строк вывода упавшей команды под её строкой `✗` и
> подсказывает `-v`/`--log-file`** (s008.97/U2), так что агент видит, *почему* упало, без повторного прогона
> всего гейта. `by` резолвится `--by` > `MTT_BY` > `author` из `config.local.yaml` (долговременный персональный
> дефолт; `role` — только флаг/env).

> **Отгружено (s006.5, атрибуция + verb-сахар):** `--why` пишет долговременную свободную причину перехода
> (`HistoryEntry.Why` — единственное изменение `pkg/mtt` — рендерится в `mtt show`); `--who` — симметричный
> алиас `--by` (взаимоисключающие; та же `history.by`). **Verb-сахар** `mtt <status> <id>` — одноребёрный
> перевод через CLI **fallback-роутинг** (`root.RunE`: ровно 2 арга, где arg0 — не зарегистрированная
> команда, arg1 — существующая задача, arg0 — статус во flow её типа → маршрут в путь `status`, переиспользуя
> `core.Transitioner`; реальная команда выигрывает клеш имён; всё прочее — `unknown command`, код 1) —
> forward-compatible к `advance` без изменения поверхности. **Required-атрибуция:** проектно-глобальный
> `require: {who, why}` в коммитящемся конфиге (адаптерный `Settings.Require`, как `command_timeout`;
> `config.local` может только **ужесточать** — захват до локального оверлея и OR-комбинация) проверяется в
> `core.Transitioner` **до** гейта (fail fast; `--no-run` не байпасит), агрегируя все недостающие поля в одну
> `ErrMissingAttribution` → код `2`. `-v`/`--log-file` стали root-persistent; `--no-run` остался локальным у
> `mtt status` (сахар не может обойти гейт).

> **Отгружено (s008.98, именованные переходы + edge-verb-сахар):** у перехода может быть опциональное
> **`Name`** (`pkg/mtt.Transition.Name` — единственное доменное изменение, простая открытая метка как
> `Description`), дающее семантический глагол для **ребра из текущего статуса** (`decline` для `review → fix`).
> Это достраивает **триаду резолва**: переход по **статусу-цели** (явно `mtt status [<id>] <status>`, сахар
> `mtt <status> [<id>]`) и по **имени ребра** (явно `mtt do [<id>] <edge>`, сахар `mtt <edge> [<id>]`). Чистая
> `Type.FindTransitionByName(from, name)` резолвит имя ребра в статус-цель, а CLI идёт **существующим**
> `runTransition(to)` — `core.Transitioner` и путь гейта **не тронуты**. Три новых структурных инварианта
> (`Config.Validate`) держат это безопасным: `Name` ребра **уникален в пределах исходного статуса** и
> **дизъюнктен с именами статусов** типа (edge-first-приоритет сахара не затеняет статус), и каждая пара
> `(from,to)` **уникальна в типе**. **Trust-граница (без изменений):** путь перехода не ревалидирует
> (валидация — на `add`/`types`), поэтому роутинг по `to` корректен ровно там, где инварианты держатся — та же
> граница, на которую уже опираются отгруженные `applyCurrent`/`FindTransition`. `mtt types`, подсказка `next:`
> и `next[].name` в `show --json` показывают глагол.

> **Реализовано (t5): опасные операции обязаны требовать `--who`/`--why`.** Обходящие флоу / рискованные
> действия принудительно требуют атрибуцию независимо от проектного `require` (можно пропустить гейт, но нужно
> за это подписаться). **Итоговое** требование — объединение `глобальное ∨ per-edge ∨ --no-run`, только
> ужесточение:
> - **`--no-run`** форсит **и who, и why** на любом ребре (проверка в `core.Transitioner` до гейта; покрывает
>   `mtt status`/`mtt do`);
> - переход можно пометить **критическим** через per-edge `require: {who, why}` в конфиге
>   (`mtt.Transition.Require`, декодируется YAML-адаптером, объединяется в core);
> - разрушительный **`rm --force`** форсит who+why как **pre-flight**-предусловие (`core.Remover`,
>   `ErrMissingAttribution` → exit 2 и для одиночного, и для bulk) и на каждый id пишет запись в
>   `.mtt/audit.log` **до** удаления — нет разрушения без следа. Лог — новый driven-порт (`mtt.AuditStore`),
>   JSONL (`{at, who, why, action, id}`), коммитится с `.gitattributes` `merge=union`.
>
> Роли / авторизация («кому *можно* делать X») остаются запаркованными — t5 только про атрибуцию.
> `init --force` вне scope (бутстрап, ещё нет истории `.mtt/`).

> Оговорка (для планирования): команды с побочными эффектами (создание ветки) ставим **после**
> проверок; при падении после сайд-эффекта переход не коммитим, но сайд-эффект уже случился —
> это на совести порядка команд. Двухфазную модель (checks → actions) вводим лишь при нужде.

> **Реализовано (t21): post-persist actions (вторая фаза).** У перехода может быть per-edge список `post:`
> (та же форма `Command`, что и `commands:`), выполняемый **после** записи статуса — фаза финализации. Репо
> использует его для авто-коммита `.mtt` на каждом move (`git add .mtt && git commit … -- .mtt`), так что move
> коммитит своё состояние сам (убирая прежний ручной `git add .mtt` после каждого move и два интерим-коммита
> после `deliver`/`cancel`; их pre-gate `git switch main` идёт первым, так что post-коммит попадает на main).
> **`deliver`/`cancel` сужают этот pathspec** до файла задачи (+`.mtt/audit.log`, если он есть), чтобы грязный
> `.mtt/config.yaml` не уехал в коммит/пуш на main мимо ревью (c3 — см. заметку SEC2 в разделе про dogfood).
> **Две фазы, разная семантика провала:** `commands:` гейтят **вход** (провал → нет записи, работает
> компенсация s008); `post:` финализируют **после** входа (провал → move **сохранён**, `core.ErrPostAction`,
> CLI exit **5** — mtt не откатывает состоявшийся move из-за икоты post). `--no-run` пропускает **обе** фазы.
> Почему не глобальный `post`-дефолт? Вопросы прецеденции/слияния/opt-out отложены (t24) — пока только per-edge.
> `git switch` в `deliver`/`start`/`cancel` — ровно причина, почему наивная одна фаза «persist → всё → откат»
> не работает: переключение контекста обязано предшествовать записи, а коммит — следовать за ней. **(c1)
> авто-пуш** расширяет это: post у `approve` также делает `git push -u origin task/<id>` (ветка для PR), у
> `deliver` **и `cancel`** — `git push origin main` (c5 — терминальное состояние не должно жить только
> локально). `approve` также открывает/обновляет PR (`gh pr create`, идемпотентно; тело из
> `docs/superpowers/pr/<id>.md` или fallback — t27), так что остаются ручными **мерж** и подтянуть main перед `deliver`.

> **Сделано (s008): откат / компенсация (внутри-пайплайновая).** У gate-команды можно объявить `rollback:`
> компенсатор (скаляр или `{run, timeout}`, тоже с экспансией плейсхолдеров); когда **следующая** команда
> того же пайплайна падает, компенсаторы уже успевших команд исполняются в **обратном порядке** — отменяя
> сделанное частичным гейтом (напр. `git checkout -b task/{{.ID}}` с `rollback: git branch -D task/{{.ID}}`).
> VO `Command` получил аддитивное `Rollback *Command` (**лист** — свой rollback обязан быть nil). Решения:
> **(1)** компенсатор **на команде** (`Command.Rollback`), не на переходе — 1:1 к «обратно по успевшим»;
> **(2)** `core.Transitioner` считает план (кто успел — по единому индексу отказа — в обратном порядке), а
> exec **`Runner.Compensate`** его исполняет; **(3)** компенсация **best-effort** (гнать все, продолжать сквозь
> упавший компенсатор) и **не маскирует отказ гейта** — исход остаётся `ErrBlocked` (exit 3); **(4)** **без
> `history`** при заблокированном+компенсированном переходе (файл задачи не тронут — `HistoryEntry` это запись
> о переходе, а компенсация — событие уровня side-effects); **(5)** rollback'и разворачиваются **заранее**
> вместе с прямыми командами (кривой rollback-шаблон — exit 1 до любого side-effect). Свой rollback упавшей
> команды не гонится. Гейт печатает живую фазу `↩ compensating (N)`, а сообщение блока несёт сводку
> `compensated N …`. См. sessions/008 и TASKS.md → e4_t10.
>
> **Ещё отложено:** компенсация через **несколько** рёбер (обрыв `--atomic`/мульти-шаг `advance` после
> сайд-эффектов) — s008 только **внутри-пайплайновая** (пайплайн одного ребра); и второй уровень компенсации
> (компенсатор компенсатора, отвергается `Valid()`).

> **Реализовано (s007): структурные команды.** `Transition.Commands` эволюционировал из `[]string` в
> value-object `Command` `{run, timeout?}` (чистый VO в `pkg/mtt`; `rollback?` — аддитивно, отложен до s008).
> Две возможности: (1) **раскрытие плейсхолдеров** в `run` — `.ID`/`.Type`/`.From`/`.To` (напр.
> `git checkout -b task/{{.ID}}` на `tbd → in_progress`); (2) **таймаут на команду** поверх глобального
> `command_timeout` (fail-fast, если быстрая команда затянулась). **Обратная совместимость:** голый скаляр в
> YAML ⇒ `{run: …}` (кастомный `UnmarshalYAML` в DTO принимает скаляр **или** map `{run, timeout}`); обе формы
> схлопываются в один `Command` на границе адаптера — выше неё формы уже нет. Четыре решения (брейнсторм):
> **(1)** таймаут на команду живёт в **доменном VO** (авторское свойство ребра, неотделимое от `run`), а
> глобальный `command_timeout` остаётся **execution policy** адаптера (`Settings`) — раннер резолвит
> per-command-иначе-глобальный; **(2)** экспансия — в **`core.Transitioner`** (у него task+edge), поэтому
> `pkg/mtt` **не знает про шаблоны** (хранит сырой шаблон, core разворачивает перед `runner.Run`), а exec-адаптер
> остаётся тупым; **(3)** защита от инъекций — **структурный whitelist**: `text/template` над структурой лишь с
> четырьмя безопасными по форме полями, поэтому свободный текст (`title`) не интерполируется, а `{{.Title}}` —
> ошибка экспансии (квотинг не нужен; если когда-нибудь выставим free-text поле — обязателен shell-quote,
> документированный шов); **(4)** `Runner.Run` принимает `[]Command` (Run уже развёрнут; `Check.Cmd` пишет
> развёрнутую команду для правдивого аудита). Ошибка экспансии обрывает переход как обычная ошибка (exit 1),
> в отличие от блокировки гейта (`ErrBlocked`, exit 3). См. sessions/007 и TASKS.md → e4_t9.

> **Шов (отложено, подумать): status-actions на узлах.** Сейчас исполняемые пайплайны висят только на
> **рёбрах** (переходах — меняют статус и гейтят). Но «зафиксировать промежуточный результат / собрать /
> прогнать проверки» **не меняя** статус — это операция на узле, которой негде жить (self-loop-переход —
> хак: ложная history, сломанная топология). Обобщаем: у статуса могут быть **именованные откатываемые
> пайплайны-действия**, каждый вызывается как **кастомный глагол** `mtt <action> <id>` в текущем статусе
> задачи. Общий примитив — *именованный откатываемый пайплайн команд* — тогда висит либо на **ребре**
> (переход), либо на **узле** (статус), оба обслуживают `Runner` + rollback. Это достраивает историю «вся
> shell-обвязка живёт во флоу / агент работает чисто в терминах задач». **Заблокировано на** структурных
> командах + rollback (переиспользует `Command`-VO + компенсацию) и на грамматике резолвинга аргументов
> (кастомные глаголы конфликтуют с реальными командами и status-сахаром — напр. `mtt check` занят под
> проверку refs); аудит не-transition действия цепляется за edit-audit. **Открытый вопрос: нужно ли до
> релиза?** Лин — *нет*: для релиза агент делает промежуточный commit обычным `git` пока `in_progress`; это
> полировка полноты, не минимум. Пересмотреть, когда приземлятся структурные команды (до них всё равно
> заблокировано). См. TASKS.md → Later.

> **Рабочий контекст: текущая задача (готово, s006.7).** Текущая ветка git, но для задач — убивает
> повтор id. **Значение** живёт в `config.local.yaml` (`current: t17`, персональное/gitignored); **правило**
> его установки/снятия — **свойство перехода** в коммитящемся flow: аддитивное поле `Transition.Current`
> (`set`|`clear`, value object `CurrentAction`; name-agnostic; дефолтные шаблоны ставят на «взять в работу» /
> снимают на →`done`, но →`cancelled` не трогают). **Опущенный id** резолвится в текущую задачу **только для
> одиночных прямых глаголов** (`status` / `mtt <status>` / `show` / `edit`) — никогда для
> `list`/`tree`/`dep`/`ready`/filter/stdin/bulk (порядок: явный id > current; тир filter/stdin из s008.9
> встанет между ними позже). Компаньон `mtt use [<id>] [--clear]` ставит/показывает/снимает без перехода.
> **Два решения (брейнсторм):** (1) указатель — **capability-порт** (`mtt.CurrentStore`), а не хелпер CLI:
> «взять в работу» — возможность, которой реальный провайдер может владеть сам (assignee), поэтому YAML пишет
> в `config.local`, а внешний адаптер маппит на свою фичу либо возвращает `ErrUnsupported`. В отличие от
> паркованного `DependencyStore`, порт оправдан **сейчас**: `current` **невстраиваемый** (персональный,
> одно-значный, не состояние задачи) — даже YAML нужен отдельный store — это тот самый случай GAP #1, что
> оправдывает порт. (2) **Применяет CLI** после успешного перехода (читая `Current` ребра через общий примитив
> `Type.FindTransition`); `core.Transitioner` не тронут — интерпретация объявленного эффекта ребра механична,
> не политика, поэтому живёт в composition root. **Кавеат:** общий checkout с несколькими агентами имеет один
> `config.local` = одну `current` → коллизия; per-agent current цепляется за вопрос идентичности субагентов
> (норм для соло / один-агент-на-checkout). **Бэклог (подумать):** раздельные команды `mtt current`/`use` для
> эргономической чистоты; показ `description` статуса/перехода при переходе (напоминание в потоке агента);
> резолвинг current для **всех** одиночных операций, включая чтение (`tree <id>`, `dep list`); multi-assignee
> провайдеры («моя current», когда назначено несколько); seam переиспользования `advance` (вынести общий
> core-шаг применения эффектов ребра, когда паркованный мульти-рёберный обход разопётся). `CapCurrent` +
> `Capabilities()` придут с `mtt caps` (e4_t6). См. TASKS.md → e4_t8a.

### Продвижение по flow: `advance` / `start` / `done`

> **ЗАПАРКОВАНО (2026-07-05, по требованию).** `advance` и глаголы `start`/`done`/`cancel`, режимы
> (`--stop`/`--atomic`/`--force`), а также **роли-на-рёбрах** — **отложены**: норма — переход в **одно
> ребро**, поэтому `mtt status` покрывает почти всё, а мульти-обход решает несуществующую пока проблему;
> роли-условия на рёбрах ≈ RBAC и преждевременны (идентичность/роль может позже прийти из внешнего
> провайдера). Этот подраздел — **дизайн-намерение** на случай, когда флоу реально ветвится; пока ничего из
> него не строим. Эргономику даём раньше и дёшево через **verb-сахар** `mtt <status> <id>` (s006.5) —
> одноребёрный перевод через fallback-роутинг по неизвестному первому аргу (не динамическая регистрация
> команд), forward-compatible: семантику можно нарастить single-edge → `advance` позже без смены поверхности.
> Атрибуция — `--who`/`--by` (кто) + `--why` (постоянная свободная причина в `history`). Ниже — целевая
> семантика.

`start`/`done` — не одиночный переход, а **мета-команда**: примитив `advance <task> --to <status>`
проводит задачу по цепочке переходов до целевого статуса, исполняя гейты на рёбрах. `start` =
`--to <active>`, `done` = `--to <terminal>` (встроенные алиасы, резолвятся по категории, не по литералам
имён). **Config-DSL команд не делаем** — режимы это флаги.

Семантика обхода (предсказуемость важнее «умности»):

- идём только по рёбрам, **ведущим к цели**; **никогда не заходим в чужой терминал** (не
  авто-`cancel`); защита от циклов; цель недостижима → ошибка;
- на настоящей развилке (≥2 прогрессирующих ребра) — **стоп, а не догадка**;
- каждое пройденное ребро исполняет свои `commands` и пишет запись в `history`;
- не-`ready` задачу (незакрытые `depends_on`) в терминал по умолчанию не гоним — предупреждаем.

Благодаря линейному дефолтному flow (`initial → active → terminal`) дефолтный случай однозначен; развилки —
только если пользователь сам добавил ветвление. (Имена статусов — из конфига, никогда из кода.)

Режимы (флаги):

- `--stop` (**дефолт**) — двигать до первого упавшего гейта или развилки, сообщить где/почему;
- `--atomic` — всё-или-ничего **по статусу** (упал гейт → не двигаем и не пишем переходы);
  **оговорка:** сайд-эффекты уже выполненных команд не откатываются;
- `--force` — двигать безусловно, игнорируя гейты (аварийный; обобщает `--no-run`).

### Роли — шов (реализацию откладываем)

Семантика `start`/`done` (и переходов вообще) зависит от **роли** вызывающего: у агента-ревьюера
`done` значит иное, чем у имплементера (напр. имплементер `done` уводит в `review`, ревьюер
`done` — из `review` в терминал). Полноценно это — роли в конфиге + `--role`/`MTT_ROLE` +
опциональный тег роли на переходах; резолвер `advance` тогда параметризуется ролью.

**Сейчас не реализуем**, но закладываем швы:

- **Единственный не-откладываемый шов — писать `role` в `history`** (рядом с `by`): задним
  числом не восстановить (как и саму историю). Поле резервируем в модели сразу.
- `advance` и verb→target — резолвер **параметризуем ролью** (сигнатура принимает role); сейчас
  role-agnostic (одна неявная роль).
- CLI резервирует `--role=` / `MTT_ROLE`; конфиг может дорасти до секции `roles` и role-тега на
  переходах — аддитивно, loader forward-compatible.

Ограничения (чтобы не разрослось): роли — **семантический роутинг** (что значит глагол для роли),
**не** RBAC/энфорсмент (агенты кооперативны — роутим, не полицейские). Имена ролей — из конфига,
в коде не хардкодятся (как типы/статусы).

> **Связь с текущей задачей (подумать, s006.7).** «Взять в работу» (`Transition.Current: set`) —
> **контекстно-ролевое** действие: `current` — лёгкая одно-значная форма assignee, а «моя» — пер-актор/пер-роль.
> Отгруженное (один глобальный указатель в `config.local`) — *вырожденная одно-ролевая проекция*; при разморозке
> ролей `current`, вероятно, станет пер-акторным, а `Transition.Current` — роль-обусловленным. Решать **вместе**
> с идентичностью субагентов, не раньше. См. TASKS.md → Later («`current` vs roles»).

> **Направление (отложено): профили актора.** Секцию `roles` ожидаемо разворачиваем в именованные
> **профили**, связывающие идентичность с ролью — `(by, role)`, напр. `(coding-agent, implementer)` /
> `(Alice, reviewer)` — ровно один помечен `default: true` и применяется, когда не заданы ни `--profile`, ни
> `--by`/`--role`. Так как mtt используют в основном **кодовые агенты**, делящие конфиг репозитория с
> человеком, профиль агента — эргономичный дефолт, а человек переопределяет. `mtt profile add/list/rm`
> управляет **только** персональными профилями в `config.local.yaml` (общие проектные профили, если есть,
> для команды read-only). Это **поглощает шов `author` из s006** (`author` = `by` дефолтного профиля) и
> forward-compatible. См. TASKS.md → Later.
>
> **Жёсткое предусловие (настоящий вопрос): идентичность субагентов.** Роли/RBAC бессмысленны, пока мы не
> умеем различать субагентов с **разными** ролями при мультиагентном доступе — именно на этом всё держится.
> Пока нет механизма идентичности (пер-агентный `config.local`, env/handshake/токен или идентичность из
> провайдера), `by`/`role` — самодекларируемые строки, а профили — лишь эргономичные дефолты, не энфорсмент.
> Идентичность субагентов решаем **до** воскрешения ролей. При этом атрибуция release-complete и без всего
> этого: `--who`/`--why` (свободный текст) + проектно-глобальный `require: {who, why}` в конфиге
> (исполнительная/адаптерная настройка, как `command_timeout`; `config.local` может только ужесточать),
> проверяемый до гейта и агрегирующий все недостающие поля в одну usage-ошибку (exit 2). Роли/профили — в парковке.

`mtt init` пишет `.mtt/config.yaml` из шаблона `default` — плоские корневые типы `task`+`chore` с линейным
flow (`initial → active → terminal`, плюс второй терминал для отмены). Эти имена — из **шаблона**, не из кода.
**Команд по умолчанию нет** — их навешивает пользователь под свой проект. Пример `epic`/`task`/`subtask`
ниже — это шаблон `hierarchy` (`mtt init --template hierarchy`).

```yaml
version: 1
project:
  name: my_tt
types:
  - name: epic
    description: Большой объём работы, охватывающий несколько задач.
    prefix: e                    # prefix — поле YAML-адаптера (кодировка ID), не домена
    parents: []                  # корневой уровень
    statuses:
      - {name: tbd,         kind: initial}
      - {name: in_progress, kind: active}
      - {name: done,        kind: terminal}
      - {name: cancelled,   kind: terminal}
    transitions:
      - {from: tbd,         to: in_progress}
      - {from: tbd,         to: cancelled}
      - {from: in_progress, to: done, description: "все задачи эпика закрыты"}
      - {from: in_progress, to: cancelled}
  - name: task                   # ДЕФОЛТНЫЙ тип (add без --type)
    description: Единица работы под эпиком.
    prefix: t
    parents: [epic]
    default: true
    statuses:
      - {name: tbd,         kind: initial}
      - {name: in_progress, kind: active}
      - {name: done,        kind: terminal}
      - {name: cancelled,   kind: terminal}
    transitions:
      - {from: tbd,         to: in_progress, description: "ревью спеки, создать ветку"}
      - {from: tbd,         to: cancelled}
      - {from: in_progress, to: done, description: "гейт качества"}
      - {from: in_progress, to: cancelled}
  - name: subtask
    description: Небольшой шаг внутри задачи.
    prefix: s
    parents: [task]
    statuses:
      - {name: tbd,         kind: initial}
      - {name: in_progress, kind: active}
      - {name: done,        kind: terminal}
      - {name: cancelled,   kind: terminal}
    transitions:
      - {from: tbd,         to: in_progress}
      - {from: tbd,         to: cancelled}
      - {from: in_progress, to: done}
      - {from: in_progress, to: cancelled}
```

Навесить команды — правкой перехода (без изменений в коде):

```yaml
      - from: in_progress
        to: done
        description: "гейт качества"
        commands: ["make lint", "make test"]   # все → 0, иначе done заблокирован
```

Так же добавляется тип `bug` (`prefix: b`, `parents: [epic]`, flow с active-статусом `review`) — только конфигом.

`mtt init --template coding` кладёт примерные coding-типы — `feature`/`bugfix`/`refactor` — каждый со своим
гейтящим Definition of Done (ветка + lint/test; у `bugfix` ещё сначала падающий тест; у `refactor` — без
diff публичного API), как готовое демо ценности enforcement. Запускаемый end-to-end проход по этому
шаблону — в `demo/`.

## Зависимости

- `depends_on: [id, …]` — **блокирующее** ребро (отличное от иерархического `parent` и информационных
  `refs`). Едет на поле `Task` и round-trip'ится через `TaskStore.Update` — **без выделенного порта** для
  YAML-референса (capability `DependencyStore` нужен только внешним адаптерам, которые не могут встроить
  поле). Добавление ребра отклоняется, если оно создаёт **цикл** (правило `core`), и self-ребро отклоняется;
  проверка цикла строит производный `core.DepGraph` над `depends_on` (параллельно `Index` над `parent` из
  s004, держится **отдельно** — у графов разная форма).
- Задача **ready** ⇔ её статус не `terminal` И все `depends_on` в `terminal`-статусе (по категории `kind`:
  `done` или `cancelled`, а не по литералу). Готовность **консервативна**: неразрешимый блокер (dangling
  `depends_on` или статус не из текущего flow) оставляет задачу **не** ready — `ready` требует позитивного
  подтверждения.
- `mtt ready` — «что можно брать в работу»; `mtt list --ready` — шорткат-компаньон (один общий примитив
  `core.Ready` за обоими). `mtt dep list <id>` показывает прямые блокеры задачи и её вычисляемых зависимых,
  с `--tree` (транзитивно) и `--cycles` (по проекту, defensive).
- **Зависимости при создании (`add --depends-on`, s008.5).** `depends_on` можно задать при создании, не только
  через `dep add`: цели валидируются (каждая должна существовать) и **дедуплицируются** в `core.Adder` до
  `Create`. Проверка цикла не нужна — id новой задачи ещё не сминчен, поэтому она не может быть целью
  (само-/обратное ребро невыразимо). Типизировано насквозь; конверсия строка→`TaskID` — только на границе CLI.

> **Удаление (`mtt rm`, s008.5).** Hard-delete — это **операция стора**, поэтому она получает метод базового
> порта `TaskStore.Delete(id)` (*D* в CRUD; не embedded-поле, так что правило «поле едет на `Update`» его не
> покрывает — YAML-reference делает `os.Remove`, внешний архив-only адаптер вернёт `ErrUnsupported`). Это не то
> же, что `cancel` (терминальный *статус*): `rm` — для **гигиены бэклога** (снести ошибочную задачу), а не
> нормальный жизненный цикл, и **не пишет `history`** — аудит = git-коммит, убирающий файл. `core.Remover`
> **консервативен**: по умолчанию **отклоняет** удаление задачи, на которую ссылаются (ребёнок через `parent`,
> зависимая через `depends_on`, **или входящий `ref`** — t1, cross-store, через вычисляемый `Backlinks`),
> чтобы удаление не оставляло молча висячие ссылки; `--force` переопределяет, оставляя их **висячими** (уже
> терпимо: `ready` консервативен, `Index` поднимает сирот в корни). Это **agent-facing** — без интерактивного
> подтверждения (оно повесило бы агента). **`--force` требует `--who`/`--why` + audit-запись** (t5,
> dangerous-ops; базовый `rm` без `--force` ничего не пишет — аудит есть git-коммит). Отсутствующий id — код
> `4`, первый потребитель теперь **единообразной** not-found-таксономии (каждый путь с одиночной задачей по id
> оборачивает `mtt.ErrNotFound`).
>
> **Оговорка, вскрытая `rm --force` — переиспользование id (пред-существующее ограничение минтинга).** YAML-адаптер
> минтит id как `max+1` на префикс без high-water mark, поэтому удаление задачи с **наибольшим** номером
> освобождает её id для переиспользования последующим `add` — что молча перенаправляет висячую ссылку
> (`depends_on`/`parent`, оставленную `--force`) на новую, неродственную задачу. `rm` впервые делает это
> достижимым. Оно латентно (нужны `--force` + удаление max-id + последующее создание) и коренится в схеме
> минтинга, а не в удалении; корректный фикс — монотонный / never-reuse минтинг (см. TASKS.md → Later). Пока
> предпочитайте `cancel` для referenced-задачи или сначала снимите висячие рёбра.

> **Отложенный дизайн-вопрос — семантика `cancelled`-блокера.** `terminal`-блокер разблокирует зависимую
> задачу, а `cancelled` — это `terminal`-`kind`, поэтому задача с блокерами `done` **и** `cancelled`
> формально ready — хотя *cancelled* (брошенный, а не выполненный) блокер, возможно, означает, что зависимую
> надо пересмотреть, а не молча разблокировать. Текущее поведение сохраняет terminal-по-`kind` (cancelled
> разблокирует). **Пересмотрено в s006** (терминалы теперь достижимы через `mtt status`): решение —
> **сохранить** terminal-по-`kind` (cancelled разблокирует); корректный фикс (различие succeeded-vs-abandoned,
> hard/soft-ребро) требует нового доменного моделирования, которое flow-gate-сессия не должна протаскивать, и
> рисковал бы name-agnostic принципом. s006 добавляет e2e (`cancel_unblock`), демонстрирующий текущую
> семантику на достижимом состоянии; более глубокий фикс остаётся отложенным. См. TASKS.md → Later.

## Приоритеты и roadmap — сессия 008.6

- **Приоритет — третья ось порядка** (наряду с `depends_on` и иерархией). Это закрытый упорядоченный
  **value object** `Priority` (`high|medium|low`) — идиома `StatusKind`, а **не** голая строка/число и **не**
  конфигурируемые уровни (YAGNI — настраиваемые уровни это revisit-at-second-caller). Пусто = **не задан**,
  сортируется как `medium` (нейтральный дефолт) и **не пишется на диск** (`omitempty`, поэтому существующие
  файлы задач байт-в-байт не меняются; back-compat). Едет на `Task.Priority` + `TaskStore.Update` — **без
  нового порта** (правило GAP #1, как `depends_on`) — и позже маппится на нативный priority/labels провайдера.
  Валидность проверяется на **границе CLI** (`--priority` отклоняет неизвестное значение); повреждённое
  значение на диске **терпимо** (сортируется как `medium`), зеркалируя ленивую валидацию `status` против flow.
  Задаётся через `--priority` на `add`/`edit` (`edit --priority ""` сбрасывает), фильтруется через `list
  --priority` (совпадает по *сохранённой* метке — задача без приоритета попадает только когда фильтр не задан),
  сортируется через `list --sort priority`, показывается в `show` + `--json` (`taskJSON.priority`). **Порядок
  `list`/`ready` по умолчанию не меняется** (свежесть); приоритет — opt-in.
- **`mtt roadmap [--json]` — производный вид «порядок исполнения»** — **чистый `core`-read** (`Roadmap(tasks,
  cfg) []RoadmapEntry`, без store/clock, **не** в контракте `pkg/mtt`), как `Ready`/`Select`. Он возвращает
  **не-терминальные** задачи в порядке исполнения по **двум осям «идти после»** — `depends_on` (явное
  блокирующее ребро) **и `parent`** (предок закрывается только когда закрыты потомки, поэтому не-терминальный
  потомок идёт перед не-терминальным предком). **Обе оси — жёсткие ограничения.** Каждая запись аннотируется
  `ready` (через `core.Ready` — единый источник истины) и `blocked_by` (записи `depends_on`, не
  terminal-satisfied), а предок дополнительно перечисляет своих не-терминальных потомков как `contains`.
  **Готовность остаётся только про `depends_on`** — ось `parent` влияет на *порядок* и аннотацию `contains`, но
  не на готовность (поэтому предок с открытыми потомками может быть `ready: true`, но стоять последним; тип
  контейнера/эпика может сам нести флоу и артефакты, значит он и задача, и контейнер — его не спец-кейсят и не
  фильтруют).
- **Приоритет распространяется (мягкий тай-брейк).** Вместо сортировки по *своему* приоритету блокер получает
  **эффективный** ранг = `min(свой, min по всему, что он транзитивно разблокирует по обеим осям)` — так
  высокоприоритетная задача **тянет свои предпосылки вперёд**, впереди независимой низкоприоритетной работы
  («чтобы закрыть важное — сделай сначала его блокеры»). Тай-брейк priority-guided Kahn — `(эффективный ранг,
  свежесть)`; детерминирован и **cycle-safe** по обеим осям (мемоизированный DFS эффективного ранга; застрявший
  узел — в цикле или ниже по течению, в т.ч. кросс-осевом — дописывается best-effort, поэтому функция всегда
  завершается и возвращает все узлы). Он строит **свой** граф, ограниченный не-терминальными задачами (не
  переиспользует `DepGraph`, чьи `Dependents` нефильтрованы — GAP #6 не выносится) и переиспользует общий
  предикат `terminalSatisfied`, вынесенный из `Ready`. Это **не** *временнóй* планировщик (без дат/критического
  пути) и, намеренно, **не** `list --sort priority` (тот сортирует по своему приоритету; roadmap
  распространяет). Это то, что агент спрашивает вместо повторного вывода «что делать дальше и чего оно ждёт» из
  сырых задач; мотивирует dogfooding (s009).

## Версионирование flow и история задач

- **Определение flow версионируется git'ом** (в YAML-адаптере): `git log .mtt/config.yaml` — вся
  эволюция графа. Явные `flow-version`/миграции существующих задач при переименовании статуса —
  откладываемо. Домен трактует `status` как **данные** и валидирует по текущему flow **лениво**
  (не падает, а флагует расхождение), чтобы дрейф конфига не ломал старые задачи.
- **История переходов** пишется в саму задачу (`history`, append-only) — это даёт аудит и, позже,
  **реконструкцию наблюдаемого графа** (read-only агрегация историй всех задач). Пишем в задаче,
  не в центральный лог, чтобы сохранить чистые пофайловые мержи. История — **опциональная
  возможность** (`HistoryStore`): YAML-адаптер её пишет, внешний бэкенд может не поддерживать
  (тогда `ErrUnsupported`), и `core` деградирует мягко.

## База знаний и ссылки (`refs`)

**KB — опциональная capability** (`KnowledgeStore`, как Confluence поверх Jira). Если её нет,
знания живут прямо в задачах и комментариях; «база знаний» — это они и ссылки между ними.

> **Готово (t47): базовый `KnowledgeStore` + `mtt note`.** Create/list/show/edit/rm markdown-заметок, один
> файл на заметку в `.mtt/knowledge/<slug>.md` — YAML-frontmatter (`title`/`tags`/`created`/`updated`, DTO-
> структура ради детерминированного порядка) + markdown-**тело verbatim**; slug — это **имя файла**, во
> frontmatter не дублируется. Идентичность — явный, **структурно валидируемый `NoteSlug`** (kebab-ASCII —
> защита от path-traversal, ре-валидация в каждом методе адаптера, строящем путь, и при загрузке).
> `CreateNote` — **reserve-then-write** (`O_CREATE|O_EXCL`, затем атомарный temp+rename) — без тихой
> перезаписи. Теги переиспользуют словарь задач (только явный `--tag` — **без** извлечения хэштегов из тела).
> Заметка **одноверсионна** (история — git); **версионирование + полнотекстовый поиск отложены в t6**
> (отдельные опциональные capability), а **вся работа с `refs` — в т.ч. на заметках — это t1**.
> `mtt.ErrNotFound` переиспользуется (exit 4) через CLI-обёртку `noteNotFound`. `CapKnowledge`/
> `Capabilities()` пока нет (e4_t6).

**Заметки будут версионироваться (t6, отложено).** Будущая capability `KnowledgeStore` не будет разрушать
содержимое: сохранение заметки создаст **новую версию со ссылкой на предыдущую**, история сохранится — тот же
принцип, что и `history` переходов задачи. YAML-референс реализует сам (файлы также трекает git); внешние
бэкенды (Confluence, …) полагаются на своё **нативное** версионирование. Доменный шов — у `Note` есть
идентичность версии + ссылка на предшественника. **Базовый store t47 — одноверсионный** (история — git);
версионирование отложено в t6.

Задачи и комментарии несут **`refs`** — структурный список проверяемых ссылок:

```yaml
refs:
  - {kind: note, id: auth-design, label: "spec"}
  - {kind: task, id: t2}
  - {kind: url,  id: "https://…"}
```

- `kind` ∈ `note` | `task` | `comment` | `url`. Это **не** `depends_on` (то — блокирующее ребро);
  `refs` — информационная, проверяемая на целостность связь.
- **Верификация capability-aware:** `task`/`comment` резолвятся через `TaskStore` (всегда);
  `note` — только при `KnowledgeStore` (иначе «не могу проверить: нет KB»); `url` — внешняя,
  не резолвим (опц. HEAD-проверка позже).
- **Семантика:** на запись — предупреждаем о битой ссылке (не хард-блок); `mtt check` — обход на
  висячие ссылки; `mtt show` — ссылки и **backlinks** («что ссылается сюда»); при удалении цели —
  **по умолчанию отказ** (удаление необратимо; `--force` перекрывает, с подписью).
- **Фазы:** поле `refs` — в модель уже в Фазе 1; вся проводка + верификация + backlinks + `mtt check` — **t1**.

> **Выпущено (t1): ссылки проведены и верифицируются.** `Ref{Kind,ID,Label}` живут на **задачах и заметках**
> (`Note.Refs` добавлено; task-адаптер YAML уже round-trip'ил `Task.Refs`); виды `note`/`task`/`url` — живые,
> **`comment` (вид и носитель) отложен в t2** (цели ещё нет — дисциплина t47 «не выпускать мёртвый вид»).
> Идентичность — натуральный ключ `(kind, target)`; наборы ссылок дедуплицируются и **сортируются**. CLI:
> `mtt ref add/rm/list` (задачи) + `mtt note ref add/rm/list` (заметки) + creation-time `--ref` на
> `add`/`note add`; `ref rm` отсутствующего ключа — **идемпотентный no-op** (как `dep rm`/`tag rm`).
> **Верификация capability-aware и производная** (`core.RefStatus` — `ok`/`dangling`/`unverified`, не хранится):
> `task`/`note` резолвятся через сторы (`note` нужен подключённый KB, иначе `unverified`); `url` внешний →
> всегда `unverified`. **Политика записи = warn-not-block** (висячая цель сохраняется с предупреждением, выход
> 0; носитель обязан существовать, иначе выход 4; битый `kind:target`/URL/slug — выход 1). **Backlinks —
> вычисляемое cross-store значение** (`core.Backlinks` из снапшота задач+заметок — инвариант «back-refs
> вычисляются, не хранятся»), показывается в `mtt show`/`note show`/`ref list`. **`mtt check`** — read-only
> обход, **выход 7** при висячих (0 на чистом/unverified) — годен как гейт/CI. **Удаление — refuse-by-default +
> `--force`, единое и cross-store:** `mtt rm`/`mtt note rm` консультируют один вычисляемый `Backlinks` (так что
> задачу, на которую ссылается **заметка**, тоже блокирует) и отказывают без `--force`, который — как
> разрушающий override — требует `--who`/`--why` + audit-запись (новый `core.NoteRemover` зеркалит task
> `Remover`; `core.Remover` **не** получает KB-порт — потребляет вычисленное значение). Self-ссылка не блокирует
> собственное удаление. **Ещё в t2:** `comment`-refs (цель + носитель); **follow-up:** `mtt check --fix`,
> проверка живости URL (HEAD).

> **Выпущено (t51): KB prime — курированный дайджест на старт сессии.** У заметок появляется **ось важности** —
> `Note.Priority` переиспользует task-`Priority` VO (high/medium/low; unset = medium в порядке, `omitempty` на
> диске). **`mtt prime`** — чистый derived read (`core.Prime`, как `Roadmap` — вне контракта), печатает
> **pointer-only** markdown-дайджест **важных** заметок для инжекта в начале сессии (аналог beads
> `prime`/`remember`). **Opt-in по построению (safety-модель):** попадают только заметки с **явным** priority ≥
> `--min-priority` (дефолт `high`) — заметка **без priority никогда** не в дайджесте — и **тела не выводятся**;
> дайджест ограничен (`--limit`). Ранжирование = бэнд priority, затем **backlink-count** (переиспользуя
> вычисляемый `Backlinks` из t1), затем свежесть. Priority заметки также задаётся через `--priority` на
> `note add`/`note edit` и виден в `note list --priority`/`--sort priority`. **sessionStart-хук — конфиг, не
> код** — mtt даёт только команду; хук — документированный сниппет `settings.json`. БЗ остаётся
> **вспомогательной** фичей (позиционирование), теперь с реальным prime-сюжетом. **Follow-up:** опц. snippet
> тела в дайджесте, URL/`--full` варианты.

> собственную разработку в закоммиченном `.mtt/` (конфиг + задачи). **Модель — одна ось:** трекаем
> **продукт** (задача = единица изменения продукта), а не **процесс** (session/phase = как *мы* работаем —
> это остаётся в `sessions/*.md` + git); структура — **deps + tags + priority**, без иерархии (**эпики**
> продуктово-валидны, но отложены; §4-гейт вернётся с ними). **ДВА типа.** `task` (дизайн ОТКРЫТ):
> `tbd → speccing → spec_review → spec_human_review ⇄ spec_fix → planning → plan_review →
> plan_human_review ⇄ plan_fix → implementing → impl_review ⇄ impl_fix → approved → done` (+`cancelled`);
> `chore` (дизайн УЖЕ ЗАФИКСИРОВАН в другом месте — финдинг ревью, записанное решение, docs-sync): только
> impl-стадия + хвост доставки. Гейты проверяют **форму, никогда содержание** (содержание = ревью-статусы):
> наличие id-keyed артефакта (`ls docs/superpowers/specs/<id>-*.md`) на spec/plan submit, `make check`
> (per-command таймаут 10m) на impl submit, и **верифицированная доставка** — `deliver` переводит дерево на
> main и грепает сквош-заголовок (`<id>: …` — заголовок PR, пробрасываемый настройкой репо
> `squash_merge_commit_title=PR_TITLE`) в локальном `git log`, так что **`done` честно означает «в main»**.
> Веточный контекст механизирован: `start` re-enter'ит или создаёт `task/{{.ID}}` от main (гард: файл
> задачи обязан существовать в дереве); `cancel` и `deliver` пишут терминальное состояние НА main;
> `approved → decline` возвращает на ветку задачи. Артефакты **коммитятся рано** (id-keyed имена
> `docs/superpowers/specs|plans/<id>-<slug>.md` (+ опциональное `docs/superpowers/pr/<id>.md` тело PR — t27) —
> конвенция v1 «незакоммичено до ревью» мертва).
> **Оставшиеся конвенции:** заголовок PR начинается с `<id>: `; оставшиеся ручные шаги — **мерж**
> и подтянуть main перед `deliver` (гейт проверяет локальный лог и fail-closed
> подсказывает «git pull first»). (С t21 каждый move авто-коммитит `.mtt` через per-edge `post:`, с c1
> `approve` авто-пушит ветку задачи, а `deliver`/`cancel` авто-пушат main (c5), и с t27 `approve`
> открывает/обновляет PR (`gh pr create`) — прежние ручные state-коммиты, PR-open и конвенция «запушь до мержа» ушли.)
> **Атрибуция:** глобальный `require: {who}` (per-edge/роль-require требует правки ядра — запаркованная
> roles-работа; мигрированная задача `dangerous-ops` — её первый триггер). **Доверие (SEC2):** гейты зовут
> **только read-only** `mtt` — никогда mtt-переход (рекурсия). **Известные пределы (записаны):**
> человеческий акт impl-стадии не оставляет записи в mtt-history — аудит человека это merge-след git
> (human sign-off спеки/плана свои записи сохраняют); задача, заблокированная посреди флоу, не имеет
> статуса парковки (отдыхает где стоит); межветочные stale-чтения — задокументированный lost-update класс
> (multi-agent кластер). **Бэклог** = `tbd`-задачи с тегом `backlog` (промоушен = снять тег). **Авторинг:**
> команды-гейты — **одинарные** YAML-скаляры. Закоммиченный конфиг стережёт `TestRepoDogfoodConfig`
> (**единственная** load-time валидация — `Config.Validate` зовётся на `add`/`types`, не на `Load`; гард
> грузит временную копию, минуя оверлей `config.local`). **Bootstrap-оговорки:** mtt-id (`t1`…) ≠
> исторические `sNNN`; ветка без slug (whitelist плейсхолдеров); сам s009 шёл на ручной
> `feat/s009-dogfood` (конфиг правит *будущие* задачи). Спеки —
> `docs/superpowers/specs/2026-07-09-session-009-dogfood-design.md` (модель, миграция) +
> `docs/superpowers/specs/2026-07-11-flow-v2-mechanized-delivery-design.md` (флоу, авторитетная).

**Позже (бэклог):**

- позже — **смена родителя** (`mtt reparent`/`move`): изменить `parent` задачи; возможно благодаря плоским,
  не завязанным на позицию ID.
- **теги — выпущено s008.7** (для ведения бэклога): сквозной набор лейблов на `Task.Tags` (едет на поле +
  `Update`, **без нового порта** — GAP #1, как `depends_on`). Два способа задать: **`#хэштеги` в title/
  description — главный путь** (извлекаются и мёржатся в канонический набор при `add`/`edit`, текст не
  переписывается), а явные `--tag` / `mtt tag add/rm` — вторичные/точечные. Согласование **при записи** —
  дельта текста на `edit` снимает тег, когда его `#хэштег` уходит из текста, и сохраняет ручные теги (провенанс
  не хранится, так что коллизия текст+ручной снимается вместе с текстом); `tag rm` **защищён** (отклоняет тег,
  чей `#хэштег` ещё в тексте — правьте текст). `list`/`ready`/`tree` фильтруют по `--tag` (OR-внутри, включающий) и
  `--exclude-tag` (негативный) — слайс-измерения `ListFilter` поверх `Match`. Вокабуляр тегов — чистая пара `pkg/mtt` (`NormalizeTag`/`ExtractTags`) над **Unicode**-
  charset'ом (`\pL\pN._-`, Unicode `ToLower`, без NFC-фолдинга); теги — нормализованный+отсортированный
  **набор** (открытый словарь → просто `[]string`, не VO). Спека:
  `docs/superpowers/specs/2026-07-09-session-008.7-tags-design.md`.
- **батч и пайплайн — реализовано s008.9** (mtt как Unix-композируемый CLI): переиспользуемый **селектор
  множества задач** для всех set-операций — явные ID | `--filter` (предикаты `list`
  `--status/--type/--kind/--parent/--priority/--tag/--ready` поверх `Select`/`Match`) | **stdin `-`** (ID
  построчно), **взаимоисключающие** (>1 или 0 источников = usage error; источник задан, но пуст = no-op, exit
  0) — плюс вывод **`--ids`** у `list`/`ready` (взаимоисключающий с `--json`), чтобы конвейеры собирались:
  `mtt list --tag x --ids | mtt tag rm x -`. Сначала для `tag add/rm` и `rm` (нет гейтов). Селектор — **дело
  CLI** (I/O stdin/флагов; ветка `--filter` переиспользует `core.Select`/`Ready`), без core-поверхности; он
  **никогда** не резолвит указатель `current`. Массовые мутации **best-effort по элементам** с отчётом
  (успехи на stdout, отказы по элементам на stderr / массив по элементам в `--json`) и **общим exit 1** при
  любом отказе (git-style; агрегат не оборачивает per-item sentinel, поэтому не мапится ошибочно на exit 3/4).
  **`--dry-run`** показывает затронутые ID без мутации. Раскладка аргументов **контекстно-зависима** для `tag`:
  маркер селектора (`-` или filter-флаг) делает позиционные ТЕГАМИ, задачи берутся из селектора
  (`mtt tag add urgent --status tbd`); без маркера одиночная форма `mtt tag add <id> <tag>…` неизменна.
  **Массовый `rm` учитывает подграф** (`core.Remover.RemoveMany`): проверка ссылок считает только рефереров
  ВНЕ набора удаления, поэтому `mtt rm <epic> <child>` удаляет поддерево одной командой без `--force`
  (одиночный `rm <id>` сохраняет exit-4 not-found и одиночную проверку ссылок). Bulk
  `status`/глаголы/`edit`/`dep` — **позже** (гейты + частичный успех + атомарность). Спека:
  `docs/superpowers/specs/2026-07-09-session-008.9-batch-design.md`.
- позже — **доски / views**: запрос/представление над тегами/статусом/типом (связано с `list` и `mtt-ui`);
  бэклог — такое представление.
- позже — **устойчивый, не зависящий от git аудит правок**: `edit` сегодня только двигает `updated`, а
  фактической историей служит git; журнал изменений или версионирование полей (аддитивно, без слома
  обратной совместимости) сделали бы историю правок доступной для запросов без git. В паре с
  **источником субъектной идентичности (`By`)** — кто «действует» для атрибуции, вероятно поле
  `.mtt/config.local.yaml`, отдельное от `--role` (`--role` — это *какую шляпу* носят, `By` — это *кто*).
  Оба отложены; поле `history` остаётся **только про переходы** (фаза 3) — оно не расширяется на обычные
  `edit`.
- позже — **валидация значений фильтра `list`**: сейчас `list --status X`/`--type Y` фильтрует только по
  задачам, поэтому неизвестное значение даёт пустой результат — неотличимо от «валидный статус, задач пока
  нет». Валидировать значения фильтра по конфигу (статус должен существовать в *каком-то* flow, тип — быть
  сконфигурированным типом) и падать с ошибкой на неизвестном значении, чтобы «нет задач в этом статусе» и
  «такого статуса не может быть во flow» различались. Требует, чтобы `list` загружал конфиг (сейчас он читает
  только хранилище).
- позже — **`mtt edit <id> --editor`**: открыть задачу в `$EDITOR` (править поля интерактивно / через
  отрисованный файл) вместо передачи `--title`/`--description` в командной строке — человеко-удобная
  альтернатива правке через флаги.
- позже — **отступ многострочного описания в `show`**: `mtt show` печатает описание через `"\n  %s\n"`,
  поэтому у многострочного описания отступ `  ` получает **только первая** строка, а продолжения идут вровень
  слева. Данные корректны (round-trip как YAML block scalar) — это косметический нит в `formatTask` (из
  сессии 002). Фикс: отступать каждую строку описания. Дёшево, косметика, не блокирует.

```
internal/
  cli/                     # cobra-команды (тонкие) + wiring адаптеров по конфигу
  core/                    # usecase-логика: иерархия, зависимости, циклы, flow — только через порты
  adapter/
    yaml/                  # дефолтный driven-адаптер: порты поверх .mtt/ файлов
docs/
  architecture/model.go    # снимок доменной модели в форме кода (целевой контракт; в бинарь не входит)
```

> **Снимок доменной модели.** [docs/architecture/model.go](docs/architecture/model.go) — code-form,
> tiered (T1/T2/T3) индекс целевой поверхности контракта: домен-типы + порты + опциональные capability,
> core-usecase с их зависимостями, производный резолвнутый граф и открытые пробелы. Это дизайн-референс
> (компилируется, lint-clean, browsable через `go doc`), в бинарь **не** входит; авторитетна эта проза, а
> снимок — её структурная карта. Два слоя: **A** (контракт/персистентные агрегаты) — идентичность
> типизированным значением (`TaskID`/`TypeName`/`StatusName`), граница сериализации/провайдера; **B**
> (резолвнутый граф в core) — указательные связи для обхода, не сериализуется.

## Релиз

Упаковка и процесс релиза (кросс-платформенные бинарники через `make release`, workflow на тег, `SHA256SUMS`)
описаны в [RELEASING.md](RELEASING.md). Версия выводится из git-тега (SemVer; см. [RELEASING.md](RELEASING.md)).

### Самообновление (t44)

`mtt self-update` замыкает петлю: потребляет опубликованные ассеты релиза + `SHA256SUMS`, чтобы обновить
установленный бинарь на месте. По гексагону добавляет три driven-порта в `pkg/mtt` — `ReleaseSource` (latest +
fetch, GitHub HTTP-адаптер, фейкается в тестах), `BinaryReplacer` (платформенная самозамена) и `GoInstaller`
(fallback через тулчейн) — за чистым `core.SelfUpdater`. `Prepare` возвращает **детерминированный** `Plan`
(`UpdateAvailable` / `NoUpdate` / `Undetermined`, с `Via ∈ {asset, go-install, none}`) — ошибается только когда
релиз не резолвится, поэтому `--check-only` никогда не падает на резолвимом релизе. `Apply` проверяет SHA-256
**до** любой записи (первичный путь — asset+checksum; `go install …@<tag>` — fallback, когда проверяемого
ассета под платформу нет). Самозамена атомарна (Unix: temp в той же папке + `rename`; Windows: rename-then-swap
— **не верифицировано в CI**, нет раннера). Сравнение версий переиспользует t30 `resolveVersion` +
`golang.org/x/mod/semver`; dev/голый-SHA current не упорядочивается и отклоняется без `--force`. Поскольку
сетевой fetch и самозамена негерметичны, они тестируются через фейки; реальный e2e — ручной real-binary smoke
на `impl_review`.
