# mtt

**Исполняемая state machine для задач кодовых агентов — предохранитель между агентом и словом «done».**

> 🇬🇧 In English: [README.md](README.md)

> **Статус:** рабочая альфа (`0.8.98-dev`). Фазы 1–3 реализованы — `init`, `add`/`show`/`list`/`edit`,
> иерархия (`tree`), зависимости (`dep`/`ready`) и **flow-гейт** (`mtt status <id> <new>` или сахар
> `mtt <status> <id>` исполняют команды перехода и блокируют на красном гейте, пишут history; структурные
> команды + rollback включены).
> Мета-обход `advance`/`start`/`done` — **отложен** (одно-рёберный `status` — норма); база знаний, поиск,
> `mtt-ui` и внешние адаптеры — поздние фазы. Полный план — в [DESIGN.ru.md](DESIGN.ru.md).

> **Питч.** Кодовые агенты умеют писать код, но плохо соблюдают жизненный цикл задачи — «готово» для них
> часто просто текстовый ярлык. mtt превращает задачу в исполняемую state machine: переход статуса проходит
> через гейты — создать ветку, запустить lint/test, проверить артефакты, — и если гейт красный, `mtt done`
> не проходит. Definition of Done **свой на каждый тип** (bugfix, refactor, feature). Это не commit-хук и
> не CI — это гейт *жизненного цикла задачи*, в терминах агента, над твоим хранилищем: zero-footprint YAML
> для одиночки или тонкий enforcement-слой поверх Jira/GitHub для команды.

## Что это

`mtt` — небольшая Go CLI (`mtt add`, `mtt start`, `mtt done`, …). Ключевая идея: **config-driven per-type
flow, переходы которого запускают гейты** (консольные команды, обязанные пройти) — так агент не пометит
задачу `done`, не выполнив её Definition of Done. Вторая идея, лестница adoption: **хранилище
абстрагировано за портами**, поэтому тот же CLI работает поверх локальных файлов или твоего существующего
трекера.

- **Одиночка / zero-footprint:** дефолтный бэкенд — локальные YAML-файлы (один файл на задачу, в `.mtt/`) —
  без демона, без БД, без портов; чистые git-мержи и ревьюабельные в PR диффы.
- **Корпорация:** через адаптер направляешь mtt на уже имеющуюся связку «таск-трекер + база знаний»
  (Jira+Confluence, GitHub Issues+Wiki, …). Агенты получают тот же чистый CLI; люди остаются в родных UI.

## Зачем

`beads` хорош, но тяжёл для этой задачи: конфликтует с прочим тулингом, а его flow — плоский enum статусов.
Клин mtt — **лёгкость** и **настоящий per-type flow**, плюс способность адаптироваться к любому бэкенду,
который у тебя уже есть. (Честное сравнение — в [DESIGN.ru.md](DESIGN.ru.md#позиционирование-честно-vs-beads).)

## Ключевые идеи

- **Исполняемые per-type переходы (killer-фича).** На переход статуса вешаются консольные команды, которые
  обязаны все пройти — `["make lint", "make test"]` как гейт `→ done` — или выполнить действия (создать
  ветку). Definition of Done свой на каждый тип (bugfix/refactor/feature). Агент работает в терминах задач
  (`mtt start`, `mtt done`), а дисциплину держит инструмент.
- **Типы и иерархия — из конфига.** Эпик → задача → подзадача — лишь дефолтный конфиг; ни имён типов, ни
  структуры ID в коде нет. ID читаемые и **плоские** по префиксу (`e1`, `t17`, `s3`) — стабильны при
  ре-парентинге (иерархия живёт в поле `parent`, а не в ID).
- **Приоритеты, теги, зависимости и роадмап.** `--priority`, `#hashtags` / `--tag`, `depends_on` / `ready` и
  `mtt roadmap` дают агенту упорядоченный фильтруемый бэклог; указатель `current` держит задачу в работе, а
  batch/pipeline (`--ids`, stdin `-`) делает CLI Unix-компонуемым.
- **Гексагон, сменные бэкенды.** Публичный контракт (`pkg/mtt`) с портами `TaskStore` и `KnowledgeStore`;
  YAML-адаптер — референс. Опциональные возможности (история, зависимости, дерево комментариев, поиск) —
  capability уровня адаптера.
- **Проверяемые ссылки и append-only история.** Задачи/комментарии несут проверяемые `refs` на
  заметки/задачи; каждый переход статуса пишется для аудита.
- **Опциональный человеческий UI (`mtt-ui`).** Маленький локальный веб-сервер (управление задачами, Гант,
  браузер БЗ) — нужен только на YAML-дефолте; при внешнем бэкенде человек использует его родной UI.

## Для агентов

Управление задачами и зависимостями из командной строки, дерево комментариев, опциональное сохранение
знаний, простой текстовый поиск и опциональный hook внешнего индексатора.

## Для людей (опционально)

`mtt-ui` (маленький локальный веб-сервер): минималистичное управление задачами, диаграмма Ганта, браузер
БЗ. Плюс CLI (не TUI) — текстовый вывод инфо о задаче, поиска по БЗ и ASCII-Ганта.

## Установка

### Готовый бинарник (рекомендуется)

Скачайте бинарник под свою платформу из
[последнего релиза](https://github.com/pashukhin/mtt/releases/latest):

| Платформа | Файл |
|---|---|
| Linux x86-64 | `mtt_<version>_linux_amd64` |
| Linux ARM64 | `mtt_<version>_linux_arm64` |
| macOS (Intel) | `mtt_<version>_darwin_amd64` |
| macOS (Apple Silicon) | `mtt_<version>_darwin_arm64` |
| Windows x86-64 | `mtt_<version>_windows_amd64.exe` |

Затем, на Unix:

```sh
chmod +x mtt_<version>_<os>_<arch>
sudo mv mtt_<version>_<os>_<arch> /usr/local/bin/mtt
mtt version
```

Проверьте контрольную сумму по `SHA256SUMS` из того же релиза (`sha256sum -c` по строке вашего файла).

### Через Go-тулчейн

```sh
go install github.com/pashukhin/mtt/cmd/mtt@latest
```

### Сборка из исходников

```sh
git clone https://github.com/pashukhin/mtt && cd mtt
make build      # -> bin/mtt
```

## Быстрый старт

```sh
mtt init                              # создать .mtt/ (шаблон default: epic > task > subtask)
mtt add "Ship auth" --type epic       # -> e1
mtt add "Login endpoint" --parent e1  # -> t1
mtt status t1 in_progress             # движение по флоу (здесь запускаются + гейтят commands перехода)
mtt done t1                           # -> терминал done (в default-флоу нет gate-команд)
mtt list                              # все задачи
mtt tree                              # иерархия epic > task
mtt roadmap                           # порядок исполнения по зависимостям + приоритетам
```

Полный список команд и коды выхода — в [CLI_REFERENCE.ru.md](CLI_REFERENCE.ru.md).
