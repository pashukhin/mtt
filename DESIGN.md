# MY_TT — Design

Источник правды по архитектуре. Меняется только вместе с изменением поведения.
Верхнеуровневое «зачем» — в [project.md](project.md).

## Принятые решения

| Тема | Решение |
|---|---|
| Язык / форма | Go CLI `mtt` (`mtt add`, `mtt list`, …) |
| Хранение | YAML, **один файл на задачу**, каталог `.mtt/` в проекте |
| Источник правды | Файлы. БД/индекс (если появится) — производный, пересобираемый, gitignored |
| ID | Стабильный числовой `e1_t3_s2` + отдельное поле `title` |
| Хостинг | GitHub `github.com/pashukhin/mtt`, GitHub Actions |
| Ветвление | Ветка+PR на задачу → CI зелёный → squash в `main` |
| Гейт | `make check`: gofmt + vet + golangci-lint + `go test -race` |

## Почему файлы, а не SQLite

Главная боль beads — демон/локи/бинарная БД: конфликты с плагинами и болезненные
merge-конфликты в ветках агентов. Файл-на-задачу это чинит:

- нет демона и фонового состояния — `mtt` это stateless CLI (прочитал → изменил → записал);
- чистые git-мержи: два агента в разных ветках добавляют разные задачи без конфликта;
- задачи ревьюабельны в PR как обычный diff.

Масштаб трекера (сотни–тысячи задач) позволяет грузить всё в память на каждый вызов;
поиск, диаграмма Ганта и зависимости считаются in-memory. SQLite для этого не нужен.

## Раскладка данных

```
.mtt/
  config.yaml            # проект, типы задач и flow
  tasks/
    e1.yaml              # эпик 1
    e1_t3.yaml           # задача 3 эпика 1
    e1_t3_s2.yaml        # подзадача 2 задачи e1_t3
  knowledge/
    <slug>.md            # заметки БЗ (markdown + YAML frontmatter)   [фаза 5]
```

`.mtt/` **коммитится** (это данные проекта). Правки только через слой `store` —
руками файлы не редактируем, иначе теряется детерминизм и валидация.

## ID и иерархия

Три уровня, последовательная нумерация в пределах родителя:

- эпик: `e{N}`
- задача: `e{N}_t{M}`
- подзадача: `e{N}_t{M}_s{K}`

ID **стабилен** и не зависит от текста. Человекочитаемое имя — в поле `title`;
CLI показывает `e1_t3_s2  fix login redirect loop`. Имя файла = `<id>.yaml`.

Аллокация ID: сканируем существующие файлы в нужной области видимости, берём `max+1`.
Создание файла атомарно (`O_EXCL`); при гонке двух процессов — коллизия ловится и ID
переаллоцируется. Merge двух веток, обе создавшие `e2`, даёт git add/add конфликт —
это ожидаемо и заметно.

## Модель задачи

Поля сериализуются в фиксированном порядке (порядок полей структуры) → детерминированный diff.

```yaml
id: e1_t3_s2
type: bug
title: fix login redirect loop
status: in_progress
parent: e1_t3
depends_on:
  - e1_t2
created: 2026-07-03T09:20:00Z
updated: 2026-07-03T10:00:00Z
description: |
  Многострочное описание.
comments:
  - id: 1
    author: agent
    created: 2026-07-03T09:25:00Z
    body: первый коммент
    replies:
      - id: 2
        author: human
        created: 2026-07-03T09:40:00Z
        body: ответ (дерево через вложенные replies)
```

- `parent` — пусто для эпика.
- `depends_on` — список ID (в т.ч. межэпиковых).
- `comments` — дерево через вложенные `replies`; `id` комментария последователен в пределах задачи.

## Типы задач и flow

Отличие от beads: тип задаёт граф статусов с (текстовыми) условиями переходов.
Описывается в `config.yaml`, задача ссылается на тип, CLI валидирует смену статуса.

```yaml
version: 1
project:
  name: my_tt
defaults:
  type: task
types:
  - name: task
    initial: open
    statuses: [open, in_progress, done]
    transitions:
      - {from: open, to: in_progress}
      - {from: in_progress, to: done, condition: "make check зелёный"}
      - {from: in_progress, to: open}
  - name: bug
    initial: open
    statuses: [open, in_progress, review, closed]
    transitions:
      - {from: open, to: in_progress}
      - {from: in_progress, to: review, condition: "есть тест, воспроизводящий баг"}
      - {from: review, to: closed}
      - {from: review, to: in_progress}
```

Условие (`condition`) — текст: показывается агенту/человеку при переходе, не исполняется.

## Зависимости

- `depends_on: [id, …]`; при добавлении проверяем отсутствие циклов.
- Задача **ready** ⇔ её статус не терминальный И все `depends_on` в терминальном статусе.
- `mtt ready` — «что можно брать в работу».

## Поиск (фаза 5)

Встроенный текстовый поиск (подстрока/токены) по задачам и БЗ. Без RAG.
«Внешний индексатор» — опциональный hook (внешняя команда) через конфиг.

## serve / Гант (фаза 7)

`mtt serve` — встроенный `net/http` + `embed.FS`, минимальный web UI, Гант через SVG.
В CLI — текстовый/ASCII Гант. Самая поздняя фаза.

## Раскладка кода

```
cmd/mtt/main.go        # entrypoint
internal/
  cli/                 # cobra-команды (тонкие; логика — в engine/store)
  store/               # файловое хранилище: load/save, ID-alloc, атомарная запись
  model/               # Task, Type, Flow, Comment
  engine/              # иерархия, зависимости, циклы, flow-валидация
```
