## What & why (c9, `[dx]`)

`mtt tags` — a read-only command listing the tag vocabulary with per-tag task counts. Replaces the `mtt list … | jq -r '.[].tags[]?' | sort | uniq -c` workaround I ran by hand repeatedly to review the queue. Reclassified from task t39 (design fixed in a brainstorm → chore); its sibling c8 (`--exclude-tag`) shipped first.

## Design (config-free, pure read)

- **core** — `TagCounts(tasks) []TagCount` tallies tags, sorted count DESC then tag ASC (each task's `Tags` is a normalized set, so it counts once per tag). Pure, like `Roadmap`/`Select`.
- **cli** — `mtt tags` (distinct from the `tag` mutation command): `TaskStore.List → core.Select → TagCounts`. Default scope is **open** tasks (initial+active) — the bare command shows the live vocabulary. A status-scoping flag (`--all`/`--status`/`--kind`) sets the scope explicitly; content filters (`--type`/`--priority`/`--tag`/`--exclude-tag`/`--parent`) narrow within it. `--json` emits `[{tag, count}]`.

Output (this repo):

```
28  backlog
10  core
 9  flow
 …
```
