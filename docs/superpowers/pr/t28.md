## t28 — actionable attribution + post-failure errors

Make mtt's errors self-evidently actionable — they now say *how to recover*, not just *what failed*.

### What changed
- **exit 2 (missing attribution)** — the message now tells you how to set who/why: `author:` in
  `.mtt/config.local.yaml`, `MTT_BY`, `--who`, `--why`. One shared hint covers the transition path **and** the
  dangerous-ops paths (`--no-run`, `rm --force`, `note rm --force`).
- **exit 5 (post-action failed)** — the move IS persisted, so the CLI now prints a recovery block: *"the status
  change IS saved; do NOT re-run the move"* + the **exact remaining `post:` commands** to finish by hand (the
  failed one + those never reached), on **stderr in both text and `--json` mode**.
- **exit 4 (not found)** — points at `mtt roadmap` / `mtt list` (`mtt note list` for notes).
- **exit 6 (invalid transition)** — already listed the allowed moves; an invalid move **out of a terminal
  status** now reads cleanly (`no moves out of <status> — it is terminal`) instead of a dangling
  `allowed from <status>: `.

### Architecture (hexagon preserved)
- **`core`** — one change: a typed `core.PostActionError{Remaining, Cause}` replaces the plain-wrapped
  `ErrPostAction`. `Is()`→`ErrPostAction` keeps the exit-5 mapping; `Remaining` carries the unfinished post
  commands (`runsOf(expanded[i:])`, guarded against an empty check slice). No config/env/flag knowledge in
  `core`.
- **`internal/cli`** — all hint *phrasing* lives here: a context-free `exitHint(err)` hook in `Execute` (exit
  2 / exit 4), and the contextual recovery block in `runTransition` (exit 5, both modes). exit 6's terminal
  parity is a small `transition.go` fix.

### Testing
- **Unit:** `core` — `PostActionError` `Remaining` on all branches (non-zero check, operational with/without
  recorded checks, expand-error), exit-6 terminal message. `cli` — `TestExitHint` (incl. no-bleed cases),
  `TestExitCode` (`*PostActionError`→5).
- **e2e (testscript, hermetic):** `actionable_errors.txt` (exit-2 transition + `rm --force`, exit-4 task +
  note, exit-6 core + terminal); `post_actions.txt` migrated to assert the recovery block in **both** text and
  `--json`.
- **Live sanity (real binary):**

```
$ mtt in_progress t1                    # require who/why, none supplied
error: mtt: missing required attribution: who, why
  set 'who': add `author: <name>` to .mtt/config.local.yaml, or `export MTT_BY=<name>`, or pass `--who <name>`
  set 'why': pass `--why "<reason>"`                                        [exit 2]

$ mtt show t999
error: task "t999": mtt: task not found
  check the id — 'mtt roadmap' or 'mtt list' show existing task ids ('mtt note list' for notes)   [exit 4]

$ mtt in_progress t1 --who agent --why demo   # post: [echo committing, false, git push origin main]
✗ false (exit 1, 1ms)
move applied — the status change IS saved; do NOT re-run the move.
finish the finalization by hand:
  false
  git push origin main
t1: tbd → in_progress
error: mtt: post-action failed after the move: command "false" exited 1     [exit 5]
```

(The recovery block lists exactly the failed `false` + the untried `git push origin main`; the already-run
`echo committing` is omitted.)
