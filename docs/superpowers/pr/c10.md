## What & why (c10, `[dx]`)

Tag-filter **symmetry**. After c8 (`--exclude-tag` on `list`/`ready`) and the earlier `--tag` (on
`list`/`tree`), the matrix was lopsided: `ready` had only `--exclude-tag`, `tree` had only `--tag`. All three
render through the **same** `core.Match` / `core.ListFilter`, so both directions were trivially wireable.

## Change (chore — design fixed, pure CLI wiring)

- **`mtt ready --tag <t>…`** → `ListFilter.Tags` (include, OR-within), mirroring `list`.
- **`mtt tree --exclude-tag <t>…`** → `ListFilter.ExcludeTags` (negative), mirroring `list`.
- Both go through the existing `toTags` boundary (normalize/validate → usage error on a bad value). **No
  `core` change** — `Match` already honors `Tags` + `ExcludeTags`; `tree`'s `markVisible` already calls
  `Match`. Now `list`/`ready`/`tree` all take **both** `--tag` and `--exclude-tag`.
