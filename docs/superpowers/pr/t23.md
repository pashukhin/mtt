## t23 — `mtt init`: flat, frictionless default template (Part B)

Makes a fresh project usable in one command: `mtt init && mtt add "x"` now works with **no `--parent`**.

### Problem
The `default` template made `task` (`default: true`) require an `epic` parent (`parents: [epic]`), so the
natural first command after `mtt init` failed with `type "task" requires a parent`. That error is *correct* —
it's a property of a hierarchy config, not an engine bug — so the fix is **config-only**.

### Change
- **`default` template → flat**: `task` (root, default) + `chore` (root), minimal generic flow
  (`tbd → in_progress → done/cancelled`, `current: set/clear`), no `make`/`git`/spec assumptions.
- **New `hierarchy` template** = today's `default` (epic/task/subtask) **byte-for-byte**, kept as an opt-in
  (`mtt init --template hierarchy`) that preserves the parent-required placement discipline.
- **`coding` unchanged** (no ergonomics bug; demo polish is t4).
- **Engine/domain untouched** — no change under `internal/core`, `pkg/mtt`, or `add.go`.

### Tests (TDD)
- Ergonomics regression in `init.txt`: fresh flat init → `mtt add 'first task'` → `created t1`, no `--parent`.
- Per-template validity guard extended (`TestInitLoadValidate` now covers `default`/`coding`/`hierarchy`).
- Goldens regenerated (+ new `hierarchy.yaml`).
- Migrated the tests that relied on the old hierarchical default to `--template hierarchy`: 12 testscripts,
  8 cli Go-init sites, and the adapter `task_test.go` helper (`initDefault` → `initHierarchy`). Tests that
  only use `--no-parent`/bare root adds stay on the flat default for coverage.

### Docs
README / DESIGN / CLI_REFERENCE (EN + RU) + adapter `CLAUDE.md` reconciled with the flat default and the new
`hierarchy` template.
