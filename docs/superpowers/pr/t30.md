## What & why (t30, `[release]`)

The version scheme **mirrored the dev session** (`sN в†’ 0.N.0-dev`, `sN.M в†’ 0.N.M-dev`), with a hand-bumped
`0.9.0-dev` literal in `internal/cli/root.go`. That is meaningless to an external adopter, drifts from the
release tag, and retires with the sessions apparatus (t31). As the first release-scoped task (external-ready,
solo, native-YAML), t30 replaces it with **SemVer whose single source of truth is the git tag** вЂ” nothing
about the version is hand-maintained in source.

## Decisions (spec: `docs/superpowers/specs/t30-semver-policy.md`)

- **D1 вЂ” SemVer 2.0.0, stay on 0.x.** Pre-1.0: MINOR = feature *or* breaking; PATCH = fix. A breaking change
  is never a PATCH. The session-mirror rule is retired.
- **D2 вЂ” compat surface** SemVer governs: CLI (commands/flags/exit codes/`MTT_DIR`В·`MTT_BY`В·`MTT_ROLE`),

- **D3 вЂ” git tag = source of truth.** `resolve(ldflags, buildVersionFn)` picks: ldflags-injected `version`

- **D4 вЂ” bump from the changelog** at release cut (Added/Changed/Deprecated/Removed в†’ MINOR; else PATCH).
- **D5 вЂ” living docs reconciled** (RELEASING, CHANGELOG, DESIGN + `.ru`, sessions/README); `NEXT_SESSION.md`
