# Session 006.5 — Attribution + verb sugar (design spec)

Date: 2026-07-06 · Branch: `feat/s006.5-attribution-sugar` · Version bump: `0.6.0-dev → 0.6.5-dev`

Authoritative design for s006.5. Prose in [DESIGN.md](../../../DESIGN.md) stays the source of truth; this
spec is the resolved decision record the plan and implementation follow.

## Goal

Make a flow transition **attributable** and **ergonomic**, and let a project **require** attribution — a
release-complete attribution slice with **no roles/profiles** (those stay PARKED). Three related pieces plus
one enforcement knob:

1. `--why <text>` — a durable free-text reason recorded on the transition.
2. `--who <subject>` — a symmetric alias of `--by` (same `history.by`).
3. `mtt <status> <id>` — verb sugar for a single-edge status move (fallback-routing, not command registration).
4. Required-attribution — a project-global `require: {who, why}` validated before the gate, exit `2`.

## Architecture (unchanged)

`cli → core → port ← adapter`. Everything typed (`mtt.TaskID`/`StatusName`); string conversion only at the
cli/adapter boundary. Reuse `core.Transitioner` (single edge) for the sugar — no new usecase for the move.

## 1. `--why` — the only `pkg/mtt` change

- `mtt.HistoryEntry` gains `Why string`, ordered per `docs/architecture/model.go`:
  `At, By, Role, Why, From, To, Checks`. Additive; no other domain change.
- YAML DTO `ymlHistoryEntry` gains `Why string \`yaml:"why,omitempty"\`` after `role`; mapped both ways in
  `fromDomainHistory`/`toDomainHistory`. `omitempty` ⇒ existing history without a reason serializes unchanged
  (no golden churn — no current golden carries history).
- `mtt show` renders the reason in the `history:` section next to `by`/`role` (e.g. `(by alice, why "…")`).
- `--why` is a **root persistent** flag (so both `mtt status` and the sugar path inherit it).
- **Out of scope:** `taskJSON` does NOT gain a history array this session (parity with s006 — `mtt show` is
  the who/why surface; a JSON history view is a later additive change).

## 2. `--who` — symmetric alias of `--by`

- Root persistent `--who`; targets the same `history.by` as `--by`.
- `--who` and `--by` are **mutually exclusive** (`cmd.MarkFlagsMutuallyExclusive("who","by")`) — they are two
  spellings of one field, so setting both is a usage error (cobra, exit 1), not a silent precedence.
- Precedence unchanged: `--who`/`--by` > `MTT_BY` > `config.local` `author` (the adapter `Settings.Author`).
- `resolveRoleBy` grows into `resolveAttribution(cmd, authorDefault) (role, by, why string)`: `by` is
  whichever of `--who`/`--by` is set (only one can be, by the exclusivity), else `MTT_BY`, else author; `why`
  is the `--why` flag.

## 3. `mtt <status> <id>` — verb sugar via fallback-routing

- Implemented on `root.RunE` (NOT dynamic command registration — avoids the config-before-parse problem,
  namespace pollution, and help clutter). Real commands always win a name clash: cobra dispatches a
  registered subcommand before ever calling `root.RunE`.
- Routing condition (decided): exactly **2 args**, arg0 is **not** a registered subcommand (guaranteed —
  cobra only calls `root.RunE` when it did not dispatch), arg1 resolves to an **existing task**, and arg0 is a
  **status name in that task's type flow** (`typ.StatusKind(arg0)` ok — reuses the pure predicate). Then route
  to the shared transition path with `id = arg1`, `to = arg0`, `noRun = false`.
- Edge validity of the *specific* current→arg0 move is left to `core.Transitioner` (it already validates and
  emits `ErrInvalidTransition` → exit `6` with an "allowed from X: …" hint). So `mtt done t1` when `t1` is
  `tbd` with no `tbd→done` edge routes and reports a helpful exit-6 error (arg0 `done` *is* a flow status).
- Not-sugar fall-through (arg0 not a flow status of arg1's type, or arg1 not a task, or arg-count ≠ 2):
  return `unknown command %q for "mtt"` (exit 1), mirroring cobra's native message. `mtt` with no args →
  `cmd.Help()`.
- The sugar path has **no gate-control flags of its own**: `--no-run` stays **local to `mtt status`** (the
  sugar is the preferred/common path, so it deliberately cannot bypass the gate); `-v`/`--log-file` are root
  persistent (below), so the sugar inherits output control but never gate-bypass.

## 4. Required-attribution — `require: {who, why}`

- Adapter-level setting (execution policy, like `command_timeout` — an external tracker runs no local commands
  and stores its own attribution), **not** `pkg/mtt`. On-disk in the **committed** `.mtt/config.yaml`:
  ```yaml
  require:
    who: true
    why: true
  ```
- DTO: `ymlConfig.Require ymlRequire{Who,Why bool}` (`yaml:"require,omitempty"`, fields `who`/`why`
  `omitempty`). `Settings` gains `Require RequireAttribution{Who, Why bool}`.
- **`config.local` may only tighten**, never relax. Mechanism in `Load`: capture the committed `require`
  **before** overlaying `config.local`, then `effective.Who = committed.Who || local.Who` (and `Why`
  likewise). OR-ing rescues a `who:false` a local overlay might try to set — a requirement present in the
  committed config cannot be removed locally; a local overlay can only add requirements.
- `init` templates do **not** emit `require` (zero-config default = no requirement; the user opts in). No
  golden change.

### Enforcement (in `core.Transitioner`, decided)

- `core.TransitionOptions` gains `Why string`, `RequireWho bool`, `RequireWhy bool`. The CLI passes the
  resolved `By`/`Why` and the `Settings.Require` flags.
- New core sentinel `ErrMissingAttribution` (flow/attribution is core policy, like `ErrBlocked`).
- Check placement: after the edge is validated (`findTransition` ok), **before** the gate block
  (`if !opts.NoRun { runner.Run… }`). So it fails **fast** (no wasted `make check`) and `--no-run` does **not**
  bypass it (attribution is about the record, not the gate).
- **Aggregate** all missing fields into one error so the agent fixes them in one shot: if `RequireWho && By==""`
  collect `who`; if `RequireWhy && Why==""` collect `why`; if any → `%w: missing required attribution: who, why`
  (`ErrMissingAttribution`). Checking the **resolved** `By` means a `config.local` `author` satisfies `who`.
- Error/exit ordering: task-not-found → 1; invalid edge → **6**; missing attribution → **2**; gate blocked → **3**.
- `cli.exitCode` maps `ErrMissingAttribution` → **2** (realized this session, mirroring how s006 realized 3/6).

## CLI flag summary (root persistent, after this session)

`--dir`, `--json`, `--role`, `--by`, **`--who`**, **`--why`**, **`-v`/`--verbose`**, **`--log-file`**. Local to
`mtt status`: `--no-run` only (`-v`/`--log-file` move up to root; `mtt status` reads the persistent ones). `-v`
short has no clash (`--version` carries no short).

## Testing (strict test-first)

Unit:
- `pkg/mtt` / YAML DTO: `HistoryEntry.Why` round-trips (build a task with a history entry carrying `why`,
  assert serialize→parse fidelity; extend the existing `task_dto_test`).
- `core.Transitioner`: `Why` written into the appended entry; required-attribution aggregates both missing
  fields into one `ErrMissingAttribution` and blocks **before** the runner is called (fake runner records it
  was never invoked); `--no-run` does not bypass the attribution check; a satisfied requirement (By/Why present,
  incl. via author default) transitions normally.
- `yaml.Load`: `Settings.Require` parsed; **tighten-only** — committed `who:true` + local `who:false` ⇒
  effective `who:true`; committed absent + local `why:true` ⇒ effective `why:true`.
- `cli`: `exitCode(ErrMissingAttribution) == 2`; `resolveAttribution` precedence; `who`/`by` mutual exclusion
  errors; sugar routing decision (arg0 ∈ flow ⇒ routed; arg0 ∉ flow / task missing ⇒ unknown-command);
  `formatTask` renders `why`.

e2e `testscript` `attribution_sugar.txt`:
- `mtt init` → put `commands: ["true"]` on `tbd→in_progress` via a `-- gated.yaml --` file `cp`'d over
  `.mtt/config.yaml` → `mtt add A` (t1).
- Sugar happy path: `mtt in_progress t1 --who alice --why "start work"` (green gate → moves); `mtt show t1`
  prints `by alice` and the `why`.
- Unknown fall-through: `mtt bogus t1` → non-zero (exit 1), `unknown command`.
- Required-attribution: a second config variant with `require: {who: true, why: true}` → `mtt status t1
  in_progress` (no `--who/--why`) → exit **2**, one error naming **both** `who` and `why`; `--no-run` still
  exits 2 (not bypassed). With `--who/--why` provided → succeeds.

`make check` green.

## Non-goals / PARKED (do not build)

- `advance`/`start`/`done`/`cancel`, the multi-edge walk, modes `--stop`/`--atomic`/`--force`, roles-on-edges,
  config verb→status mapping. On-demand only, when a flow actually branches.
- `taskJSON` history array (deferred additive).

## Deferred (capture in backlog — not this session)

- **Dangerous operations must mandate `--who`/`--why`.** Flow-bypassing / risk actions — `--no-run` on a
  transition whose edge has a **non-empty** command list (skipping a real gate), and later `--force` / atomic
  aborts — should **force** attribution regardless of the project `require` setting (you may skip the gate, but
  you must sign for it). Independent of the global `require` knob; ties into the s007+ structured-commands /
  rollback risk surface. Record in DESIGN.md (required-attribution note) + TASKS.md → Later.
