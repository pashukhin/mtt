# Session 008.6 — Priorities + roadmap (design spec)

Date: 2026-07-07 · Branch: `feat/s008.6-priorities-roadmap` · Version bump: `0.8.0-dev → 0.8.6-dev`

Authoritative design for s008.6. Prose in [DESIGN.md](../../../DESIGN.md) stays the source of truth; this spec
is the resolved decision record the plan and implementation follow. Bundled per the "batch small unambiguous
features into one session" preference: a `Priority` value object (the missing scheduling axis) + `mtt roadmap`
(the agent-friendly execution-order view that consumes it).

## Goal

Two coupled, agent-facing capabilities:

1. **Priorities** — tasks gain a `high|medium|low` priority (the missing ordering axis alongside `depends_on`
   and hierarchy), authored via `--priority`, filterable and sortable.
2. **`mtt roadmap [--json]`** — a **derived execution-order view**: the non-terminal tasks in a
   dependency-respecting, priority-weighted sequence, each annotated with `ready` / `blocked_by`. This is what
   an agent (or human) asks instead of re-deriving "what do I do next, and what's it waiting on" from raw
   tasks. Not a Gantt / scheduler — no dates, no critical path.

## Architecture (resolved)

`cli → core → port ← adapter`. Everything typed (`mtt.TaskID`/`TypeName`/`StatusName`/`Priority`); string
conversion only at the cli/adapter boundary. `roadmap` is a **pure derived read** (like `Ready`/`DepGraph`) —
no new port, no store injected into the view. Priority rides the `Task` field + `TaskStore.Update` (the GAP #1
rule — like `depends_on`/`tags`): **no new port.** Decisions (brainstormed):

- **Q1 — priority is a fixed closed value object** (`pkg/mtt.Priority`), the `StatusKind` idiom: a small
  ordered vocabulary the code reasons about (`high`/`medium`/`low`), **not** a bare string/int and **not**
  config-defined levels (YAGNI — configurable levels are a later revisit-at-second-caller). Empty = unset,
  ordered as `medium` (the neutral default), and **not materialized on disk** (existing task files are
  untouched; back-compat). It maps to a provider's native priority/labels later.
- **Q2 — `roadmap` is a distinct command**, not folded into `ready`/`list`. `ready` is "unblocked *now*"
  (flat); `list --sort priority` has no dependency ordering. `roadmap` is the **look-ahead sequence** —
  including still-blocked tasks with their blockers — which neither gives. Kept **thin**.
- **Q3 — roadmap scope is all non-terminal tasks, flat.** Confirmed-terminal (`done`/`cancelled`) excluded;
  an unresolvable status is included (conservative). `parent` is shown as a field, not a grouping. Leaf-only
  filtering is a later `--filter` option (YAGNI).
- **Ordering rule (roadmap):** a `depends_on` edge is a **hard constraint** (a non-terminal blocker is placed
  before the task it blocks, even if the blocker has *lower* priority); priority is a **soft tie-breaker**
  among the tasks a topological step leaves free. A satisfied (terminal) blocker imposes no constraint.
- **Sub-decisions (approved):** (a) empty priority orders as `medium`; nothing written to disk for the
  default. (b) The **default order of `list`/`ready` is unchanged** (recency) — priority is opt-in via
  `--sort priority`; only `roadmap` is inherently priority-ordered. (c) Session number **008.6** (`0.8.6-dev`),
  landing before the still-pending dogfood-enablers (008.5); point-session numbers are planning labels, not
  strictly monotonic (the repo already interleaves 006.5 / 006.7).

## 1. `pkg/mtt` — the `Priority` value object

### 1a. `priority.go`

```go
// Priority is a task's scheduling priority — a closed, ordered vocabulary (like
// StatusKind). Empty means unset and orders as PriorityMedium (the neutral
// default); it is not written to disk, so existing tasks are unaffected.
type Priority string

const (
	PriorityHigh   Priority = "high"
	PriorityMedium Priority = "medium"
	PriorityLow    Priority = "low"
)

// Valid reports whether p is a known priority or empty (unset).
func (p Priority) Valid() bool {
	switch p {
	case "", PriorityHigh, PriorityMedium, PriorityLow:
		return true
	}
	return false
}

// Rank returns the sort rank (lower = higher priority): high=0, medium=1,
// low=2. Empty (unset) ranks as medium.
func (p Priority) Rank() int {
	switch p {
	case PriorityHigh:
		return 0
	case PriorityLow:
		return 2
	default: // medium and unset
		return 1
	}
}
```

### 1b. `Task.Priority`

- Add `Priority Priority` to `pkg/mtt.Task`, placed **after `Status`** (a task's scheduling attributes
  together): `… Status, Priority, Parent, Tags, …`. Field order == on-disk order (deterministic diff); a task
  with no priority serializes unchanged (`omitempty`), so goldens for existing fixtures are untouched.
- **No `Config.Validate` change** — priority is per-task, not config. Validity is enforced at the **CLI input
  boundary** (`--priority` rejected if `!Valid()`); a corrupt/unknown on-disk value is **tolerated** (ranks as
  medium) — mirroring how `Status` is validated lazily against the flow, not on load.

## 2. `internal/adapter/yaml` — DTO round-trip

- `ymlTask` gains `Priority string \`yaml:"priority,omitempty"\`` in the same position (after `Status`).
- `fromDomain`: `string(t.Priority)`. `toDomain`: `mtt.Priority(yt.Priority)` — a **plain conversion**
  (optional field, empty is legitimate; no smart constructor, no error path — like `Parent`/`Tags`). An
  unknown on-disk value round-trips as-is and ranks as medium.
- Golden update: only if a fixture task carries a priority (add one deliberately to a golden to prove
  round-trip); untouched otherwise.

## 3. `internal/core` — priority surface + roadmap

### 3a. Edit + list/select

- `EditParams` gains `Priority *Priority` (nil = unchanged). `Editor.Edit` applies it when non-nil; the
  "nothing to edit" guard becomes `p.Title == nil && p.Description == nil && p.Priority == nil`. The
  title-or-description invariant is unchanged (priority does not satisfy it).
- `ListFilter` gains `Priorities []mtt.Priority` (a new OR-within/AND-across dimension); `Match` gains a
  priority check (a task with unset priority matches `--priority medium`? **No** — match on the *stored* value;
  unset only matches when no `--priority` filter is given. Rationale: filtering is about the authored label,
  not the ordering default. Documented.).
- `SortKey` gains `SortPriority = "priority"`. `Select` with `SortPriority` orders by `Priority.Rank()` **asc**
  (high first), tie-broken by the existing recency comparator (`Created` desc, then ID). Extract a
  `lessByPriority` that falls back to `lessByRecency` (DRY).

### 3b. `roadmap.go` — the derived view (pure)

```go
// RoadmapEntry is one task in the computed execution order, annotated with
// whether it is actionable now and what still blocks it.
type RoadmapEntry struct {
	Task      mtt.Task
	Ready     bool         // membership in core.Ready (all blockers terminal)
	BlockedBy []mtt.TaskID // depends_on entries not terminal-satisfied (non-terminal or dangling)
}

// Roadmap returns the non-terminal tasks in a dependency-respecting,
// priority-weighted order. Pure — no store, no clock. Composes DepGraph + Ready
// + Priority.
func Roadmap(tasks []mtt.Task, cfg mtt.Config) []RoadmapEntry
```

Algorithm (priority-guided Kahn topological sort):
1. **Nodes** = tasks whose status kind is **not** confirmed-terminal (`kindOf` terminal → excluded; an
   unresolvable status → included, conservative). Reuse the `kindOf` helper shared with `Ready`.
2. **Ready set** = `core.Ready(tasks, cfg)` membership (single source of truth for the `Ready` flag).
3. **Edges** = `depends_on` restricted to blockers that are themselves **nodes** (non-terminal & existing) — a
   terminal (satisfied) or dangling blocker imposes no ordering constraint. Build `indeg[node]` and
   `dependents[blocker] → node`. `BlockedBy[node]` = every `depends_on` entry that is **not** terminal-satisfied
   (non-terminal or dangling).
4. **Emit** by Kahn: while the available set (`indeg == 0`) is non-empty, pop the entry with **min `Rank`**
   (high first), tie-broken by **oldest `Created`**, then **ID** (fully deterministic); append its
   `RoadmapEntry`; decrement `indeg` of its `dependents`, adding newly-zero ones to available.
5. **Cycle-safety:** any node never reaching `indeg 0` is in a `depends_on` cycle (unreachable via the CLI —
   `DependencyEditor` rejects cycles — but defended against a hand-edited store). Append the remaining nodes
   deterministically sorted by `(Rank, Created, ID)`, so the function always terminates and returns every node.

`Ready` (the flag) and `BlockedBy` agree in the common case; they can diverge only for an unresolvable own
status (included in roadmap but `Ready` false) — the conservative signal wins, as in s005.

## 4. `internal/cli` — surface

- **`mtt add`** gains `--priority high|medium|low` (default: unset). Reject an unknown value (`Priority.Valid()`
  is false and non-empty) with a usage error. Wire into `AddParams` (new `Priority` field) → `Adder` stamps it.
- **`mtt edit`** gains `--priority` (→ `EditParams.Priority`).
- **`mtt list`** gains `--priority <p>` (repeatable/`,`-split like `--status`) and accepts `--sort priority`.
- **`mtt show`** prints `priority: <p>` when set (omitted when unset).
- **`mtt roadmap [--json]`** — a new pure-read command (`roadmap.go` in cli): load `cfg` + `TaskStore.List`,
  call `core.Roadmap`, render.
  - **Display rule (consistent everywhere): echo the STORED priority; unset is not faked as medium.** Human:
    a numbered list, one row per entry — `1. t3  [high]  (tbd)  schema design`, with the `[..]` **omitted when
    unset** (as in `show`), and, when blocked, `  ↳ blocked by: t1, t2`. `--json`:
    `[{"id","title","status","priority","ready","blocked_by":[…]}]` where `priority` is the stored value
    (**`""` when unset** — honest; consumers apply their own default). Zero tasks → `[]` (built with
    `make([]T,0)`), never `null`. Note: the *ordering* still treats unset as medium (via `Rank`), but the
    *label* is never fabricated — an unlabeled task simply sorts in the medium band.
- `AddParams`/`EditParams`/`ListFilter` conversions: the CLI parses the `--priority` string → `mtt.Priority`
  at the boundary (validating), never leaking bare strings into `core`.

## 5. Tests (test-first)

**Unit:**
- `pkg/mtt`: `Priority.Valid` (known + empty ok; garbage rejected), `Priority.Rank` (high<medium<low; unset ==
  medium).
- `adapter/yaml`: `ymlTask` priority round-trip (`fromDomain`/`toDomain`), `omitempty` (unset → absent on
  disk), unknown value tolerated; a golden with a priority.
- `core`: `Select` `--sort priority` order (high first, recency tiebreak; unset ranks medium); `Match`
  `--priority` filter (matches stored value; unset excluded by a filter); `Editor` priority-only edit (guard
  allows it) + unchanged-when-nil; `Roadmap` — (i) a blocker placed before its dependent even when the blocker
  is lower priority (hard constraint), (ii) priority tie-break among independent available tasks, (iii) `Ready`
  / `BlockedBy` annotations (terminal blocker satisfied, dangling blocker → not ready + listed), (iv)
  terminal tasks excluded, (v) cycle-safe on a hand-built `depends_on` cycle (terminates, returns all nodes),
  (vi) deterministic order.
- `cli`: `mtt add/edit --priority garbage` → usage error; roadmap human render + `--json` shape + `[]` on
  empty.

**e2e — `roadmap.txt`:** `mtt init` → add several tasks with `--priority` → `mtt dep add` to chain a couple →
`mtt roadmap` shows the expected dependency-then-priority order and `blocked by:` annotations; `mtt roadmap
--json` is valid JSON with `ready`/`blocked_by`; `mtt list --sort priority` orders high-first; `mtt show`
prints the priority. (Deterministic order is the unit test; the e2e asserts presence + key ordering
relationships, per the s003 lesson about wall-clock ties.)

## Non-goals (explicitly out)

- Config-defined priority levels; numeric priority (fixed VO only).
- Changing the default `list`/`ready` order (priority is opt-in via `--sort priority`).
- Roadmap scheduling: dates, durations, critical path, readiness "waves", hierarchy grouping (`--tree`),
  leaf-only filtering — all later / YAGNI.
- A new port for priority (it rides `Task` + `Update`).
