# Session 008.9 — Batch & pipeline (design spec)

Date: 2026-07-09 · Branch: `feat/s008.9-batch` · Version bump: `0.8.7-dev → 0.8.9-dev`

Authoritative design for s008.9. Prose in [DESIGN.md](../../../DESIGN.md) stays the source of truth; this spec
is the resolved decision record the plan and implementation follow. This slice makes mtt **Unix-composable**: a
reusable **task-set selector** shared by every set-operating command, an **`--ids`** output on `list`/`ready`,
and **bulk** `tag add/rm` + `rm` (the no-gate mutations). Gated bulk (`status`/verbs/`edit`/`dep`) is **later**
(gates + partial-success + atomicity are trickier).

## Goal

1. **A reusable task-set selector** with three **mutually exclusive** sources: explicit IDs ∪ `--filter` (the
   `list` predicates over `Select`/`Match`) ∪ **stdin `-`** (IDs one per line).
2. **`--ids` output** on `list`/`ready` (one ID per line) so pipelines compose:
   `mtt list --tag x --ids | mtt tag rm x -`.
3. **Bulk** `tag add/rm` and `rm` over the selector, **best-effort per item** with a report and a **non-zero
   exit on any failure** (git-style), plus a **`--dry-run`** preview.

## Architecture (resolved)

`cli → core → port ← adapter`. The selector is a **thin CLI helper** (`internal/cli/selector.go`), not a core
layer: two of its three branches are pure I/O (reading stdin, parsing args) and the third (`--filter`) is
already `core.Select`/`core.ListFilter`. Putting stdin reading or cobra-flag knowledge into `core` would break
the "core has no I/O" invariant. So:

- **`internal/cli`** owns source resolution (`selectTaskIDs`), the `--ids` writer, the shared bulk-mutation
  runner (`runBulk`), and `--dry-run`.
- **`internal/core`** stays predicate-only. We reuse `Select`/`Match`/`ListFilter`/`Ready` **unchanged**, and
  add exactly **one** core method where cross-item policy is genuinely needed: `Remover.RemoveMany` (the
  subgraph-aware referenced check for bulk `rm`).

*Rejected:* "selector in core" — would force an `io.Reader` and CLI-flag knowledge into `core` (layer
violation). *Rejected:* "a separate `selector` package" — over-engineering; it has no consumers outside `cli`.

Decisions (brainstormed):

- **Q1 — argument layout for `tag add/rm` (id vs tags): CONTEXT-SENSITIVE.** The **selector marker** = a `-`
  among the positionals **or** any `--filter` flag being set. Marker present ⇒ **bulk** (all positionals except
  `-` are **tags**; tasks come from the selector). Marker absent ⇒ **single** (back-compat: `args[0]`=id,
  `args[1:]`=tags; the existing `TagEditor` path, unchanged). This keeps the ergonomic single form
  (`mtt tag add t1 backend`), matches the roadmap examples (`tag rm x -`, `tag add y --status tbd`), and covers
  every case. Consequence: for `tag`, **explicit multi-id is expressed via stdin**, never positionally (the
  positional slot holds tags) — `rm` has no such conflict, so its explicit IDs stay positional.
- **Q2 — an empty selector result (stdin/`--filter` returned 0 tasks) is a NO-OP, exit 0.** Pipeline-friendly:
  an empty upstream (`… | mtt tag rm x -`) does not fail. This is distinct from **no source at all** (neither
  ids, nor `-`, nor a filter) — that is a **usage error**. "Source present but empty" ≠ "no source".
- **Q3 — any per-item failure ⇒ exit 1 (git-style, generic).** Best-effort: process every item, report each
  failure on stderr, and exit `1` if ≥1 failed (`0` otherwise, including `affected 0`). A heterogeneous
  aggregate is **not** mapped onto the 3/4/6 taxonomy — `0` = all ok, `1` = read the report. (A **single**
  explicit `rm <id>` keeps its precise exit 4 on not-found — see §4.)
- **Q4 — bulk `rm` referenced-check ignores IN-SET references (subgraph-aware).** During a bulk `rm`, a
  reference counts **only if the referent is OUTSIDE the deletion set**. So deleting an epic together with all
  its tasks works **without** `--force` (important for the s009 backlog migration). An **external** referent
  still rejects (without `--force`). A single-id `rm` degenerates to today's behaviour (set = `{id}` ⇒ every
  referent is external ⇒ identical reject).

## 1. The selector — `selectTaskIDs` (`internal/cli/selector.go`)

```go
// selectTaskIDs resolves a task set from exactly one of three mutually exclusive
// sources and returns deduplicated ids (first-occurrence order). It NEVER consults
// the current-task pointer (bulk/set ops don't resolve current — like resolveTaskID).
//
//   - explicit ids: positional args (none of them "-"), only when allowExplicitIDs
//   - stdin "-":     a single "-" among the positionals -> read ids one per line
//   - --filter:      any of --status/--type/--kind/--parent/--priority/--tag/--ready
//                    set -> core.Select(core.Ready?(tasks,cfg), ListFilter, cfg)
//
// >1 active source is a usage error ("choose one source: ids | - | --filter flags").
// 0 active sources is a usage error ("no tasks selected: give ids, '-' for stdin, or
// --filter flags"). A source that is present but yields 0 ids returns an empty slice
// with no error (the caller does a no-op, exit 0 — Q2).
func selectTaskIDs(cmd *cobra.Command, positional []string, allowExplicitIDs bool) ([]mtt.TaskID, error)
```

Source detection is by **presence of a marker**, never by result count:

- `stdinActive`  = `positional` contains `"-"`.
- `filterActive` = any filter flag `Changed()` (shared reader — see §5).
- `explicitActive` = `allowExplicitIDs && len(positional-without-"-") > 0`.

Rules:
- **>1 active ⇒ usage error.** `rm t1 --status x` (explicit+filter), `rm t1 -` (explicit+stdin),
  `rm - --status x` (stdin+filter) all error. (For `tag`, `allowExplicitIDs=false`, so positional non-`-`
  tokens are **tags**, not a source — no spurious conflict.)
- **0 active ⇒ usage error** ("no tasks selected …"). For `tag` single-mode this branch is unreachable — the
  command only calls the selector in bulk-mode (a marker is present by definition).
- **dedup**, first-occurrence order preserved (determinism).
- **stdin**: read `cmd.InOrStdin()` line by line; `strings.TrimSpace`; skip empty lines. Existence is **not**
  checked here — the per-item mutation checks it and reports not-found (§5). (The `--filter` branch yields
  existing tasks by construction.)
- Order: explicit = arg order; stdin = line order; filter = `Select` order (Created desc, ID tiebreak).

A tiny helper `stripDash(args) []string` (positionals minus every `"-"`) is shared by `rm` (its explicit ids)
and `tag` (its tags). `hasDash(args) bool` reports `stdinActive`.

## 2. `--ids` output on `list` / `ready`

- `mtt list --ids` / `mtt ready --ids` print **one `<id>` per line** on stdout, honouring the existing filters
  and `--sort`. Shared writer `writeIDs(w io.Writer, ids []mtt.TaskID)`; `list`/`ready` pass `idsOf(selected)`
  (a tiny `[]Task → []TaskID` extractor), and `--dry-run` (§5) reuses the same writer.
- **Mutually exclusive with `--json`** (both set ⇒ usage error) — both are machine formats with different
  purposes (`--ids` feeds stdin, `--json` carries structure).
- Zero-match `--ids` prints nothing (empty output, exit 0) — the natural pipeline terminus.
- Only `list`/`ready` (not `tree` — out of scope).

## 3. Bulk `tag add/rm` (context-sensitive; `internal/cli/tag.go`)

The command classifies mode by the **selector marker** (`hasDash(args) || filterActive(cmd)`):

- **single (marker absent)** — unchanged back-compat path: `idAndTags` requires ≥2 args; `args[0]`=id,
  `toTags(args[1:])`; `core.TagEditor.AddTags/RemoveTags(id, tags)`; prints `tagged/untagged <id>: …` (or the
  `--json` task view). No selector, no bulk report.
- **bulk (marker present)** — `tags := toTags(stripDash(args))`; **≥1 tag required** (empty ⇒ usage error
  `provide at least one tag`); `ids := selectTaskIDs(cmd, args, allowExplicitIDs=false)`; then
  `runBulk(cmd, ids, apply)` where `apply(id) = TagEditor.AddTags/RemoveTags(id, tags)`.

Notes:
- The per-item `TagEditor` calls are the **existing** usecase (idempotent add; guarded rm; `%w` ErrNotFound).
  The `tag rm` text-anchor guard fires **per item** — a guarded task lands in the failure report (exit 1) while
  the rest still apply (best-effort).
- On `tag`, the `--tag` **filter** flag is the tag-**filter** (select tasks carrying a tag), distinct from the
  positional tags being added/removed: `mtt tag rm urgent --tag backend` = "remove `urgent` from tasks carrying
  `backend`". Reads naturally; documented.
- `mtt add`'s existing `--tag` (author a tag at creation) is untouched (single-task create; no selector).

## 4. Bulk `rm` (subgraph-aware; `internal/cli/rm.go` + `internal/core/remove.go`)

`rm` always goes through the selector (no tags ⇒ no positional conflict). Explicit ids are positional.

- **single (exactly one explicit id, no `-`, no filter)** keeps today's surface **verbatim**: prints
  `removed <id>`, exit **4** on not-found, single-store reject-if-referenced. Detected syntactically:
  `len(args)==1 && args[0] != "-" && !filterActive(cmd)`.
- **bulk** (multiple ids / `-` / `--filter`): `ids := selectTaskIDs(cmd, args, allowExplicitIDs=true)` →
  `core.Remover.RemoveMany(ids, force)` → bulk report. `--force` overrides even an **external** reject.

New core method (the one core change):

```go
// RemoveResult is one task's outcome in a bulk delete.
type RemoveResult struct {
    ID  mtt.TaskID
    Err error // nil on success; wraps ErrNotFound (missing) or a referenced/store error
}

// RemoveMany deletes each id best-effort (Q3). It builds Index+DepGraph ONCE from a
// single List snapshot; a referenced-check counts only referents OUTSIDE the id set
// (Q4 — a child/dependent that is itself being deleted does not block), so deleting a
// subtree in one call needs no --force. force overrides an external referent. A
// single id degenerates to Remove's semantics (set={id} => every referent external).
func (r *Remover) RemoveMany(ids []mtt.TaskID, force bool) []RemoveResult
```

- `Remove(id, force)` becomes a **thin wrapper** over `RemoveMany([]TaskID{id}, force)` returning `[0].Err`, so
  the existing single-`rm` behaviour and its error wording are preserved (remove_test.go stays green). The
  wording stays: not-found `task %q: %w`; external referent `task %q is referenced by %s; use --force to delete
  anyway`.
- `externalReferencingIDs(tasks, id, exclude set)` = `referencingIDs` minus `exclude` (the deletion set). The
  existing `referencingIDs` (children via `Index` + dependents via `DepGraph`, deduped) is the base; the set
  filter is applied on top (reused for both single and bulk).
- **CLI current-cleanup**: after a successful delete of any id equal to the current pointer, clear it (existing
  `rm.go` logic, applied per deleted id). `--dry-run` clears nothing.

## 5. Output, `--dry-run`, exit — the shared `runBulk` (`internal/cli/bulk.go`)

```go
// runBulk applies apply(id) to each id best-effort, renders a report, and returns a
// generic aggregate error (exit 1) when any item failed. --dry-run previews without
// mutating. verbPast is the success label ("tagged"/"untagged"/"removed").
func runBulk(cmd *cobra.Command, ids []mtt.TaskID, verbPast string, apply func(mtt.TaskID) error) error
```

- **`--dry-run`** (a flag on the bulk mutations `tag add/rm` and `rm`): resolve the selector, then **preview
  without mutating**, exit 0:
  - human: `writeIDs(stdout, ids)` (one id per line — pipeline-ready) + `(dry run) would affect N task(s)` on
    **stderr**.
  - `--json`: `[{"id":"t1"}, …]` on stdout (no `status`; nothing happened).
- **mutate (not dry)**:
  - human: success summary on stdout `<verbPast> N task(s): id1, id2, …`; per-item failures on stderr
    `<id>: <reason>`.
  - `--json`: `[{"id":"t1","status":"<verbPast>"}, {"id":"t4","status":"error","error":"…"}]` on stdout.
  - **exit**: `0` when all succeeded (incl. `affected 0`); on ≥1 failure, `runBulk` returns
    `fmt.Errorf("%d of %d task(s) failed", nFail, nTotal)` → `Execute` prints it on stderr, exit **1** (generic;
    per-item detail already on stderr / in json).
- **`--dry-run` is checked uniformly, before any mutation, in both the single and bulk paths** of `rm`/`tag`:
  each command resolves its target id(s), and if `--dry-run` is set it previews (`writeIDs` + the stderr
  summary, or the `--json` id array) and returns without mutating. So a single `rm t1 --dry-run` previews `t1`
  and deletes nothing.
- **Non-dry single paths are unaffected**: single `tag add/rm` and single `rm` keep their exact current output
  and (for `rm`) exit-4 on not-found.

Shared **filter-flag** registration/reading for the selector commands (`rm`, `tag add`, `tag rm`):
`addSelectorFilterFlags(cmd)` registers `--status/--type/--kind/--parent/--priority/--tag/--ready`;
`readSelectorFilter(cmd) (core.ListFilter, ready bool)` builds the `ListFilter`; `filterActive(cmd) bool`
reports whether any is `Changed()`. `list`/`ready` keep their own flag wiring (already present) and add only
`--ids`; the shared reader is used by the new selector commands (avoids a churny migration of `list`/`ready`).

## 6. Tests (test-first)

**Unit:**
- `internal/cli` (`selector_test.go`): source detection & resolution — explicit ids; stdin lines
  (trim/skip-empty/dedup); `--filter` via `core.Select`; **conflict** (two sources ⇒ error); **no source** ⇒
  error; **empty stdin / empty filter** ⇒ empty slice, no error; `allowExplicitIDs=false` treats non-`-`
  positionals as non-sources; dedup order preserved.
- `internal/core` (`remove_test.go` additions): `RemoveMany` — subgraph-ignore (epic+children in-set delete
  with no force); external referent still rejects without force; `--force` overrides; best-effort (one
  not-found doesn't stop the rest); a real delete happens; **N==1 ≡ Remove** (same error wording, exit-4 path);
  dedup of duplicate ids. `Remove` wrapper still passes the existing tests.
- `internal/cli` (`bulk_test.go` / command tests): `runBulk` report shapes (human summary + stderr errors;
  `--json` per-item array; `--dry-run` ids + stderr summary, **no mutation**); aggregate error ⇒ exit 1;
  `affected 0` ⇒ exit 0; `list/ready --ids` output; `--ids` + `--json` ⇒ usage error.

**e2e — `batch.txt`** (`testscript`, single-root config so `add` needs no `--parent`):
- seed a few tasks with tags; `mtt list --tag x --ids` prints ids one per line.
- **pipeline**: feed those ids via stdin to `mtt tag rm x -` (testscript `stdin` file) → the tag is gone from
  each (`show` / `list --tag x` empty).
- **filter source**: `mtt tag add y --status tbd` tags the tbd tasks; `list --tag y` finds them.
- **bulk rm subgraph**: `mtt rm <epic> <child>` deletes both with no `--force` (subgraph-ignore); an external
  reference still needs `--force`.
- **`--dry-run`**: `mtt rm --status cancelled --dry-run` lists affected ids, changes nothing (a follow-up
  `list` still shows them).
- **empty set**: an empty stdin / a no-match `--filter` ⇒ exit 0, nothing changed.
- **partial failure**: an stdin list mixing a real id and a missing one (e.g. `t1` + `t99`) piped to
  `mtt tag add z -` ⇒ `t1` tagged, `t99` reported not-found on stderr, exit non-zero.
- **conflict**: `mtt rm t1 --status x` ⇒ non-zero (usage error).

Assertions are **anchored** and assert **presence/relationships**, not absolute positions (the s003 wall-clock
lesson); zero-match `--ids`/`--json` are correct (empty / `[]`).

## Non-goals (explicitly out)

- **Gated bulk** — bulk `status`/verbs/`edit`/`dep` (gates + partial-success + atomicity are trickier) — later.
- **Bulk over the current pointer** — the selector never resolves `current` (bulk/set ops are explicit).
- **AND-mode tag filtering** (`--all-tags`) — still later/YAGNI (noted in s008.7).
- **`--ids` on `tree`** — out of scope (only `list`/`ready`).
- **A union of sources** — the three sources are mutually exclusive (no confusing merge).
- **`--sort` on the bulk commands** — mutation order is immaterial; the selector uses the `Select` default.
- **A transactional/atomic bulk** — bulk mutation is best-effort per item (git-style), not all-or-nothing.
  (Bulk `rm`'s subgraph-ignore is a referenced-check refinement, not transactionality.)
- **A `selector` core package / a core stdin reader** — the selector is a CLI concern (I/O + flags).
