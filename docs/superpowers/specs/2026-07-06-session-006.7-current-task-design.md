# Session 006.7 ÔÇö Current task / working context (design spec)

Date: 2026-07-06 ┬Ě Branch: `feat/s006.7-current-task` ┬Ě Version bump: `0.6.5-dev Ôćĺ 0.6.7-dev`

Authoritative design for s006.7. Prose in [DESIGN.md](../../../DESIGN.md) stays the source of truth; this
spec is the resolved decision record the plan and implementation follow.

## Goal

Give tasks a **git-`HEAD`-for-tasks**: a personal "current task" pointer so an agent stops repeating the id
on every command. `mtt done` finishes *the current task*; `mtt show` shows it; `mtt use t5` selects it. Three
pieces:

1. A **`current`** value in `config.local.yaml` (personal, gitignored) ÔÇö behind a **capability port** so an
   external provider can map it to its native "assign to me" instead.
2. A **rule** on the committed flow ÔÇö a new additive `Transition.Current` field (`set` | `clear`) ÔÇö that
   moves the pointer as the task flows (take-into-work sets it; reaching a terminal clears it).
3. An **omitted id** resolving to `current` for single-task direct verbs only (`status` / `mtt <status>` /
   `show` / `edit`).

## Architecture (resolved)

`cli Ôćĺ core Ôćĺ port ÔćÉ adapter`. Everything typed (`mtt.TaskID`/`StatusName`/`CurrentAction`); string
conversion only at the cli/adapter boundary.

Two decisions were brainstormed and locked:

- **The pointer is a capability port, not a CLI helper.** "Take a task into work" is a capability a real
  provider may own (assignee). So *how* `current` is stored lives behind a port the adapter implements (YAML
  Ôćĺ `config.local`; external Ôćĺ assignee / `ErrUnsupported`). This is the `Capabilities()`/`ErrUnsupported`
  pattern. Justified **now** (not premature like the parked `DependencyStore`) by GAP #1's own logic:
  `depends_on`/`history`/`comments` embed in the `Task` and ride `Update`, so YAML needs no port; `current` is
  **non-embeddable** ÔÇö a personal, gitignored, single-value pointer that is *not* task state ÔÇö so even YAML
  needs a separate store. The port is honestly required by the first adapter.
- **The CLI applies set/clear; `core.Transitioner` is untouched (option ii).** The port already delivers
  provider-agnosticism regardless of who invokes it. Interpreting `Transition.Current` is a mechanical
  dispatch of a declared effect (`set`Ôćĺset, `clear`Ôćĺclear), **not** policy ÔÇö no decision is made ÔÇö so it
  belongs at the composition root, not in core. This keeps `core.Transitioner`'s tests (fake `Runner`)
  untouched, keeps `CurrentStore` a clean SRP object (not bolted onto `TaskStore`), and adds zero core churn.
  The only axis favoring core-applies (option i) is future reuse by the **parked** multi-edge `advance`; that
  is discounted by YAGNI and is a cheap extract-when-the-second-caller-appears migration (the DepGraph/Index
  "revisit at the second graph" discipline). Consequence to accept: when `advance` unparks, extract a shared
  core "apply edge effects" step (documented seam below).

## 1. `pkg/mtt` ÔÇö two additive changes

### 1a. `Transition.Current` (the rule, a value object)

- New value object `CurrentAction string` with `CurrentSet = "set"`, `CurrentClear = "clear"`; the empty
  string means "no effect" (the default ÔÇö most edges do nothing to the pointer).
- Smart constructor `NewCurrentAction(s string) (CurrentAction, error)`: accepts `""`/`set`/`clear`, rejects
  anything else ÔÇö the boundary guard (mirrors `NewStatusName`), invoked by the YAML `toDomain` so a corrupt
  `current: foo` on an edge fails the config load fast.
- `Transition` gains `Current CurrentAction` after `Commands` (field order = model.go order = on-disk order).
  Additive; existing configs without it parse to the empty action.

### 1b. `CurrentStore` capability port (the mechanism)

```go
// CurrentStore is the personal "current task" pointer ÔÇö a capability, since a
// provider may back it natively (assignee) or not at all. YAML stores it in
// config.local; an external adapter maps it to assignment or returns ErrUnsupported.
type CurrentStore interface {
    Current() (TaskID, bool, error)   // ok=false when no current is set
    SetCurrent(id TaskID) error
    ClearCurrent() error
}
```

- Add `CapCurrent Capability = "current"` to the capability vocabulary; the YAML reference adapter reports it
  via `Capabilities()`.
- `Current()` returns `("", false, nil)` when unset (not an error). `SetCurrent`/`ClearCurrent` are the writes.

### 1c. `Type.FindTransition` (pure primitive, a DRY win)

- `func (t Type) FindTransition(from, to StatusName) (Transition, bool)` ÔÇö the single edge lookup, joining the
  existing pure primitives (`TypeByName`/`StatusKind`/`InitialStatus`). `core.findTransition` **delegates** to
  it (dedup, not a new abstraction). The CLI uses it post-transition to read the applied edge's `Current`.

## 2. `internal/adapter/yaml` ÔÇö `CurrentStore` impl + DTO

- **`yaml.NewCurrent(root)`** implements `mtt.CurrentStore`, owning **only** the top-level `current:` key of
  `.mtt/config.local.yaml`:
  - Read/modify/write via **`yaml.Node`** (not a struct decode) so `author`, comments, and any other local
    keys survive a rewrite ÔÇö `config.local` is human-edited (users set `author` there). Atomic write
    (temp+rename), reusing the adapter's existing write discipline.
  - `Current()` parses the `current:` scalar (absent file / absent key Ôçĺ `ok=false`). `SetCurrent` upserts
    `current: <id>` (creating the file if absent). `ClearCurrent` removes the `current:` key (leaving the rest
    intact; an emptied file is left in place, not deleted).
  - Independent of `Load`: `Load` decodes `config.local` non-strictly, so an unknown `current:` key is already
    ignored there ÔÇö the committed `Config`/flow never sees it. Clean SRP: one object owns the pointer.
- **DTO:** `ymlTransition.Current string \`yaml:"current,omitempty"\``; mapped in `toDomain` via
  `mtt.NewCurrentAction` (fail-fast). `omitempty` Ôçĺ edges without a rule serialize unchanged.
- **Templates:** `default` and `coding` gain `current: set` on the initialÔćĺactive edge (`tbd Ôćĺ in_progress`)
  and `current: clear` on every Ôćĺterminal edge (`* Ôćĺ done`, and `coding`'s review edges into a terminal).
  Regenerate the template goldens. This is the accepted variant-A cost: the feature works out of the box, so
  `mtt in_progress t1` sets current and `mtt done t1` clears it with no per-project config.

## 3. `internal/cli` ÔÇö `mtt use`, resolution, apply-on-transition

### 3a. `mtt use` ÔÇö one command, three modes (git-checkout-like)

- **`mtt use <id>`** ÔÇö set current. Validates existence (`TaskStore.Get`; `ErrNotFound` Ôćĺ error, pointer
  **not** written), then `CurrentStore.SetCurrent(id)`. Prints `current: <id>` (or the JSON task with
  `--json`).
- **`mtt use`** (no arg) ÔÇö show the current task as one `taskLine`; if unset, print `no current task` to
  stdout, exit 0.
- **`mtt use --clear`** ÔÇö `CurrentStore.ClearCurrent()`; prints `current cleared` (idempotent ÔÇö clearing an
  unset pointer is fine). `<id>` and `--clear` are mutually exclusive.

### 3b. Apply set/clear after a successful transition

- In `runTransition`, after `tr.Transition(...)` succeeds: read `from/to` from the appended history entry,
  `edge, ok := typ.FindTransition(from, to)`, and dispatch `edge.Current`: `CurrentSet` Ôćĺ
  `NewCurrent(root).SetCurrent(id)`; `CurrentClear` Ôćĺ `ClearCurrent()`; empty Ôćĺ nothing. A `CurrentStore`
  write failure surfaces as an error (the transition already persisted; the pointer is best-effort but a
  failure is reported, not swallowed). Shared by `mtt status` and the sugar (both funnel through
  `runTransition`).

### 3c. Omitted-id resolution (the grammar)

- Shared helper `resolveTaskID(cmd, explicit string) (mtt.TaskID, error)`: if `explicit != ""` Ôćĺ parse it;
  else `CurrentStore.Current()` ÔÇö if set, return it (existence validated by the caller's own `Get`, giving the
  stale-current message); if unset Ôćĺ error `no task id given and no current task set`. Order is
  **explicit id > current** (the s008.9 `filter/stdin` tier slots between them later ÔÇö its "no source"
  single-verb case becomes `current`).
- **`mtt status`**: `Args` accepts **1 or 2** ÔÇö `mtt status <id> <new>` (explicit) or `mtt status <new>`
  (id=current). Resolve id, then `runTransition`.
- **Sugar** (`root.RunE`): resolve the id first, then classify arg0 against **that** task's flow:
  - **2 args** `mtt <status> <id>` ÔÇö id=arg1 (unchanged from s006.5).
  - **1 arg** `mtt <status>` ÔÇö id=current. If current set and arg0 is a status in its type flow Ôćĺ route
    (this is `mtt done` on the current task). If current **unset**: when arg0 matches a status name in *any*
    configured flow Ôćĺ error `no current task set; run \`mtt use <id>\` or give an id` (a plausible-verb hint);
    otherwise Ôćĺ `unknown command` (unchanged fall-through). `mtt` with no args Ôćĺ help.
- **`mtt show`** / **`mtt edit`**: `<id>` becomes optional (`Args` 0-or-1); omitted Ôćĺ `resolveTaskID` Ôćĺ
  current. All other reads/writes are unchanged.
- **Stale current:** whenever a resolved-from-current id fails to `Get` (`ErrNotFound`), the verb reports
  `current task <id> no longer exists; run \`mtt use <id>\` or \`mtt use --clear\`` rather than a bare
  not-found ÔÇö the pointer is validated at the point of use, never silently stale.
- **Never resolved to current:** `list`, `tree`, `dep`, `ready`, and any future filter/stdin/bulk path. These
  operate over sets or take an explicit target; an omitted id is not a single-task-verb signal there.

## Error / exit taxonomy (unchanged codes)

No new exit codes. `mtt use` on a missing task, an unresolvable omitted id, and a stale current are ordinary
errors (exit 1). Transition errors keep s006/s006.5 codes (3 blocked / 6 invalid / 2 attribution).

## Testing (strict test-first)

Unit:
- `pkg/mtt`: `NewCurrentAction` accepts `""`/`set`/`clear`, rejects other; `Type.FindTransition` hit/miss;
  `core.findTransition` still behaves (delegation).
- YAML DTO: `ymlTransition.current` round-trips a `set`/`clear` edge; a bad value fails `toDomain`.
- `yaml.NewCurrent`: setÔćĺread round-trip; **preserves `author`** and unknown keys/comments across a rewrite
  (write a `config.local` with `author:` + a comment, `SetCurrent`, assert `author` intact and `current` set);
  `ClearCurrent` removes only `current:`; read of an absent file/key Ôçĺ `ok=false`.
- `cli`: `resolveTaskID` (explicit wins; current fallback; unset Ôćĺ error); sugar 1-arg routing on current;
  1-arg no-current plausible-status hint vs unknown-command; `mtt use` mode dispatch (set validates, show,
  clear); `use`/`--clear` mutual exclusion; apply-on-transition picks the right `CurrentAction` from the edge.

e2e `testscript` `current_task.txt`:
- `mtt init` Ôćĺ a `-- local.yaml --` config with `current: set` on `tbdÔćĺin_progress` and `current: clear` on
  `in_progressÔćĺdone` (edges keep empty `commands` so the gate passes trivially and no `require` is set),
  `cp`'d over `.mtt/config.yaml` (the s006 config-swap pattern) Ôćĺ `mtt add A` (t1), `mtt add B` (t2).
- **Explicit set + show:** `mtt use t1` Ôćĺ `mtt use` shows the t1 row Ôćĺ `mtt show` (no id) shows t1.
- **Edge-set:** `mtt use --clear` Ôćĺ `mtt use` says `no current task`; `mtt in_progress t1 --who a --why w` Ôćĺ
  the edge's `current: set` sets current=t1 Ôćĺ `mtt use` shows t1 (proves the transition moved the pointer).
- **Omitted-id verb + edge-clear:** `mtt done` (no id Ôćĺ current=t1) Ôćĺ t1 moves to `done`, the edge's
  `current: clear` clears the pointer Ôćĺ `mtt use` Ôćĺ `no current task`.
- **No-current error:** `mtt done` (still unset) Ôćĺ non-zero, `no current task`. `mtt bogus` Ôćĺ `unknown
  command` (unchanged fall-through).
- **Explicit independence:** `mtt use t2` Ôćĺ `mtt show` (no id) shows t2.
- **Stale current** cannot be produced via the CLI (no `rm` until s008.5), so it is proven by a **unit** test
  (`Current()` returns an id whose `Get` is `ErrNotFound` Ôçĺ the resolve-site error), a documented gap
  mirroring s005's approach.

## Backlog / think (capture, do not build)

- **Separate `mtt current` (show) + `mtt use <id>` (set) + clear** ÔÇö semantically cleaner ergonomics than one
  overloaded command; revisit as a usage-ergonomics pass.
- **Show the status/transition `description` on a successful move** ÔÇö an in-flow reminder for the agent
  ("what this transition is for"). A read-side nicety.
- **`current` for *all* single-task operations, including reads** (`tree <id>`, `dep list`, `show` already) ÔÇö
  today only `status`/`mtt <status>`/`show`/`edit` resolve current; generalize deliberately.
- **`dep` on current** ÔÇö mild but real value (`mtt dep add <dep>` on current); folded into the above.
- **Per-agent current** ÔÇö a shared checkout with multiple agents has one `config.local` = one `current` Ôćĺ
  collision; ties to the parked subagent-identity question. Fine for solo / one-agent-per-checkout.
- **Multi-assignee providers** ÔÇö what "my current" means when a provider allows several assignees (team work);
  YAML is single-valued. Decide with the first external adapter.
- **`advance` reuse seam** ÔÇö when the parked multi-edge `advance` unparks, extract the CLI apply-set/clear step
  into a shared core "apply edge effects" helper so `Transitioner` and `Advancer` share it (avoids the DRY
  split option ii accepts now).
