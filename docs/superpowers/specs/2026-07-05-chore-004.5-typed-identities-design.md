# Chore 004.5 — Typed-Identity Retrofit — Design Spec

Date: 2026-07-05 · Branch: `chore/typed-identities` · Roadmap: [../../../sessions/README.md](../../../sessions/README.md) (row `004.5`)

Language note: agent-facing docs are English only (see AGENTS.md). This spec is the authoritative statement
of the 004.5 chore. It realizes Gap #2 of the domain-model snapshot
([../../../docs/architecture/model.go](../../../docs/architecture/model.go), section "Identity types") and
prepares a typed contract surface for session 005 (dependencies).

## 1. Goal

Introduce the named string identities from the snapshot — `TaskID`, `TypeName`, `StatusName` — across the
shipped `pkg/mtt` / `internal/core` / `internal/adapter/yaml` / `internal/cli`, replacing the bare `string`
those layers use today. A `TypeName` can no longer be passed where a `TaskID` is wanted (a compile-time
guard), and s005's dependency code is written against the typed surface from the start.

This is a **pure internal refactor: no behavior change** (one deliberate exception, §4). e2e scripts are not
touched; the on-disk YAML stays plain strings; `.mtt/` round-trips byte-for-byte; `make check` is green at
every step. "Test-first" here means leaning on the existing tests as a safety-net — we retype signatures and
update tests to the new types, adding no new logic (except the identity constructors).

## 2. Scope

- **In:**
  - `pkg/mtt`: new `identity.go` (`TaskID`/`TypeName`/`StatusName` + trivial smart constructors); retype
    `Task`, `Type`, `Status`, `Transition`, `HistoryEntry`, the `TaskStore` port, and the type-query /
    `Validate` methods.
  - `internal/core`: retype `AddParams`, `Editor.Edit`, `ListFilter`, `Index`, and the `Match`/`Select`/
    `anyOrEmpty`/`lessByRecency` helpers.
  - `internal/adapter/yaml`: keep the on-disk DTO (`ymlTask` etc.) as `string`; map `string ↔ typed` in
    `fromDomainTask` / `toDomain`; the identity constructors live at `toDomain` (deserialization boundary).
  - `internal/cli`: convert cobra `string` args/flags to identities at the boundary before calling `core`.
  - Unit tests updated to the typed signatures; new tests for the constructors and the `toDomain` guard.
  - Docs: `pkg/mtt` / `core` / `yaml` CLAUDE.md; `model.go` Gap #2 (retrofit done); status in TASKS.md /
    sessions/README.md.
- **Out (unchanged / deferred):**
  - `Ref.ID` stays `string` — the target is heterogeneous (a `TaskID`, a `NoteSlug`, or a URL selected by
    `Kind`); no single identity type fits (snapshot §3).
  - `Comment.ID` (`int`), `Comment.Author`, `HistoryEntry.By`, `HistoryEntry.Role` stay their current types —
    they are not the Identity-types set (`By`/`Role` are the subject/role seams, out of this snapshot section).
  - `NoteSlug` is **not** introduced — it is T3/KB with no caller yet (would be a golangci `unused` symbol).
  - No new port method, no dependency logic, no CLI surface change — those are s005.

## 3. Identities (`pkg/mtt/identity.go`)

```go
// TaskID is an adapter-minted task identity — opaque. core never parses it.
type TaskID string

// TypeName is a configured type's name (e.g. epic/task/subtask).
type TypeName string

// StatusName is a status name. Full identity is (TypeName, StatusName), per-flow.
type StatusName string

func NewTaskID(s string) (TaskID, error)
func NewTypeName(s string) (TypeName, error)
func NewStatusName(s string) (StatusName, error)
```

- The three types are plain `string` underneath, so they **marshal as strings** (yaml/json unchanged) and
  stay OPAQUE — nothing here parses an id's structure (core never interprets an id's shape; that is
  adapter-specific).
- The constructors are **trivial smart constructors**: they **reject empty** and otherwise return the value
  unchanged. They deliberately do **not** transform (no trimming): transforming would break the byte-for-byte
  round-trip. "normalize" is documented as the reserved future extension point (e.g. user-entered status
  names in s006). They do **not** parse structure and do **not** do an exists-check (that is contextual —
  `Config.Validate` / usecases).
- Error text: `mtt: empty task id` / `mtt: empty type name` / `mtt: empty status name` (a plain
  `errors.New` each; not a wrapped sentinel — callers surface the message, they do not branch on it).

## 4. The one deliberate behavior change

The constructors are the guard at the **deserialization boundary** (`ymlTask.toDomain`). A task file has no
`omitempty` on `id`/`type`/`status` — a valid task always carries all three. Wiring
`NewTaskID`/`NewTypeName`/`NewStatusName` there means a **corrupt on-disk task with an empty `id`, `type`, or
`status` now fails to load** with a clear error, instead of silently producing a broken in-memory task. This
is the single observable change, and it is a desirable one (fail-fast on malformed storage). It is covered by
a new unit test (§9). `parent` and `depends_on` are `omitempty` (legitimately empty), so they use **plain
conversion**, not the constructors.

The constructors are **not** wired onto optional boundaries where empty is meaningful — an empty `--type`
means "default type", an empty `--parent` means "no parent", empty `--status`/`--type` filters mean "match
all". Rejecting empty there would break existing semantics. Those boundaries use plain conversion.

## 5. Type map (`string → named`)

### 5.1 `pkg/mtt` domain (`config.go`, `task.go`, `store.go`, `type_query.go`, `validate.go`)

| Field / signature | Was | Now |
|---|---|---|
| `Task.ID`, `Task.Parent` | `string` | `TaskID` |
| `Task.Type` | `string` | `TypeName` |
| `Task.Status` | `string` | `StatusName` |
| `Task.DependsOn` | `[]string` | `[]TaskID` |
| `Type.Name` | `string` | `TypeName` |
| `Type.Parents` | `[]string` | `[]TypeName` |
| `Status.Name` | `string` | `StatusName` |
| `Transition.From`, `Transition.To` | `string` | `StatusName` |
| `HistoryEntry.From`, `HistoryEntry.To` | `string` | `StatusName` |
| `TaskStore.Get(id string)` | `string` | `TaskID` |
| `Config.TypeByName(name string)` | `string` | `TypeName` |
| `Type.AcceptsParent(parentType string)` | `string` | `TypeName` |
| `Type.StatusKind(status string)` | `string` | `StatusName` |
| `Validate` / `validateFlow` internal maps | `map[string]…` | `map[TypeName]…` / `map[StatusName]…` |

`Ref.ID` stays `string`. `Create`/`Update`/`List` typing follows from `Task`.

### 5.2 `internal/core`

| Symbol | Was | Now |
|---|---|---|
| `AddParams.TypeName` | `string` | `TypeName` |
| `AddParams.Parent` | `string` | `TaskID` |
| `Adder.Add` internals (`parent`) | `string` | `TaskID` |
| `Editor.Edit(id string, …)` | `string` | `TaskID` |
| `ListFilter.Statuses` | `[]string` | `[]StatusName` |
| `ListFilter.Types` | `[]string` | `[]TypeName` |
| `ListFilter.Parent` | `string` | `TaskID` |
| `Index` maps + `Get`/`Children`/`Ancestors` keys | `string` | `TaskID` |
| `anyOrEmpty(values []string, v string)` | `[]string` | generic `anyOrEmpty[T comparable](values []T, v T)` |
| `lessByRecency` ID tiebreak | `strings.Compare(a.ID, b.ID)` | `strings.Compare(string(a.ID), string(b.ID))` |

`ListFilter.Kinds` is already `[]mtt.StatusKind` (unchanged). `SortKey` is already a named type (unchanged).
`EditParams` (title/description pointers) is unchanged.

### 5.3 `internal/adapter/yaml`

- `ymlTask` and the other DTOs keep **`string` fields + yaml tags** (on-disk format is unchanged).
- `fromDomainTask`: `string(t.ID)`, `string(t.Type)`, `string(t.Status)`, `string(t.Parent)`, and a
  `[]TaskID → []string` map for `DependsOn`.
- `toDomain`: `NewTaskID(yt.ID)` / `NewTypeName(yt.Type)` / `NewStatusName(yt.Status)` (the §4 guard);
  `mtt.TaskID(yt.Parent)` and a `[]string → []TaskID` map for `DependsOn` (plain, optional fields).
- `Store.Get(id TaskID)`: path built from `string(id)`. `Create`: the `prefixes` lookup key becomes
  `string(t.Type)` (or the prefixes map is keyed by `mtt.TypeName` — decide during impl, whichever is
  cleaner; the config-DTO layer already maps names).

### 5.4 `internal/cli`

- Required positional `<id>` (`show`, `edit`, `tree`) → `mtt.NewTaskID(args[0])` (surfaces the empty guard;
  cobra guarantees arity, not non-emptiness).
- `add --parent <id>` when set → `mtt.NewTaskID(flag)`; when unset it stays the empty-`TaskID` "no parent".
- `--type` (add/list), `--status`/`--type` filters, `--parent` filter → **plain conversion**
  (`mtt.TypeName(...)`, `mtt.StatusName(...)`, `mtt.TaskID(...)`) — empty stays meaningful.
- Output paths (`format.go`, `json.go`, `taskJSON`): named string types print and JSON-marshal exactly as
  strings, so renders are unchanged. Where a raw `string` is needed for formatting, convert explicitly
  (`string(t.ID)`).

## 6. Architecture & layer boundaries

The hexagon `cli → core → port ← adapter` is unchanged; dependencies still point inward. The identities live
in the public `pkg/mtt` contract (they are domain vocabulary, like `StatusKind`). `core` imports only
`pkg/mtt`. The adapter owns the `string ↔ typed` mapping in its DTO layer — the domain still never sees YAML.
No layer boundary moves; only the types crossing them get names.

## 7. Order of work (incremental; `make check` green at every step)

1. `pkg/mtt/identity.go`: the three types + constructors + their unit tests (red → green).
2. `pkg/mtt` domain: retype `config.go` / `task.go` / `store.go` / `type_query.go` / `validate.go`; update
   `pkg/mtt` tests to typed literals.
3. `internal/core`: retype `add.go` / `edit.go` / `list.go` / `index.go` (+ generic `anyOrEmpty`); update
   core tests.
4. `internal/adapter/yaml`: DTO mapping + constructors in `toDomain`; update adapter tests + the
   corrupt-load guard test.
5. `internal/cli`: boundary conversions across the commands.
6. Docs (§8).

Each step compiles and passes `make check` before the next (the type change ripples, so a step may touch
several files to restore green — that is expected for a retrofit).

## 9. Testing (safety-net first)

- **Existing tests are the safety-net.** They are updated to the typed signatures (e.g. a test literal
  `"e1"` for a task id becomes `mtt.TaskID("e1")` where the type is now `TaskID`) — mechanical, no new
  assertions on behavior.
- **New unit tests:**
  - `pkg/mtt`: each constructor — non-empty input returns the value; empty input returns the empty-id/name
    error. (Justifies the constructors as `used` symbols per golangci.)
  - `internal/adapter/yaml`: `toDomain` (or a `Get`/`List` path) rejects a task file with an empty
    `id`/`type`/`status` with the constructor's error (the §4 behavior change).
- **e2e:** unchanged. The existing `testscript` scenarios (init/add/list/show/tree/edit) must still pass
  untouched — they drive the real binary and assert user-visible output, which does not change.
- **Round-trip:** golden serialization tests stay green (DTO is still `string`); a byte-for-byte `.mtt/`
  round-trip is the proof of "no behavior change" on the storage side.

## 10. Carry-over lessons applied

- `golangci unused`: every new symbol (`identity.go` types + constructors) is exercised by a test or a
  call-site in the same change — declared with its first use.
- Keep each affected `CLAUDE.md` current (`pkg/mtt` / `core` / `yaml`).
- CLI output stays `fmt.Fprint(cmd.OutOrStdout(), …)`; named string types don't change formatting.
- Bilingual docs untouched here (English source) — no DESIGN/CLI_REFERENCE change.
- Reuse the existing DTO seam for the `string ↔ typed` map — one place, no duplication.
