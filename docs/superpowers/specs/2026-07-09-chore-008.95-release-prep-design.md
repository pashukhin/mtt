# Chore 008.95 ‚Äî Release prep (design spec)

Date: 2026-07-09 ¬∑ Branch: `feat/s008.95-release-prep` ¬∑ Version bump: **none** (`0.8.9-dev` unchanged)

Authoritative design for the release-prep chore. This is **sub-project 1** of the "first release" milestone
(sub-project 2 is s009 dogfood, spec'd separately). It is a **chore**: build/release infrastructure + docs
hygiene, **no runtime behavior or feature change** ‚Äî so the code version string stays `0.8.9-dev`. The actual
`v0.9.0` tag + published binaries are cut **after** the dogfood session lands, by the user.

**Why now (before dogfood):** the milestone is framed as mtt's *first small release*. For the dogfood config +
self-host approach to spread to **neighboring repos** (not in context), mtt must install via a **downloaded
binary** (not only `go install`, which needs a Go toolchain), and the docs (install + quickstart) must be
presentable. This chore lays that groundwork; dogfood then adds the self-host config + a "self-host in your
project" guide (which references the committed dogfood config as the example).

## Goal

1. **Cross-platform binaries** from a hand-rolled `make release` target (version-stamped, `dist/` + a
   `SHA256SUMS`), and a **tag-triggered** `.github/workflows/release.yml` that builds them and attaches them to
   a GitHub Release ‚Äî so users download a binary instead of running `go install`.
2. **Docs hygiene**: a real **Install** section (prebuilt binary + `go install` fallback + build-from-source), a
   **Quickstart**, and a refresh of anything stale ‚Äî in `README.md` **and** its `README.ru.md` mirror. A light
   `CHANGELOG.md` seeded for the upcoming release.

## Decisions (brainstormed)

### Structure ‚Äî split, chore `008.95`, version-neutral.
The "first release" scope (release-eng + docs + dogfood) is **split** into a compact release-prep chore
(this) **then** s009 dogfood, honoring the project's **compact-sessions** value (three concerns in one session
would bloat it). Numbered `008.95` (sorts between 008.9 and 009). **Version stays `0.8.9-dev`** ‚Äî a chore that
changes no runtime behavior does not bump the version; the machinery it adds is precisely what *stamps*
`v0.9.0` at the dogfood milestone. *Rejected:* combining into one large s009 (bloat); renumbering the roadmap
(ripples 009‚Üíreferences‚Üí‚Ä¶).

### Release tooling ‚Äî hand-rolled `make release` + `gh` in a workflow.
A `make release` GOOS/GOARCH matrix + a `release.yml` that runs it and uploads via the **`gh` CLI** (built into
GitHub runners) ‚Äî **no new build-time tool or third-party action** (matches the project's "no heavy deps
without reason" + zero-footprint ethos). *Rejected:* **goreleaser** (idiomatic and lower long-term maintenance,
but adds a tool + `.goreleaser.yaml`; the ~30-line hand-rolled matrix keeps full control for a small first
release ‚Äî revisit goreleaser if the matrix grows); *rejected:* local-only `make release` with manual upload
(no CI automation ‚Äî the user asked for a repeatable binary install path).

### Artifacts ‚Äî raw version-stamped binaries + `SHA256SUMS`.
`dist/mtt_<version>_<os>_<arch>[.exe]` raw binaries plus one `SHA256SUMS` file ‚Äî simplest to `curl` + `chmod +x`
and to document for a first release. *Rejected:* per-platform archives (`tar.gz`/`zip` with LICENSE/README) ‚Äî
more script + an extra extract step for the user; can arrive with goreleaser later. **Matrix (5):**
`linux/amd64`, `linux/arm64`, `darwin/amd64`, `darwin/arm64`, `windows/amd64`. (mtt is pure Go ‚Äî **no cgo** ‚Äî so
cross-compilation is a plain `GOOS`/`GOARCH` `go build`, runnable in CI and locally.)

## Architecture (resolved)

**No change** to `pkg/`/`internal/{cli,core,adapter}` ‚Äî this chore touches only build/CI/docs. The version
string (`internal/cli.version = "0.8.9-dev"`) is **unchanged**. Files added/changed:

- **`Makefile`** ‚Äî a new `release` target (+ helper vars). Reuses the existing `LDFLAGS` version-stamping
  (`-X ‚Ä¶/internal/cli.version=$(VERSION)`). `VERSION` is **required** for `release` (fail fast if empty ‚Äî a
  release must be stamped). Not part of `make check` (like `make smoke`: cross-compiles + writes `dist/`,
  non-hermetic). Emits the 5-target matrix into `dist/` then a `SHA256SUMS` (via `sha256sum`, Linux/CI).
- **`.github/workflows/release.yml`** ‚Äî `on: push: tags: ['v*']`; `permissions: contents: write`; ubuntu-latest;
  `actions/checkout@v7` + `actions/setup-go@v6` (`go-version-file: go.mod`) mirroring `ci.yml`;
  `make release VERSION=${GITHUB_REF_NAME}`; then `gh release create "$GITHUB_REF_NAME" dist/* --generate-notes`
  (or upload to an existing release) with `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}`. No third-party action.
- **`README.md` / `README.ru.md`** ‚Äî an **Install** section (prebuilt binary per-OS from Releases + integrity
  via `SHA256SUMS`; `go install ‚Ä¶@latest` as the Go-toolchain fallback; build-from-source `make build`), a
  **Quickstart** (a correct, runnable `init ‚Üí add ‚Üí status/gate ‚Üí list/tree/roadmap` flow ‚Äî verified by running
  it), and a staleness pass (reflect shipped features: priorities/roadmap, tags/#hashtags, current task,
  batch/selector). English is the source of truth; the `.ru` mirror stays in sync (bilingual rule).
- **`CHANGELOG.md`** (new, light, English) ‚Äî Keep-a-Changelog format; shipped features listed under
  **`[Unreleased]`**. The dogfood session moves them under `[0.9.0]` at tag time. *Trimmable* if the user
  prefers to defer a changelog.

## Acceptance (must pass)

- **`make release VERSION=v0.9.0-rc` runs locally** and produces `dist/` with **5** version-stamped binaries
  (`mtt_v0.9.0-rc_{linux_amd64,linux_arm64,darwin_amd64,darwin_arm64,windows_amd64.exe}`) + a `SHA256SUMS`; the
  **host-arch** binary executes and `mtt version` prints `v0.9.0-rc` (a `make smoke`-style check on the one
  binary that can run here). This is the session's runnable verification (cross-compile is real, pure-Go).
- **`release.yml` is valid** and mirrors `make release` (its live run ‚Äî a real tag push creating a GitHub
  Release ‚Äî is **user-triggered/deferred**, out of scope here; the workflow is reviewed, not executed).
- **README.md + README.ru.md** carry the Install + Quickstart sections and are **in sync**; the Quickstart
  commands were actually run and match their described output.
- `make check` green (unchanged version, no behavior change).

## Out of scope (explicitly deferred)

- Actually **publishing** `v0.9.0` (tag push + GitHub Release) ‚Äî user-triggered, and after dogfood merges.
- **goreleaser**, per-platform **archives**, package managers (Homebrew/Scoop/apt), a `curl | sh` **install.sh**
  (nice-to-have; revisit after the raw-binary path proves out).
- The **"self-host mtt in your project" guide** and any **version bump to `0.9.0-dev`** ‚Äî both belong to s009
  dogfood (the guide references the dogfood config; the bump + tag mark that milestone).
- Any `pkg/core/adapter/cli` logic change.

## Docs sync (same session)

`README.md`/`README.ru.md` (Install + Quickstart + staleness); `CHANGELOG.md` (new); `sessions/README.md` (add
an `008.95` row before `009`); `NEXT_SESSION.md` (a short "release-prep shipped" note + carry-over); the

becomes a documented rule (light "Releasing" note in DESIGN or a `RELEASING.md` ‚Äî optional, decided during
implementation). Version string **unchanged**.
