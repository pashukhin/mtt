# Session 008.5 — dogfood enablers (chore) — design spec

Status: brainstormed (this doc) → writing-plans next. Branch: `feat/s008.5-dogfood-enablers`.
Version bump: `0.8.0-dev` → `0.8.5-dev`.

## Purpose

Three small, unambiguous enablers, batched into **one** session/PR (batching is a process preference —
don't spawn sessions per feature), that unblock moving mtt's own backlog onto mtt (dogfood, s009):

1. **`mtt rm <id>`** — hard-delete a task (distinct from `cancel`, which is a *terminal status*, not removal).
2. **`--depends-on` on `mtt add`** — set `depends_on` at creation time.
3. **Packaging** — `make install` version stamping via `-ldflags` + a `make smoke` install/run test.

Non-negotiables: test-first (red→green→refactor), `make check` green before commit, SOLID/DRY/KISS/DDD,
clean/hexagonal architecture (`cli → core → port ← adapter`), storage only through a port.

## Brainstorm decisions (locked)

All four `rm` open questions from the kickoff were resolved with the user:

1. **Port for delete → base `TaskStore.Delete`.** Delete is the *D* in CRUD — a **store operation**, not an
   embedded `Task` field, so the GAP #1 rule ("a field rides `Update`, no port") does **not** cover it: you
   cannot embed "remove the aggregate" in the aggregate. It joins the **mandatory-minimum** port (every
   adapter implements it); the YAML reference does `os.Remove`. An external adapter that cannot hard-delete
   (archive-only trackers) returns `ErrUnsupported` from the method. Chosen over an optional `DeletableStore`
   capability: `rm` is a base operation, and adding a capability vocabulary now (before `mtt caps`, e4_t6)
   would be premature surface. This is the first extension of the base `TaskStore` since it was defined.
   - *Philosophical note:* the port was **deliberately** `Create/Get/List/Update` without `Delete` — mtt's
     model prefers driving a task to a terminal (`cancel`/`done`) over deleting it. `rm` is the escape hatch
     for **backlog hygiene** (dogfood: purge mistaken/obsolete task files), not part of the normal lifecycle.

2. **Dangling references on delete → reject-if-referenced, `--force` overrides.** By default `rm` **refuses**
   to delete a task that other tasks reference — someone `depends_on` it, or it has children (`parent` points
   at it) — and lists the referencing ids (safe; in the spirit of `dep add` rejecting a cycle). `--force`
   deletes anyway, leaving the references **dangling**, which the system already tolerates by design: `Ready`
   is conservative (a dangling blocker leaves the dependent **not** ready), and `Index` surfaces orphans
   (dangling `parent`) as **roots**. Rejected: silent leave-dangling (surprising — a blocker vanishing
   silently flips a dependent to not-ready) and cascade delete (dangerous — `rm` of an epic would wipe the
   subtree; overkill for hygiene).

3. **Safety for an unreferenced `rm` → delete immediately, no confirmation.** mtt is **agent-facing**;
   interactive `y/N` prompts hang agents (no stdin), so no `rm -i`. The de-facto audit of a deletion is the
   **git commit** that drops the `.mtt/tasks/<id>.yaml` file. `--who`/`--why` are **not** mandated: a deleted
   task has **no `history`** (the file is gone) — there is nowhere to record attribution (this is why the
   TASKS.md think-item "dangerous ops must mandate --who/--why", which targets *flow-bypassing* ops that keep
   a `history`, does **not** apply to `rm`). `--force` is therefore single-purpose: override the
   referenced-check from decision (2).

4. **Exit code for "not found" → `4`.** CLI_REFERENCE already reserves `4 = Not found`; `rm <missing-id>` is
   its **first** consumer. (`ErrNotFound` → exit 4.)

Two more decisions taken during design (confirm at spec review):
- **`rm` output → plain `removed <id>`**, no `--json` branch (symmetric with `add`'s `created <id>`, which
  also ignores `--json`). The object is gone — nothing to echo; the agent branches on the exit code.
- **`rm` requires an explicit `<id>` — it does NOT resolve the current-task pointer.** For a destructive op,
  an explicit target is safer than the s006.7 omitted-id ergonomics, and it stays clear of the future
  bulk-selector (s008.9). `Args: ExactArgs(1)`.

## Scope

### In

**Feature 1 — `mtt rm <id>` (hard-delete):**
- **`pkg/mtt`** (`store.go`): add `Delete(id TaskID) error` to the `TaskStore` interface (mandatory-minimum;
  `ErrNotFound` when the id does not resolve). Update `pkg/mtt/CLAUDE.md`.
- **`internal/adapter/yaml`** (`task.go`): `Store.Delete` = `os.Remove(.mtt/tasks/<id>.yaml)`; `os.ErrNotExist`
  → `mtt.ErrNotFound`, other errors wrapped `%w`. `os.Remove` is atomic (mirrors the atomic temp+rename on
  write). The existing `var _ mtt.TaskStore = (*Store)(nil)` guard now also covers `Delete`. Update
  `yaml/CLAUDE.md`.
- **`internal/core`** (`remove.go`): `Remover` usecase — `Remove(id TaskID, force bool) error`:
  1. `store.Get(id)` → not found → return an error wrapping `mtt.ErrNotFound` (CLI maps to exit 4).
  2. if `!force`: `store.List()`, then reuse `NewIndex(tasks).Children(id)` (children: `parent==id`) and
     `NewDepGraph(tasks).Dependents(id)` (dependents: who `depends_on` id). If either is non-empty, return a
     descriptive error listing the referencing ids (CLI → exit 1). *No new derived graph — reuse Index+DepGraph.*
  3. `store.Delete(id)`.
  `NewRemover(store)` — no clock (delete records nothing). Update `core/CLAUDE.md`.
- **`internal/cli`** (`rm.go`): `newRmCmd()` — `mtt rm <id>` (`Args: ExactArgs(1)`), `--force` bool flag; wire
  `core.NewRemover(yaml.NewTaskStore(root))`; on success `fmt.Fprintf(cmd.OutOrStdout(), "removed %s\n", id)`.
  Register in `root.go`. `exitCode(err)` gains an `errors.Is(err, mtt.ErrNotFound) → 4` arm. Update `cli/CLAUDE.md`.

**Feature 2 — `--depends-on` on `mtt add`:**
- **`internal/core`** (`add.go`): `AddParams.DependsOn []mtt.TaskID`. In `Adder.Add`, **before** `Create`
  (like placement validation): dedupe the list, and for each target `store.Get` → not found → error
  (`depends-on target %q not found`, exit 1); set `t.DependsOn`. No cycle check needed (the new task's id is
  unminted, so a self-edge is impossible). Update `core/CLAUDE.md`.
- **`internal/cli`** (`add.go`): `--depends-on` as a `StringSlice` (repeatable + csv); convert `[]string` →
  `[]mtt.TaskID` at the CLI boundary; pass into `AddParams`. Update `cli/CLAUDE.md`.

**Feature 3 — packaging:**
- **`Makefile`**: a `VERSION` variable (empty by default) and a **conditional** `LDFLAGS` — when `VERSION` is
  set, pass `-ldflags "-X github.com/pashukhin/mtt/internal/cli.version=$(VERSION)"` to `build`/`install`;
  when empty, the binary keeps the code default (`0.8.5-dev` after this session's bump). So `make build` is
  predictable and `make build VERSION=v0.8.5` stamps a release. Add a **`smoke`** target: `go install` into a
  temp `GOBIN`, then assert `mtt version` exits 0 with non-empty output and `mtt --help` exits 0 (combine
  stdout+stderr since `version.go` uses `cmd.Println`). `smoke` is **not** added to `check` (it does a real
  install; keep `check` hermetic) — it is a manual/CI packaging gate.
- **`internal/cli/root.go`**: bump the code default `version` `0.8.0-dev` → `0.8.5-dev` (the `-ldflags` `-X`
  target is already `internal/cli.version`).

## Acceptance (must pass)

- **User scenario (rm):** `mtt init` → `mtt add A` (t1) → `mtt add B` (t2) → `mtt dep add t2 t1` (t2 depends
  on t1). Then `mtt rm t1` is **rejected** (exit 1) naming t2; `mtt rm t1 --force` **removes** t1 (t1 gone,
  t2 now has a dangling blocker and is not `ready`); `mtt rm t2` removes t2 (exit 0, `removed t2`);
  `mtt rm t9` (absent) exits **4**.
- **User scenario (--depends-on):** `mtt add A` (t1) → `mtt add B --depends-on t1` creates t2 with
  `depends_on: [t1]` (verify via `mtt dep list t2` / `mtt show t2`); `mtt add C --depends-on t99` (missing
  target) errors (exit 1), no task created.
- **User scenario (packaging):** `make smoke` installs the binary and `mtt version` / `mtt --help` work.
- **e2e (`testscript`):**
  - `rm.txt` — init → add t1,t2 → `dep add t2 t1` → `rm t1` fails (referenced, mentions t2) → `rm t1 --force`
    succeeds → `list` no longer shows t1 → `rm t9` fails with exit 4 (`! mtt rm t9` + a stderr assertion) →
    `rm t2` succeeds. Anchored assertions (not loose substrings).
  - `add_depends_on.txt` — init → add t1 → `add B --depends-on t1` → `dep list t2` shows `depends on: t1`;
    `add C --depends-on t99` fails, `list` still shows only the expected tasks.
- `make check` green (and `make smoke` green, run manually / in CI).

## Carry-over lessons applied (from NEXT_SESSION)

- **New port method justified when data is not embeddable** — `Delete` is a store op, so it earns a base-port
  method (unlike `depends_on`, which rides `Update`).
- **Non-zero exit is DATA** — exit 4 (not found) is a taxonomy code, unit-tested via `exitCode`.
- **CLI output via `fmt.Fprint(cmd.OutOrStdout(), …)`** — `removed <id>` to stdout.
- **Anchored testscript assertions**, not loose substrings.
- **golangci `unused`** — declare a symbol where it is first used (e.g. `Remover` lands with its CLI caller).
- **Atomic file ops in the YAML adapter** — delete is atomic (`os.Remove`), like temp+rename on write.
- **Keep every `CLAUDE.md` current** and do the full docs-sync tick.
